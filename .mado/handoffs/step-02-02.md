# Session Handoff — Step 2.2: Stress-score endpoint (score, zone, sub-scores)

**Date:** 2026-07-02
**Status:** completed

## What Was Done

Exposed the Financial Stress Score over REST so the Step 2.3 Home/Radar gauge can render live:
`GET /api/v1/clients/{id}/stress-score` → the shared success envelope with score, zone, zone colour, the
three sub-scores, and the snapshot date.

- **`stress.StressScoreResponse`** (record DTO) — `score (int)`, `zone (OPTIMAL|WARNING|CRITICAL)`,
  `color (hex)`, `spendingVelocity`/`incomeConsistency`/`liabilityRatio (BigDecimal, 0–100)`,
  `asOfDate (LocalDate)`. Static `from(StressScore)` factory projects the entity and resolves the zone
  colour via a `switch` over `Zone` → §8 hex (`OPTIMAL #1D9E63`, `WARNING #E5A63A`, `CRITICAL #E0574F`).
  The JPA entity never crosses the controller boundary.
- **`stress.StressScoreService`** (`@Service`, read-side) — resolves the client through
  `ClientService.requireClient` (the single 404 contract), serves the latest snapshot via
  `StressScoreRepository.findFirstByClientIdOrderByAsOfDateDesc`, and — only when none exists — lazily
  computes+persists the first one through `StressScoreSnapshotWriter.writeSnapshot` (idempotent per day).
  `@Transactional` (not read-only) so the lazy write path commits.
- **`stress.StressScoreController`** — thin `GET /api/v1/clients/{id}/stress-score`; binds `{id}` as a raw
  string (strict-UUID resolution happens in the service, so malformed/unknown → shared 404), wraps the DTO
  in `ApiResponse.ok`.
- **`StressScoreControllerTest`** (`@SpringBootTest` + `MockMvc`, Postgres-guarded, `@Transactional`
  rollback): envelope shape; `zone` bands `score` per §5.1 and `color` matches the §8 hex; 2.5 s NFR for
  the large-volume persona; unknown id → 404 error envelope.

## Decisions Made

- **New read-side `StressScoreService` rather than bolting a read onto the writer.** The step table said
  "modify the Step 2.1 service to add a latest-score read", but the only existing service is the
  `StressScoreSnapshotWriter` (a write concern). Adding a dedicated read service matches the established
  `ClientService`/`AccountService` read-service pattern and keeps read/write responsibilities separate.
  The writer is reused (not duplicated) for the lazy-compute fallback — no scoring logic is re-implemented.
- **Zone→colour lives in the DTO (`StressScoreResponse.from` / `colorFor`), not on `Zone`.** The step
  scopes colour to the API response and its "Files to Modify" table does not include `Zone.java` (AGENT.md:
  don't touch previous steps' code unless the step says so). Hex constants are documented as needing to
  stay identical to the Step 2.3 Flutter theme tokens.
- **Lazy compute on first hit, serve persisted afterwards.** Nothing writes snapshots at boot, so the
  first request per client per day computes (one 90-day window scan) and persists; every later request is a
  single indexed read — keeping the endpoint well inside the 2.5 s NFR. Verified live: the boot log shows
  the `select stress_scores → insert stress_scores` lazy path firing on the first call.
