# Session Handoff — Step 2.1: StressScoreCalculator + Liquibase stress_scores + snapshots

**Date:** 2026-07-02
**Status:** completed

## What Was Done

Phase 2 begins. Implemented the Financial Stress Score (DESIGN §5.1) end-to-end on the backend: a
Liquibase-managed snapshot table, a pure calculator over seeded telemetry, and an imperative writer that
upserts one daily snapshot per client. No endpoint yet — that's Step 2.2.

- **Liquibase `002-stress-scores.sql`** — `stress_scores` per DESIGN §4.2: `id (uuid pk)`,
  `client_id (uuid fk → clients)`, `as_of_date (date)`, `score (int)`, `zone (text)`, and the three
  sub-scores `spending_velocity`/`income_consistency`/`liability_ratio` as `numeric(14,2)`,
  `created_at (timestamptz default now())`. **Unique `(client_id, as_of_date)`** (the upsert key) plus a
  `(client_id, as_of_date desc)` index for the latest-snapshot read. Verified in Postgres.
- **`stress.Zone`** enum (`CRITICAL|WARNING|OPTIMAL`) with `forScore(int)` applying the §5.1 thresholds
  (`>=70 OPTIMAL`, `40–69 WARNING`, `<40 CRITICAL`) — the single source of truth for banding.
- **`stress.StressScore`** JPA entity + **`StressScoreRepository`** with
  `findFirstByClientIdOrderByAsOfDateDesc` (read path for 2.2) and `findByClientIdAndAsOfDate` (upsert
  lookup). Entity stays in the service layer.
- **`stress.StressScoreCalculator`** (`@Service`, pure/deterministic) — the functional core. Given a
  window of `Transaction`s it derives three metrics and normalises each to a `[0,1]` healthiness
  sub-score, then combines with §5.1 weights `w1=0.4, w2=0.3, w3=0.3` → `round(100*weighted)` → `Zone`.
  Returns an immutable **`StressScoreResult`**.
- **`stress.StressScoreSnapshotWriter`** (`@Service`) — the imperative shell: injectable `Clock`
  (prod = `systemUTC`), loads the trailing 90-day window via
  `TransactionRepository.findByAccount_Client_IdAndBookingDateBetween`, scores it, and upserts the
  `as_of_date = today` row (scales sub-scores to 0–100 for storage). Reuses `ClientService.requireClient`
  for the 404 contract.
- **Tests** — `StressScoreCalculatorTest` (pure, no Spring; reads the two personas straight from
  `data-mocks/`) and `StressScoreSnapshotWriterTest` (`@SpringBootTest`, Postgres-guarded + `@Transactional`
  rollback so it never pollutes the seeded DB).

## Decisions Made

- **Functional core / imperative shell.** The calculator never reads the clock or DB (stable unit tests);
  the writer is the only piece touching either — per the step's explicit instruction.
- **Normalisation curves (the §5.1 open decision) — transparent piecewise-linear ramps.** Chosen for
  demo-defensibility over smooth/exponential curves. Documented in each method; `clamp01` guards range:
  - **velocity** `= 1 − (burnRate − 0.5)/1.0` — burn `debits/credits`; ≤0.5 → 1.0, ≥1.5 → 0.0. No income
    at all short-circuits to 0.
  - **consistency** `= clamp01(1 − incomeCv) · cadenceCoverage` — CV of monthly income × share of window
    months that received income. Steadier + more regular → higher.
  - **liability** `= 0.5·clamp01(1 − obligationShare/0.6) + 0.5·clamp01(bufferMonths/3)` — recurring
    obligations as a share of income (≥0.6 → 0) blended with runway = balance/monthly-expense (≥3mo → 1).
  - Weights left at the §5.1 defaults (0.4/0.3/0.3). Retune point is the boxed section at the bottom of
    `StressScoreCalculator`.
- **Recurring-obligation detection is cadence-based, not category-hardcoded** (Global Rules / step note):
  debits are grouped by `resolveCategory()+cleansedDescription` and a group counts as recurring if it
  appears in ≥ `max(2, ceil(months/2))` distinct months. New categories need no code change.
- **Sub-scores persisted on a 0–100 scale** (not raw ratios) despite the DESIGN column names
  (`spending_velocity` etc.) — they hold the normalised sub-scores the step asks to store, directly usable
  by the Step 2.3 radar chart. `numeric(14,2)` keeps 2 decimals.
- **Window = trailing 90 days from `today`** in the writer (calculator itself is window-agnostic — it
  scores whatever list it's given; the calculator test feeds full 6-month history for a stable ordering).
