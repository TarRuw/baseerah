# Handoff — Step 1.1: Liquibase core schema (clients, accounts, transactions)

**Status:** completed
**Date:** 2026-07-02

## What was done

- Created the first Liquibase changeset, `backend/src/main/resources/db/changelog/changes/001-core-schema.sql`,
  as a **formatted-SQL** file (`--liquibase formatted sql` + `--changeset baseerah:001-core-schema`). The
  master changelog's `includeAll` over `changes/` picks it up automatically — no master edit needed.
- Introduced the three core tables per **DESIGN.md §4.2**, exact columns/types:
  - `clients` — UUID PK (`gen_random_uuid()`), `external_id text not null unique`, `profile_label`, `persona`,
    `created_at timestamptz not null default now()`.
  - `accounts` — UUID PK, `client_id uuid not null references clients(id)`, `external_id`, `bank_name`,
    `display_color`, `currency`, `latest_balance numeric(14,2)`, `tokenized_account_id`.
  - `transactions` — UUID PK, `account_id uuid not null references accounts(id)`, `external_id`,
    `direction text not null`, `amount numeric(14,2) not null`, `currency`, `raw_description`,
    `description_cleansed`, `category`, `category_confidence numeric`, `booking_date timestamptz not null`,
    `closing_balance numeric(14,2)`.
- Indexes: `idx_accounts_client_id (client_id)`, `idx_transactions_account_id_booking_date (account_id, booking_date)`.
- Enabled `pgcrypto` (`create extension if not exists pgcrypto`) for `gen_random_uuid()` — not `uuid-ossp`, per step.

## Decisions

- **Added the optional unique index `uq_transactions_external_id` on `transactions(external_id)`.** The step
  left this optional; I added it because **Step 1.3 (MockDataSeeder) must be idempotent** — a unique key lets
  the seeder use `INSERT ... ON CONFLICT (external_id) DO NOTHING`. The mock `transaction_id`s
  (e.g. `tx-client_001_family-99`) are stable and unique, so the constraint is satisfiable. It is **null-safe**
  without a partial predicate: PostgreSQL treats each NULL as distinct in a UNIQUE index. `accounts.external_id`
  was intentionally left non-unique (no such downstream need identified yet).
- **`category` stored as `text`** (not a DB enum) per DESIGN.md §4 — the category set is data-driven with a
  fallback; JPA-level enum handling is deferred to Step 1.2.
- No other DESIGN.md §4.2 tables were created — later phases add them in their own changesets.
