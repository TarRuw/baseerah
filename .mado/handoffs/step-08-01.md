# Session Handoff — Step 8.1: Localize server-provided content strings (Accept-Language)

**Date:** 2026-07-07
**Status:** completed — Phase 8 begins (31/35, project ~88.6%). Closes QA finding **I18N-01** and **COPY-01**.
Verified by **102/102 backend tests** (incl. a new 4-case `I18nContentTest` hitting the live API in ar/en)
**and a live curl dry-run** against Postgres + `bootRun` (JDK 21, `GENAI_PROVIDER=mock`).

## What Was Done

The backend emitted hardcoded **English** content for both `ar` and `en`, so English leaked into the
Arabic-first RTL UI. Introduced a locale-aware content seam and routed every user-facing content string
through it, resolved per request from `Accept-Language`.

- **Content bundles + resolver (Task 1):**
  - `messages.properties` (English **default** bundle) + `messages_ar.properties` (Arabic, UTF-8). Used the
    default bundle as the English source (rather than a separate `messages_en`) with
    `spring.messages.fallback-to-system-locale: false`, so any non-`ar` locale lands on English instead of the
    JVM's system locale — more robust than an `en`-only bundle.
  - `common/Messages.java` — `@Component` over Spring's auto-configured `MessageSource`, with
    `get(key, args…)` (current request locale via `LocaleContextHolder`) and `get(locale, key, args…)`
    (explicit locale, for the challenge read path). A `Messages.forTests()` factory backs pure unit tests
    with the real bundles.
  - `application.yml` — `spring.messages.{basename,encoding=UTF-8,fallback-to-system-locale=false}`.
- **Loan verdict (Task 2):** `LoanCalculator` verdict constants are now message **keys**; the pure `compute`
  emits the key, the shell `simulate` resolves it to the locale — so the §5.3 math stays a pure, locale-free
  function (its unit test is unchanged).
- **GenAI chat (Task 3):** `MockGenAi` templates every reply (`chat.generic` / `chat.scenario.*` /
  `chat.invoice.action`) from keys with pre-formatted figure args; `RemoteGenAi.systemPrompt` now pins the
  reply language to the request locale explicitly ("Reply strictly in Arabic/English"). Wired `Messages` into
  the `MockGenAi` bean via `GenAiConfig`.
- **Rescue (Task 4):** `RescueService` resolves both bridge-option `label`/`detail` and the confirmation line
  from keys; the `RescueOptionType` enum and all figures are untouched.
- **Challenges (Task 5 + COPY-01):** `ChallengeService` title/subtitle now localise **at read time** — see
  the schema decision below. The category label moved into per-locale `category.*` keys, which also delivers
  the COPY-01 grammar fix ("restaurant dining", not "restaurants dining"). Progress-text currency label now
  comes from `currency.sar` (removed the duplicate SAR constants).

## Key decisions

- **⚠️ Task 6 (bank verdict) is a deliberate no-op — the spec's premise was wrong.** The API emits a
  locale-independent `Verdict` **enum** (`OK`/`WARN`/`BAD`); the Flutter client already maps it to its own
  localized headline (`VerdictVisuals.of(verdict, l)` → `l.bankVerdictOkTitle`, present in both `app_en.arb`
  and `app_ar.arb`). There is **no English content literal in `UnderwritingService`** to localize —
  `"Sustains the debt"` lives only in the frontend. Localizing it server-side would be dead code the UI never
  reads, so `UnderwritingService` was left unchanged. `I18nContentTest` documents this exclusion.
- **Challenge copy must resolve at read time, not generation time (schema change).** Title/subtitle are
  **persisted** when challenges are generated (on boot), so per-request localization is impossible if resolved
  text is stored. Instead the `title`/`subtitle` columns now store message **keys**, and a new nullable
  `challenges.text_args` column (Liquibase `009-challenge-i18n.sql`) stores the pipe-delimited, pre-formatted
  numeric arguments the templates interpolate. `toDto(locale)` resolves key + per-locale category label +
  numeric args. `ChallengeSpec.title/subtitle` became `titleKey/subtitleKey` + `textArgs`; the pure `detect`
  test is unaffected (it asserts on `code`/`categoryTrigger`/`targetValue`/`pct`, not the copy).
- **Figures stay locale-neutral (Western digits) in both languages** — all numeric args are pre-formatted to
  strings before `MessageFormat`, so Arabic replies never shape digits to Arabic-Indic and grouping never
  shifts. Honors the Global Rule: only *wording* becomes locale-aware; figures, colours and enums are
  identical.
- **Frontend needs no change** — it already renders these server strings raw; once the backend localizes, the
  Goals/Simulate/Rescue/chat UI follows. (Bank was the sole exception, already handled client-side.)

## Notes / gotchas

- **Tests share the live `baseerah-postgres` (no Testcontainers).** The first full run showed 5 transient
  gamification failures from **stale pre-migration rows** (a prior app boot had committed challenge rows before
  `009` added `text_args`); a clean `./gradlew test --rerun-tasks` regenerated them and is green and stable
  across repeated runs. Beware Gradle UP-TO-DATE caching masking whether tests actually executed — use
  `--rerun-tasks` to force a real run.
- **A stale backend instance (old code) was occupying port 8080** and initially answered curls with English
  Arabic-labelled requests; killed by PID (not `pkill`, to avoid touching this session) before booting the
  freshly-compiled instance. The `bootRun` started for the dry-run was stopped; `baseerah-postgres` left up.
- **Test-construction churn:** the new `Messages` dependency changed constructor arities for
  `LoanCalculator`, `MockGenAi`, `RescueService`, `ChallengeService` and the `genAiClient` bean — updated
  `LoanCalculatorTest`, `MockGenAiTest`, `GenAiProviderSelectionTest`, `RewardsServiceTest`,
  `GamificationControllerTest` accordingly (all pass).
