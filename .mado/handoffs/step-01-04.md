# Session Handoff — Step 1.4: Core DTOs + client/account/transaction endpoints

**Date:** 2026-07-02
**Status:** completed

## What Was Done

Exposed the seeded data over REST as DTOs through the Step 0.4 envelope — the first three read
endpoints from DESIGN.md §6. Entities never cross the controller boundary.

- **DTOs** (immutable records):
  - `client/dto/ClientDto` — `id, externalId, profileLabel, persona, createdAt`.
  - `account/dto/AccountDto` — `id, bankName, displayColor, currency, latestBalance,
    tokenizedAccountId`. **No raw `external_id`** (SAMA §9).
  - `transaction/dto/TransactionDto` — `id, direction, amount, currency, rawDescription,
    descriptionCleansed, category, categoryConfidence, bookingDate, closingBalance`.
- **Mappers** — `ClientMapper` / `AccountMapper` / `TransactionMapper`: `final` classes, static
  `toDto`, stateless. Read scalar fields only (no lazy association access → no N+1).
- **Services** (`@Service @Transactional(readOnly=true)`, constructor injection):
  - `ClientService.list()` → `List<ClientDto>`; `requireClient(String)` → resolves the client entity
    or throws `NotFoundException`. **Single source of truth for `{id}` → 404**, reused by the other two.
  - `AccountService.accountsForClient(String)` — validates via `requireClient`, then maps accounts.
  - `TransactionService.transactionsForClient(String, page, size)` — validates, then returns a
    `Page<TransactionDto>` ordered **newest-first** (sort enforced in the service, not the caller);
    size clamped to `[1, 200]`, default 50.
- **Controllers** (thin, envelope-wrapping only): `ClientController` (`GET /api/v1/clients`),
  `AccountController` (`GET /api/v1/clients/{id}/accounts`), `TransactionController`
  (`GET /api/v1/clients/{id}/transactions?page=&size=`).
- **`common/PageResponse<T>`** — stable pagination envelope (`content, page, size, totalElements,
  totalPages`) so we don't serialize Spring's `PageImpl` (whose JSON shape Spring warns is unstable).
  Sits inside `ApiResponse.data`.
- **`web/CoreEndpointsTest`** — `@SpringBootTest @AutoConfigureMockMvc` against the live seeded
  Postgres, guarded by the `postgresReachable` `@EnabledIf` (skips cleanly when the DB is down).

## Decisions Made

- **`{id}` is strict UUID-only (user-approved).** `external_id` is NOT accepted as a path id. The
  UUID is parsed inside `ClientService.requireClient`; a malformed non-UUID and an unknown UUID both
  throw `NotFoundException` → the Step 0.4 handler renders `404 NOT_FOUND`. Parsing in the service
  (not binding `@PathVariable UUID`) keeps all not-found behaviour in one place and avoids a separate
  `MethodArgumentTypeMismatch` → 500 path. Flutter must `GET /clients` first to learn each UUID.
- **`TransactionDto.category` carries the raw feed string verbatim** (lossless), not the typed
  `Category` enum. The typed `Category.resolve(...)` view stays an internal scoring/analytics concern
  (Phase 2+); a history list should show the actual feed label.
- **Pagination via `?page=&size=`** (not `?limit=`). Default size 50, hard cap 200
  (`TransactionService.DEFAULT_PAGE_SIZE` / `MAX_PAGE_SIZE`); out-of-range params degrade gracefully
  (page floored at 0, size clamped) rather than 400-ing.
- **Ordering enforced server-side** (`Sort.by(DESC, "bookingDate")` built in the service), so
  "newest-first" holds regardless of query params.
- **Cross-service reuse:** `AccountService`/`TransactionService` depend on `ClientService` for
  resolution. An entity passing *between services* stays within the service layer (only DTOs cross the
  controller boundary), consistent with the ORCHESTRATION rule.
