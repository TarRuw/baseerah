# Session Handoff — Step 5.2: Challenge/rewards endpoints (list, claim, balance)

**Date:** 2026-07-04
**Status:** completed (build green; all 63 tests pass against live Postgres — 57 from 5.1 + 6 new web-layer
tests, 0 skipped/0 failures)

## What Was Done

Exposed the Phase 5 gamification feature over REST (FR-09/10, DESIGN §6), all enveloped, so the Step 5.3
Goals screen can drive the live claim flow. Thin controller over the existing 5.1 services; entities are
mapped to DTOs inside the service layer (Global Rules).

- **DTOs** (`com.baseerah.gamification.dto`):
  - `ChallengeDto {id, icon, title, subtitle, reward, pct, progressText, claimable, claimed}` — the
    client-facing projection of `Challenge` + its `ChallengeProgress`.
  - `RewardsDto {points, riskTier}` — balance + tier for `GET /rewards`.
  - `ClaimResponse {points, riskTier, challenge}` — the post-claim state (new balance/tier + the challenge
    now `claimed`), so the screen can flip the card in place without a re-fetch.
- **`ChallengeService`** (modified) — added read/mapping methods that keep entities in the service:
  `listForClient(UUID, Locale)` (resolves the client via `ClientService.requireClient` → shared 404, then
  lists + maps) and `challengeDtoFor(UUID, UUID, Locale)` (single, client-scoped, used post-claim; a foreign
  challenge is reported as `NotFoundException`, never leaked). Plus the `progressText(current, target, locale)`
  helper.
- **`GamificationController`** (new, thin) under `/api/v1/clients`:
  - `GET /{id}/challenges` → enveloped `List<ChallengeDto>`.
  - `POST /{id}/challenges/{cid}/claim` → delegates to `RewardsService.claimChallenge`, then re-projects the
    claimed challenge via `ChallengeService`; returns `ClaimResponse`. Malformed/foreign ids → 404.
  - `GET /{id}/rewards` → enveloped `RewardsDto`. This route explicitly resolves the client through
    `ClientService.requireClient` (the ledger sum alone would report a bare `0` for a never-seeded id instead
    of a 404).
- **Web-layer tests** (`GamificationControllerTest`, `@SpringBootTest` + MockMvc, live-DB, `@Transactional`
  rollback, `@EnabledIf(postgresReachable)`): list returns the enveloped seeded challenges with progress
  fields; claim on the completed goal returns increased `points` + `claimed=true` + one ledger row; a second
  claim is a **409** with no balance change; an incomplete claim is a **409** with no ledger write; rewards
  returns `points`+`riskTier`; unknown id → 404. Deterministic via a `FIXED_CLOCK` regeneration in
  `@BeforeEach` (same technique as `RewardsServiceTest`).

## Decisions Made (the step left the error contract open — finalized here)

- **Claim-rejection error contract → `409 CONFLICT`.** Introduced `com.baseerah.common.ConflictException`
  (parallels `NotFoundException`) with a `GlobalExceptionHandler` mapping → HTTP 409, code `CONFLICT`.
  `RewardsService.claimChallenge` now throws it (instead of the bare `IllegalStateException` from 5.1, which
  the generic handler would have turned into a 500) for both guards — *already-claimed* and *not-complete*.
  A well-formed-but-conflicting request is neither "missing" (404) nor "malformed" (400), so 409 is the
  honest status. **The two 5.1 `RewardsServiceTest` assertions were updated** from `IllegalStateException`
  to `ConflictException` accordingly (the handoff explicitly deferred this contract to 5.2). The internal
  "challenge has no progress row" invariant stays an `IllegalStateException` (→ 500) — it is a data-integrity
  error that cannot occur through the API, not a client-recoverable conflict.
- **`RewardsDto.riskTier` carries the gamification `RewardTier`.** The step spec and DESIGN §5.6 (line 225,
  Goals screen) both label the field `riskTier`; its *value* is the consumer `RewardTier` (BRONZE…PLATINUM),
  which remains a distinct scale from the bank-side underwriting `risk_tier` (Phase 6) — no shared mapping
  (5.1 handoff). Only the JSON field name follows the design label.
- **`progressText` is server-formatted and locale-aware.** Format `"<current> / <target> <label>"` with
  thousands-grouped whole SAR (matching the DESIGN §8 `fmt(n)` grouping via `String.format(Locale.US,"%,d")`),
  label `"SAR"` by default and `"ريال"` when `Accept-Language` is Arabic (Spring resolves the header into the
  controller `Locale`, threaded to the service). Digits stay Western — the Goals screen does full RTL
  re-formatting in 5.3.
