## What Was Done

Built the consumer **Goals** screen (DESIGN §7.4, FR-09/10) as a small state machine wired to the Step 5.2
gamification endpoints — a gold reward-points card over a live challenge list with a claim flow that flips
the card and the points header in place. RTL-ready in Arabic, using only shared theme tokens. Mirrors the
established `data/ · state/ · widgets/` feature layout (as Rescue/Simulate) rather than the flat paths the
step sketch listed — the step note said to follow the Phase-0.3 convention.

- **Data layer** (`features/goals/data/`):
  - `challenge.dart` — `Challenge` (mirrors `ChallengeDto`) with `isComplete` (`pct >= 100`, drives the
    gold→green bar) and a `buttonState` getter (`ClaimButtonState.{inProgress,claim,claimed}`) that is the
    single source of the §7.4 button rule so no widget re-derives it.
  - `rewards.dart` — `Rewards {points, tier}` + a `RewardTier` enum (`bronze…platinum`). `fromWire` tolerates
    an unknown tier as `bronze` (forward-compatible if the server adds a tier). Parses the wire field named
    `riskTier` (its value is the gamification tier, distinct from the bank `risk_tier` — Step 5.1/5.2).
  - `claim_result.dart` — `ClaimResult {rewards, challenge}` mirroring `ClaimResponse {points, riskTier,
    challenge}`; reuses `Rewards.fromJson` on the response's top level (same `points`+`riskTier` shape).
  - `goals_repository.dart` — dio calls `fetchChallenges` / `fetchRewards` / `claimChallenge`, each
    unwrapping the shared `{status,data}` envelope via `unwrapEnvelope` (widgets never touch dio).
- **State** (`features/goals/state/goals_providers.dart`): `GoalsController` (`StateNotifier`), phases
  `loading | error | ready`. `_load()` fetches rewards + challenges **in parallel** (`Future.wait`) for a
  faster first paint. `claim(id)` is **on-response, not optimistic** — the card only flips after the server
  confirms, so a rejected claim leaves state unchanged with nothing to roll back; it returns a `bool` (like
  Rescue's `confirm`) so the *screen* owns the snackbar and the notifier never touches `BuildContext`.
  `claimingIds` is a `Set` so a per-card spinner locks only that card. Public `goalsClientIdProvider`
  (overridable in tests) resolves the persona via Home's `currentClientProvider`.
- **Widgets**:
  - `reward_points_card.dart` — the 135° `goldGradient` card: large points via the §8 `Fmt.fmt` (locale
    digits) + a translucent tier badge (glyph per tier + localized name). Copy: "Reward points" / "Alinma
    Akthar" (from the prototype). Renders the **real** tier (BRONZE…PLATINUM), not the prototype's hardcoded
    "Risk Tier B" demo stand-in (Global Rule: no magic values from the prototype).
  - `challenge_card.dart` — semantic-icon→glyph map (`restaurant/shopping/flight/hotel/subscription/fitness/
    savings/star`, default `savings`), title/subtitle, a reward-points chip, a `LinearProgressIndicator`
    animated 0→`pct` on the theme's score curve (gold, → success-green at `pct >= 100`), `progressText`, and

  - `goals_screen.dart` — title/subtitle header (copy "Savings Challenges") over the phase switch; ready

- **i18n** (`app_en.arb` / `app_ar.arb` + the three hand-maintained generated `app_localizations*.dart`):

## Decisions Made

- **On-response claim merge (not optimistic).** The AC "a rejected claim leaves state unchanged" is free this
  way — no optimistic mutation to revert. On a local backend the brief per-card spinner is imperceptible; the
  merge rebuilds off the *latest* state (not the captured snapshot) so a concurrent claim isn't clobbered.
- **Render the real `RewardTier`, not the prototype's "Risk Tier B".** The 5.2 contract's `riskTier` value is
  the gamification tier (BRONZE…PLATINUM); the prototype string was a demo stand-in. Tier→label via new
  `tier*` i18n keys; `fromWire` degrades unknown values to `bronze` rather than throwing.
- **Progress bar = `LinearProgressIndicator`** (not a hand-rolled `Stack`/`FractionallySizedBox`): guaranteed
  full-width and Directionality-aware (fills from the right under RTL), so Arabic mirrors correctly for free.
- **Feature-folder layout** follows Rescue/Simulate (`data/state/widgets/`), overriding the flat file paths
  the step table sketched — per the step's own "mirror the Phase-0.3 convention" note.
