## Decisions Made

- **Client `external_id` derived from the filename** (`client_001_family_6_months_data.json →
  client_001_family`), falling back to the `acc-<id>` account-id convention. The file is the client
  unit, so filename derivation is the robust choice and generalises to multi-account clients (the mock
  is single-account per file).
- **Deterministic bank + colour via `String.hashCode()`** (`Math.floorMod(hash, size)`) against a fixed
  Saudi-bank list + the DESIGN.md §8 palette. `String.hashCode` is spec-stable across JVMs/reboots, so
  no seeded RNG is needed. Collisions are acceptable — the multi-bank badge (DESIGN.md §7) needs *stable*
  assignment, not *unique*. (`family` and `vip` both map to Banque Saudi Fransi/gold — expected.)
- **Tokenization = `TKN-` + first 24 hex chars of SHA-256(external_id)** — deterministic, non-reversible,
  never equal to the raw id (SAMA §9). No raw account id ever reaches the bank side.
- **Idempotency at client granularity** via `clientRepository.existsByExternalId`; per-client `save`
  is transactional so a graph is all-or-nothing. A file partially loaded after a crash would be skipped
  — recovery is a manual truncate/re-seed (documented, out of scope).
- **Path resolution walks parent dirs.** The documented run is `cd backend && ./gradlew bootRun`, so the
  working dir is `backend/` but `data-mocks/` is at the repo root. The default `data-mocks` is resolved
  by searching the working dir and its parents, so it works whether launched from root or `backend/`.

## Issues Encountered — ⚠️ MISSING SOURCE DATES (needs review)

- **12 of the freelancer's 78 transactions have no `booking_date_time`** (the key is absent in
  `client_003_freelancer_6_months_data.json`). `booking_date` is `NOT NULL` and acceptance criterion #2
  requires transaction counts to match the source, so the rows cannot be dropped → the dates are
  **imputed**.
- **Strategy (user-approved): linear interpolation as a placeholder.** The mock array is confirmed
  strictly newest-first, so each missing date is set to the **midpoint of its nearest real-dated
  neighbours** (or that single neighbour at an array edge; the file `timestamp` only if an account had
  no dates at all — does not occur). This is a *placeholder* — the seeder logs a **WARN** listing every
  imputed transaction id so it can be reviewed/adjusted:
  - Affected ids: `tx-client_003_freelancer-{2,12,16,23,26,28,34,36,46,55,60,63}`.
  - `latest_balance` is unaffected: the most-recent transaction is always real-dated (imputed dates are
    bounded by the real max), so balances are never derived from a placeholder.
  - **If true dates surface later**, update these 12 rows; downstream Phase 2/3 time-window analytics read
    `booking_date`, so the placeholders slightly affect stress-score/forecast bucketing for this persona.
