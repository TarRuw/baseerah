# Session Handoff — Step 3.1: ForecastEngine interface + HeuristicForecast + Liquibase forecasts

**Date:** 2026-07-02
**Status:** completed (with an approved, documented deviation from one acceptance criterion — see below)

## What Was Done

Phase 3 begins. Delivered predictive cash-flow forecasting (FR-04, DESIGN §5.2) behind a swappable
`ForecastEngine` seam: the interface + result records, the default heuristic implementation, the Liquibase
`forecasts` table, and a pure persona-driven unit test. No HTTP exposure or persistence-writing yet — that's
Step 3.2.

- **`forecast.ForecastEngine`** — the stable seam. Single method `ForecastResult project(UUID clientId,
  int horizonDays)`. Two nested records carry the outcome: `ForecastPoint {LocalDate date, BigDecimal
  projectedBalance}` and `ForecastResult {List<ForecastPoint> points, LocalDate deficitDate (nullable),
  BigDecimal minProjectedBalance}`. Exposes only domain types — no JPA entity, no reference to
  `HeuristicForecast` (Global Rule: engines stay behind interfaces). This is the seam a Python
  Prophet/XGBoost sidecar replaces later.
- **`forecast.HeuristicForecast`** (`@Service` implementing the interface) — functional core / imperative
  shell, the same split as `StressScoreCalculator`/`StressScoreSnapshotWriter`:
  - **Shell** `project(...)`: reads an injectable `Clock` (prod = `systemUTC`) for "today", loads the
    trailing **90-day** window via `TransactionRepository.findByAccount_Client_IdAndBookingDateBetween`,
    delegates to the pure core.
  - **Core** `forecast(window, today, horizonDays)` (package-visible, no clock/DB → unit-testable): starts
    from the latest `closing_balance`; groups transactions by `direction + category +
    description_cleansed`; classifies each group as **recurring-monthly** by cadence; splits recurring
    **inflows** (salary) from recurring **outflows** (rent/subs/instalments); averages all non-recurring
    debits into a **daily discretionary burn**; then walks `[today, today+horizonDays]` day-by-day,
    applying scheduled flows on their due day-of-month and subtracting burn — recording every point, the
    first date balance < 0 (`deficitDate`), and the running `minProjectedBalance`. Computed entirely from
    seeded transactions — no prototype constants (Global Rules).
- **Liquibase `003-forecasts.sql`** — `forecasts` per DESIGN §4.2: `id (uuid pk)`, `client_id (uuid fk →
  clients)`, `horizon_days (int)`, `generated_at (timestamptz default now())`, `deficit_date (date null)`,
  `min_projected_balance (numeric(14,2))`, `points (jsonb)`, plus a `(client_id, generated_at desc)` index.
  Verified applied cleanly in Postgres (`databasechangelog` shows `003-forecasts EXECUTED`; `\d forecasts`
  matches). No JPA entity/repository for it yet — Step 3.2 owns writing/serving forecasts.
- **`HeuristicForecastTest`** (pure, no Spring/DB) — reads the two personas straight from `data-mocks/`
  (same pattern as `StressScoreCalculatorTest`), projects each persona's trailing 90-day window from a
  fixed `today = 2026-07-02`, horizon 60. Asserts series well-formedness (non-empty, size = horizon+1,
  starts at today, ends at horizon, strictly date-ordered, `minProjectedBalance` = series min) and the
  persona contrast (see deviation). Full suite: **21 tests green** against the live DB.

## Decisions Made

- **Functional core / imperative shell**, matching the Phase-2 convention — the core is pure so the
  persona test needs no Spring or Postgres.
- **Cadence-based recurring detection (design decision the step left open).** A group is recurring-monthly
  when it has **≥2 occurrences** whose **median consecutive-gap is 30 ± 7 days**. This cleanly keeps
  ~monthly items (salary on a stable day) as scheduled flows while letting frequent bursty spend
  (groceries on 18 days/month, ride-hailing) fall through to discretionary burn (tiny median gap → rejected).
  Typical amount = mean per occurrence; typical due day = median day-of-month, clamped into short months
  (31 → 30/28). Constants live at the top of `HeuristicForecast` (`WINDOW_DAYS`, `MONTHLY_DAYS`,
  `CADENCE_TOLERANCE_DAYS`, `MIN_RECURRING_OCCURRENCES`) — the retune point.
- **Discretionary burn denominator = the observed window span** (days between earliest and latest booking
  date), not a hardcoded 90, so it stays correct when a client has < 90 days of data.
- **No double-counting:** recurring debits are excluded from the burn total; they're applied on their due
  days instead.

## ⚠️ Deviation from acceptance criteria (approved by user)

The step's criterion *"the freelancer persona forecast produces a non-null `deficit_date` within the
horizon"* is **not satisfiable by a faithful engine on the frozen mock data**, and was realigned with the
user's explicit approval (chosen option: *"Faithful engine, realign test"*).

**Why:** `client_003_freelancer` is actually the **healthiest saver** in the data. Their irregular
`INFLOW_BUSINESS` income (~185k over six months, all *undated* in the mock) has driven their balance to
**239,048 SAR and rising**. DESIGN §5.2 mandates projecting from the latest `closing_balance`; from 239k,
with only discretionary burn + minor recurring outflows (~3–5k/month) and no recurring inflow, the balance
would take **years** to reach zero — no deficit in a 30-day or even 12-month horizon. Forcing one would
require fabricating burn constants (violates "compute from seeded transactions") or editing the read-only
`data-mocks/` (forbidden). The family half of the criterion (`deficitDate == null`) holds naturally.

**What the test asserts instead** (data-honest, and it still pins the behaviour the design cares about):
- **Family** (recurring salary detected) → `deficitDate == null` **and projected balance recovers above
  its start** (the recurring inflow lifts it).
- **Freelancer** (irregular income → *no* recurring inflow detected) → projection **only declines**, but
  its large accumulated buffer means it **does not cross zero within the horizon** → `deficitDate == null`.

## Flag for later phases

- **Phase 4 (Smart Rescue, the freelancer's headline demo per §11):** the freelancer's deficit must be
  driven by a **what-if scenario** (e.g. a large loan repayment, an income-drop stress, or a shorter
  liquidity-buffer assumption) layered on top of this base projection — it is *not* a property of the base
  forecast. The Phase-4 step files should make that explicit. Alternatively, if a base-forecast deficit is
  desired for the demo, that requires an explicit decision to adjust the (currently read-only) freelancer
  mock — out of scope here and not done.
- **Test isolation (pre-existing, not caused by this step):** the full suite hit a stray `stress_scores`
  row left in the shared local Postgres from earlier dev activity, which blocked `MockDataSeederTest`'s
  client delete (FK `stress_scores → clients`, ON DELETE NO ACTION). Cleared with
  `truncate stress_scores, forecasts;` and the suite went green. The new `forecasts` FK follows the same
  NO-ACTION convention as `stress_scores`; if snapshot/forecast tests start persisting rows against the
  live DB, that delete-order fragility in `MockDataSeederTest` may resurface and want a proper fix.
