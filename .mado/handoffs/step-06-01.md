## What Was Done

Delivered the B2B credit-verification core (FR-08, DESIGN §5.5) in a new `com.baseerah.bank` package.

- **Migration `006-bank.sql`** — `loan_applications` (request + report columns per §4.2, all report fields
  nullable so a queued applicant can sit un-underwritten) and a singleton `risk_policy`. Singleton is enforced
  in-schema: a `singleton boolean not null default true` guard column made `unique` + `check (singleton is
  true)` so a second row is impossible; the migration inserts exactly one default policy. `loan_applications`
  carries a `seed_key text unique` idempotency handle (mirrors the `challenges (client_id, code)` key). `verdict`
  / `decision` are text with domain CHECKs (NULL passes → the un-scored / undecided states).
- **Entities/repos** — `LoanApplication` (report fields nullable; `applyReport(...)` stamps a computed report
  back on; `clientRef` stored as a raw client UUID, **not** a `@ManyToOne** — an applicant may be synthetic,
  and the bank side holds only a tokenized reference, DESIGN §9). `RiskPolicy` (the `singleton` guard column is
  intentionally **unmapped** — fine under `ddl-auto=validate`). `RiskPolicyRepository.loadPolicy()` is the
  canonical single-row accessor. `Verdict {OK,WARN,BAD}`, `Decision {APPROVE,DECLINE}` enums.
- **`UnderwritingService`** — functional-core/imperative-shell like the other engines. `generateReport(id)`
  (shell) projects the linked client 12 months via the injected **`ForecastEngine` interface**, reads the
  trailing 90-day window, computes the report, persists it, returns it. `assemble(app, window, projection)`
  (pure core) computes the five metrics deterministically — unit-tested with no Spring/DB.
- **`UnderwritingReport`** record — the value returned to Step 6.2 (keeps the entity in the service layer).
- **`BankApplicantSeeder`** — 7-applicant queue (`@EventListener(ApplicationReadyEvent)`, after `MockDataSeeder`,
  idempotent by `seedKey`). 5 linked (incl. `client_005_vip`) underwritten at seed time so the pipeline has
  verdicts at rest; 2 synthetic fillers left un-scored.
- **Tests** — `UnderwritingServiceTest` (7 unit tests, no Spring/DB): the three §5.5 verdict bands via the pure
  core, `verdictFor` thresholds pinned directly, a mock-`ForecastEngine` shell test asserting forecast reuse +
  persistence, and unlinked-applicant rejection. Also updated `MockDataSeederTest` to clear
  `loan_applications` in its clean-slate step (see Decisions).

## Decisions Made (DESIGN §5.5 fixes only the verdict thresholds — the curves were the engineer's call)

Everything is computed from seeded telemetry; no prototype constants (Global Rules). All documented in the
`UnderwritingService` class Javadoc:

- **Stamina (0–100)** = `0.6·survival + 0.4·retention` over a 12-month `ForecastEngine` projection — *survival*
  = fraction of the horizon reached before a deficit (1.0 if none), *retention* = how much of the starting
  buffer remains at the projection's trough. Never-deficit + kept buffer → high; early/deep deficit → low.
- **Forecast DTI (%)** = `(recurring obligations + requested-loan servicing) / monthly income`. The request has
  no rate/term, so servicing is amortised at a standard product term (60 mo) + rate (9%/yr) via the §5.3
  annuity formula (real product params, like `LoanCalculator`'s rate input). Income + recurring-obligation
  definitions are **copied from `StressScoreCalculator`'s §5.1 aggregation** so DTI agrees with the client's
  stress score. Zero income → 200% cap → BAD.
- **Income stability (%)** = the §5.1 income-consistency sub-score × 100 (direct reuse of
  `StressScoreCalculator`).
- **12-month default probability (%)** = `0.5·staminaRisk + 0.3·leverageRisk + 0.2·stabilityRisk`, scaled to a
  60% ceiling. **This is the one purely judgment-driven curve** and is the natural place to retune risk
  appetite — it does **not** affect the verdict (§5.5 fixes that on stamina + DTI alone), only the reported
  risk figure + future portfolio NPL metrics.
- **Risk tier** = verdict-aligned letter: OK→A, WARN→B, BAD→C.
- **`MockDataSeederTest` change (test-isolation, not schema):** `loan_applications.client_ref` is a new sibling
  aggregate with a `NO ACTION` FK to `clients` (kept NO ACTION deliberately — a live application should block
  deleting its client, and synthetic applicants have no client to cascade from). The test's "delete all
  personas" clean-slate step therefore had to clear it first, exactly as it already clears `stress_scores` /
  `rescue_events`. Added one `deleteAllInBatch()` + updated the comment.
- **Persona id gotcha:** DESIGN §11 writes `client_002_tech`, but the seeded `external_id` (from the data-mock
  filename) is **`client_002_tech_bro`**. The seeder uses the real id; the mismatch cost one boot to catch
  (the seeder skipped gracefully with a warning rather than crashing).
