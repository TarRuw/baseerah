# Session Handoff — Step 3.4: GenAiClient interface + MockGenAi + chat/invoice endpoints

**Date:** 2026-07-03
**Status:** completed

## What Was Done

Introduced the pluggable conversational-AI seam (FR-03, DESIGN §2/§3/§5/§6/§9): a `GenAiClient` interface
with a deterministic, key-free `MockGenAi` default selected by `GENAI_PROVIDER`, plus the chat and
invoice-parse endpoints — the AI half of the Step 3.5 Simulate screen.

- **`genai.GenAiClient`** (interface) — `chat(ChatContext, String) → ChatReply` and
  `parseInvoice(byte[]) → InvoiceParseResult`. The three package domain records are declared on the
  interface per the step's file list: **`ChatContext`** (grounding telemetry — `profileLabel`,
  `currentScore`, `zone`, `monthlyIncome`, `monthlyOutflow`, plus a `surplus()` helper), **`ChatReply`**
  (`reply`), **`InvoiceParseResult`** (`merchant`, `amount`, `category`, `suggestedAction`). No Spring/JPA
  on the interface — all grounding data arrives via `ChatContext`.
- **`genai.MockGenAi`** (plain class, wired by the config) — deterministic, telemetry-grounded, **no key,
  no clock, no randomness**. `chat` keyword-routes: car / lease / vehicle / `2100` / `2,100` and Arabic
  سيارة / تأجير / مركبة / إيجار / ٢١٠٠ → the **2,100 SAR lease scenario** (payment as a % of the client's
  real surplus, current score/zone, a comfortable-payment cap = 0.5 × surplus, mirroring
  `LoanCalculator`'s §5.3 fraction); anything else → the **24-month cash-flow analysis** referencing the
  client's telemetry. `parseInvoice` returns a deterministic, clearly-labelled stub (no OCR — DESIGN §9),
  independent of the image bytes.
- **`genai.GenAiConfig`** — `@ConditionalOnProperty` on `baseerah.genai.provider`: `mock`
  (`matchIfMissing=true`) → `MockGenAi`; `remote` → **`RemoteGenAiPlaceholder`** (a package-private nested
  class that fails fast with "reserved for Phase 7 (Step 7.2)"). Exactly one bean is created, so callers
  inject a single `GenAiClient` and never see a concrete type (Global Rule).
- **`genai.ChatService`** (`@Service`) — resolves the client (reusing `ClientService`'s single 404
  contract), builds `ChatContext` from `StressScoreService.latestFor(...)` (score + zone) and mean monthly
  inflow/outflow over a trailing 90-day window, then delegates to the injected `GenAiClient`. Also
  `Clock`-injectable (package ctor) matching the Phase-2/3 convention.
- **`genai.ChatController`** (thin) — `POST /clients/{id}/chat` (`@Valid` body `ChatRequest{@NotBlank
  message}` → `{reply}`) and `POST /clients/{id}/chat/invoice` (multipart part `image` → parsed-action
  stub), both wrapped in the shared `ApiResponse` envelope.
- **`application.yml`** — added `baseerah.genai.provider: ${GENAI_PROVIDER:mock}` (env-overridable,
  offline-safe default).
- **Tests** — `MockGenAiTest` (pure, no Spring/DB): routing (car/2100/Arabic/vehicle → scenario; else →
  24-month), grounding assertions pinned to real numbers (score 62, surplus 4,300, 49%, cap 2,150), the
  non-positive-surplus branch, determinism (identical input → identical output), the invoice stub, and a
  nested `ApplicationContextRunner` block proving **`mock` resolves with no provider property and no API
  key**, `mock` explicit resolves `MockGenAi`, and `remote` resolves the reserved placeholder.

## Decisions Made

- **Added `ChatService`, which is not in the step's representative file table.** The same step mandates
  "thin controllers" and the project puts all logic in services (`ClientService`, `StressScoreService`,
  `ForecastService`, `LoanCalculator`). Context-building (windowing + aggregation) is service-layer work,
  so a service is the convention-correct home. This is not new scope — it only orchestrates this step's own
  endpoints — and keeps the controller thin as instructed.
- **Grounding telemetry = simple monthly means, not the loan package's cadence detector.** `MonthlyCashFlow`
  divides total credit/debit over the 90-day window by the number of distinct months. I deliberately did
  **not** reuse `LoanCalculator.deriveTelemetry`'s recurring-cadence logic: it is `private` to the completed
  Step 3.3 (AGENT.md: don't modify prior steps), and re-deriving it a third time would duplicate code the
  3.3 handoff already flagged for future extraction. Monthly means are sufficient to ground a conversational
  reply and keep this step self-contained. (If a third consumer ever needs the cadence detector, promoting
  it to a shared `common`/`transaction` utility — as the 3.3 handoff suggested — remains the right move.)
- **`RemoteGenAi` is a fail-fast placeholder inside `GenAiConfig`, not a new file.** The step says to wire
  the `remote` switch now but not implement the remote call. A nested `RemoteGenAiPlaceholder` throwing
  `UnsupportedOperationException` satisfies the switch (and is unit-tested) without adding an unlisted file
  or a half-built adapter. Phase 7 / Step 7.2 replaces it with the real streaming adapter.
- **English reply text; Arabic keyword *matching* only.** The interface signature `chat(ChatContext,
  String)` carries no locale, so the mock emits English and routes Arabic keywords to the scenario (exactly
  what the step's acceptance requires). Localised Arabic reply text is deferred to the remote adapter / a
  later step — noting the prototype does switch on `lang`, so Step 3.5 or Phase 7 may want to thread
  `Accept-Language` through if Arabic AI copy is desired in the demo.
- **`GENAI_PROVIDER` bound via `baseerah.genai.provider: ${GENAI_PROVIDER:mock}`.** Keeps the internal
  property under the project's `baseerah.*` namespace (like `baseerah.seed.*`) while honouring the exact
  env-var name the step specifies; `matchIfMissing=true` on the mock bean is a belt-and-suspenders default.

## Note (pre-existing, not caused by this step)

The recurring `MockDataSeederTest` FK fragility resurfaced. Multiple `@SpringBootTest` classes share one
real Postgres with no cross-class ordering guarantee; score-reading web tests (`StressScoreControllerTest`,
`LoanControllerTest`) lazily persist `stress_scores` rows, and when `MockDataSeederTest` later deletes the
personas the `stress_scores_client_id_fkey` (`ON DELETE NO ACTION`) rejects the delete. Adding this step's
two green test classes merely reshuffled execution order enough to surface it. **Proven pre-existing:** the
seeder test passes alone after a truncate, and no other class fails. The proper fix (cascade the FK via a
new Liquibase changeset, or truncate `stress_scores`/`forecasts` in the seeder test teardown) is still out
of scope for a Phase-3 feature step; the shared DB was left truncated for the next session.
