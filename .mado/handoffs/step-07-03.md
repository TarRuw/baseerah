## What Was Done

Profiled the hot analytics paths across all 5 seeded personas, fixed the two real N+1s + the report's
double-projection that profiling surfaced, added the one missing index, and added a full-surface 5-persona
integration suite that asserts the §11 expectations and the < 2.5s NFR on a warm run.

### 1. Profiling (baseline)
Measured end-to-end latency via MockMvc against the live seeded Postgres. Data volumes are modest (largest is
`client_002_tech_bro` at 300 txns, 1 account each), so warm latency is single-digit ms everywhere — the budget
risk is cold-start/JIT, not steady state. Findings that drove the fixes:
- **N+1 — bank portfolio:** `BankService.portfolio()` called `accountTokensFor` once per applicant → one
  `accounts` query per monitoring row.
- **N+1 — challenge list:** `ChallengeService.listForClient()` fetched each challenge's progress row
  separately → one `challenge_progress` query per challenge.
- **Repeated recomputation — bank report:** `ForecastEngine.project(clientRef, 365)` computed **twice** per
  request (once in `UnderwritingService` for stamina, once in `BankService.monthlyCashFlow` for the chart);
  `RiskPolicyRepository.loadPolicy()` and `findById` also called twice.
- **Indexes:** every analytics query pattern was already covered by 001–006 **except** the underwriting
  queue's `order by created_at`.

### 2. Fixes (all business logic stays in services; everything computed from seeded telemetry)
- **`CachingForecastEngine`** (new, `@Primary implements ForecastEngine`) — a memoizing decorator **in front of
  the `ForecastEngine` interface** (exactly where the step wanted the seam — a Python sidecar inherits the
  cache for free), keyed by `(clientId, horizonDays, today)`. Eliminates the report's double-projection and
  makes warm repeats instant. **Invalidation story:** a projection is a pure function of the (frozen, seed-time
  immutable — DESIGN §10) telemetry and the anchor day, so entries never need eviction within a run; including
  `today` in the key keeps it correct across a midnight rollover. Hand-rolled (ConcurrentHashMap) to match the
  project's zero-extra-dependency ethos (no `spring-boot-starter-cache` on the classpath).
- **Bank portfolio N+1 → batched.** New `AccountRepository.findByClientIdIn` + a batch
  `BankComplianceMapper.accountTokensFor(Collection<UUID>, RiskPolicy)` that resolves every facility's tokens in
  **one** query and groups in memory. Critically, the mapper stays the *sole* account→token choke-point (the
  leak-guarantee backstop from Step 7.1) and still honours the tokenization toggle. `portfolio()` batch-loads
  once and passes each row its pre-resolved tokens.
- **Challenge list N+1 → batched.** New `ChallengeProgressRepository.findByChallenge_IdIn` + a
  `toDto(Challenge, ChallengeProgress, Locale)` overload; `listForClient` loads all progress in one query and
  maps in memory. (Grouping keys use `getX().getId()` on the lazy proxy — reads the FK without a per-row
  SELECT, so the batch stays a single round-trip.)
- **`008-perf-indexes.sql`** — `idx_loan_applications_created_at` for the queue ordering. The changeset header
  documents that the analytics-window indexes already existed in 001–006, so this closes the one uncovered
  read pattern rather than duplicating.
- **`StressScoreSnapshotWriter`** — its `Clock` constructor widened package-private → **public** so the IT can
  pin deterministic zones (no logic change).

## Key decision — §11 "tech WARNING" deviation (faithful engine, documented)

DESIGN §11 casts `client_002_tech_bro` as a **warning-zone** gauge. On the frozen `data-mocks/`, the score
computed from seeded telemetry is **OPTIMAL (92)** — and in fact **all five personas land OPTIMAL** at the
anchor window (family 94, tech 92, freelancer 84, student 73, vip 93), because each carries a healthy closing
balance and the score weights liquidity heavily. Rather than fabricate a WARNING (forbidden by the Global Rule
"no prototype magic numbers") or edit the read-only mock, the IT asserts the **true computed zones** and pins
`tech_bro`'s §11 role to what it really is — the **large-volume latency canary** (proven at ~2ms). This is the
same faithful-engine-over-prototype-values realignment the project already made for the freelancer's
`alertRaised` in Step 4.1. **Flagged for the user** — if a demoable WARNING persona is wanted, it needs a
telemetry change to a mock (out of scope here / read-only) or a DESIGN §11 update, not a hardcoded score.

## Determinism note
Stress zones depend on the trailing window ending "today", so the IT **pins a fixed clock (2026-06-30, inside
the Jan–Jun mock window)** when pre-writing each persona's snapshot — same technique as
`GamificationControllerTest`. Rescue/forecast assertions use clock-robust facts (freelancer always eventually
deficits; family stays solvent) and run on the system clock like `RescueControllerTest`.
