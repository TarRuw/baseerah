## What Was Done

Implemented the gamified micro-saving backend (FR-09/10, DESIGN §5.6) end-to-end behind the service layer:
`005-gamification` migration, three JPA entities + repositories, `ChallengeService` (anomaly-driven
generation), `RewardsService` (Akhtar Points balance / tier / guarded claim), and a boot `ChallengeSeeder`.

- **Migration** `005-gamification.sql` — `challenges`, `challenge_progress`, `rewards_ledger` per DESIGN §4.2
  (uuid PKs via `gen_random_uuid()`, `numeric(14,2)`, `timestamptz`). Idempotency keys: `unique
  (client_id, code)` on challenges, `unique (challenge_id)` on progress. `pct` has a DB `check (0..100)`.
  **FKs are `ON DELETE CASCADE`** to clients/challenges (see decision below).
- **Entities** (`com.baseerah.gamification`) — `Challenge`, `ChallengeProgress` (one-to-one, holds
  `pct`/`claimed`/`claimed_at`; `markClaimed` is the only claim mutator), `RewardsLedgerEntry` (append-only,
  `@CreationTimestamp`). `RewardTier` enum (BRONZE/SILVER/GOLD/PLATINUM) with a `forBalance(int)` factory.
  All map their tables exactly — verified by `ddl-auto: validate` booting cleanly.
- **Repositories** — `ChallengeRepository` (`findByClient_Id`, `findByClient_IdAndCode`),
  `ChallengeProgressRepository` (`findByChallenge_Id`), `RewardsLedgerRepository` (`findByClient_Id` +
  a `@Query sumPointsByClientId` that `COALESCE`s the empty sum to 0 — balance is computed in the DB).
- **`ChallengeService`** — pure `static detect(List<Transaction>)` (deterministic, no clock/DB → unit-tested
  directly) + `@Transactional generateForClient(UUID)` shell that reads the injectable `Clock`, loads the
  trailing 90-day window, and upserts challenges + progress. Idempotent per `(client_id, code)`; a re-run
  refreshes sizing/progress but **never touches claim state**.
- **`RewardsService`** — `summaryFor(clientId)` (ledger sum → balance + `RewardTier`) and a doubly-guarded
  `claimChallenge(clientId, challengeId)`: rejects an unknown/foreign challenge (`NotFoundException`), an
  already-claimed one, and an incomplete one (`IllegalStateException`) — writing no ledger row on rejection;
  on success flips `claimed`/`claimed_at`, appends one ledger row, returns awarded points + new balance/tier.
- **`ChallengeSeeder`** — `@EventListener(ApplicationReadyEvent)`, so it runs strictly **after**
  `MockDataSeeder` (an `ApplicationRunner`) without adding an `@Order` to that prior-step file. Generates for
  all seeded clients on boot; idempotent.
- **Tests** — `ChallengeServiceTest` (pure, reads `data-mocks/`): student challenges derive from real
  observed categories, targets positive, `pct ∈ [0,100]`, and a completed (`pct==100`) goal exists.
  `RewardsServiceTest` (`@SpringBootTest` + `@EnabledIf(postgresReachable)` + `@Transactional` rollback,
  same convention as `RescueServiceTest`): claim awards points + writes exactly one ledger row + flips
  claimed + tier stays consistent; incomplete and re-claim are both rejected with no ledger write.

## Decisions Made (the step left these open — documented here for 5.2/5.3 and reviewers)

- **Anomaly heuristic (data-driven, not persona-hardcoded).** Over each client's trailing 90-day window,
  DEBIT txns are bucketed by `Category.resolve()` (the §4.1 enum-with-fallback). A category is a spending-cap
  candidate only if it is **discretionary** (a documented `DISCRETIONARY` `EnumSet` — essentials like
  groceries/telecom/utilities/healthcare/transportation/education/business and any unmodelled `OTHER` are
  excluded, so unknown feed categories are never capped) **and** a genuine habit (`≥ 8` txns in the window).
  The client's **dominant** discretionary category (most txns) drives the goals. *Which* category a client
  gets is entirely their own telemetry — the student surfaces `RESTAURANTS_DINING` because that is genuinely
  their habit (221 small purchases); the family/freelancer surface no discretionary habit and honestly get
  no challenges.
- **Three challenge archetypes + target sizing (all from the client's own numbers, no prototype constants):**
  - `WELCOME_MINDFUL` — retrospective starter reward. `target = 50% of tracked category spend`, `current =
    tracked spend` → complete on generation. **This is the demoable "claim me" goal** (guarantees the
    student has a completed-unclaimed challenge deterministically, rather than hoping raw data hits pct=100).
  - `CAP_<category>` — forward monthly cap `= 80% of the client's average monthly category spend`;
    progress = "staying under the cap" (100 if this month ≤ cap, else scales down by the overspend).
  - `SAVE_MICRO` — micro-saving target `= 10% of the client's monthly discretionary burn`; progress = saving
    realised this month vs the client's own average (0 for a steady spender like the student).
  - Targets rounded to the nearest SAR 100 (floored at 100 so a goal is never SAR 0). Reward points are
    proportional to the SAR target (`target/10`, clamped to `[50,250]`) — a product parameter, not a demo
    number.
- **Reward-tier thresholds (consumer gamification tier, distinct from the bank-side `risk_tier` in §4.2):**
  BRONZE `[0,150)`, SILVER `[150,400)`, GOLD `[400,800)`, PLATINUM `[800,∞)`. Everyone starts BRONZE; a
  first claim (50–250 pts) can lift a new saver toward SILVER. Kept deliberately separate from underwriting
  risk (no shared mapping) per the step's Notes.
- **FKs are `ON DELETE CASCADE`** (challenges/ledger → clients, progress → challenges). The
  `stress_scores`/`rescue_events` aggregates use NO-ACTION FKs, so `MockDataSeederTest` clears them by hand
  before deleting a client. Rather than modify that prior-step test to also clear the two new tables, I made
  gamification rows follow the client lifecycle — a client delete self-cleans them. This also fixed the
  4 pre-existing tests that broke on the first run (the new FKs were blocking that test's client deletes).
- **Determinism vs the clock.** `detect()` is pure so tests are stable; only `generateForClient` (window
  load) and `claimChallenge` (claimed_at) read the clock. `RewardsServiceTest` builds a fixed-clock
  `ChallengeService` (anchored `2026-06-30`) so detection is date-independent; the boot `ChallengeSeeder`
  uses the system clock (fine on this 2026-07-04 machine, same wall-clock assumption as stress/rescue).
