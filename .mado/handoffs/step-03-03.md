# Session Handoff — Step 3.3: LoanCalculator + loan-simulate endpoint

**Date:** 2026-07-03
**Status:** completed

## What Was Done

Delivered interactive loan affordability (FR-05, DESIGN §5.3): a `LoanCalculator` mirroring the prototype's
math exactly, with income/essentials derived from the client's own telemetry (no prototype constants),
exposed via `POST /api/v1/clients/{id}/loan-simulate` for the Step 3.5 Simulate-screen sliders. Loan
simulation is stateless — no new table (the bank-side `loan_applications` table is a separate Phase-6
concern).

- **`loan.LoanCalculator`** (`@Service`) — functional core / imperative shell, matching the
  Phase-2/3.1 convention:
  - **Core** `compute(principal, rate, term, income, essentials, currentScore)` (package-visible, pure →
    unit-tested with no Spring/DB). Reproduces §5.3 exactly: `r = rate/100/12`;
    `installment = P*r/(1-(1+r)^-n)` with the **`r == 0` → `P/n`** edge case; `surplus = income -
    essentials`; `DTI = (essentials+installment)/income`; verdict thresholds **0.50 / 0.85 of surplus**;
    score impact `strain = installment/surplus`, `proj = currentScore - max(0,(strain-0.35))*90`, **clamped
    to `[9,84]`**. Returns whole-SAR `installment`/`total` (HALF_UP) while keeping intermediate math
    precise (`total = precise installment × term`, then rounded).
  - **Shell** `simulate(clientId, request)`: resolves the client via `ClientService.requireClient` (single
    404 contract), reads the current score via `StressScoreService.latestFor(...).score()` (the projected
    score is derived from the client's real current score, not the prototype's hardcoded 62), loads the
    trailing **90-day** transaction window, derives telemetry, and calls the core.
  - **`deriveTelemetry(window)`**: income & essentials from the seeded transactions — see Decisions.
- **`loan.LoanController`** — thin `POST /clients/{id}/loan-simulate`; `@Valid @RequestBody` →
  `LoanSimulateRequest {principal, rate, term}`; wraps the result in the shared `ApiResponse` envelope.
- **DTOs** — `LoanSimulateRequest` (`@Positive principal`, `@PositiveOrZero rate`, `@Positive term`) and
  `LoanSimulateResponse {installment, total, dti, dtiColor, verdict, verdictColor, projectedScore}`.
- **Tests** — `LoanCalculatorTest` (pure, 4 tests: known-input, `r=0`→`P/n`, clamp floor `9`, clamp ceiling
  `84`) and `LoanControllerTest` (web slice against live seeded Postgres, gated on `postgresReachable` like
  the other controller tests: envelope shape, telemetry-independent `installment`/`total` asserted exactly,
  DTI/verdict/colours/score asserted well-formed and in-band, plus 404 + 400 envelopes).

## Decisions Made

- **`dtiColor` banding recovered from the prototype.** DESIGN §6's endpoint contract (line 197) lists only
  `installment/DTI/verdict/scoreImpact` — it did **not** pin DTI colour thresholds. The step's response
  shape adds `dtiColor`, so I took the banding from the prototype (`design/Baseerah.dc.html` line 845):
  `dtiPct < 70 → green, < 90 → orange, else red`, and **remapped the prototype's pastel hexes**
  (`#7fdcae/#ffd27a/#ff9d97`) to the canonical **DESIGN §8 palette** (`#1D9E63/#E5A63A/#E0574F`) so the
  served colour matches the Step 3.5 Flutter theme tokens. `verdictColor` uses the same palette per §5.3.
- **Recurring detection = cadence-gap (design decision the step left open; user delegated the choice).**
  Chosen over the month-coverage approach (StressScoreCalculator's) because it is **more accurate for
  loan essentials**: a grocery store visited every month has high month-coverage but a tiny median gap, so
  cadence-gap correctly rejects it as discretionary — exactly what "essentials" (rent/instalments/subs)
  must exclude. This mirrors `HeuristicForecast`'s §5.2 concept (the step's "reuse from Step 3.1"
  guidance). Group by `direction + category + cleansed description`; recurring when **≥2 occurrences** with
  **median gap 30 ± 7 days**; each group contributes its **mean per-occurrence amount** (its typical
  monthly magnitude). `income` = Σ recurring inflows' typical amounts; `essentials` = Σ recurring debits'.
- **Re-expressed rather than extracted.** The forecast engine's recurrence logic is `private`; extracting
  it into a shared helper would modify the completed, tested Step 3.1 code (AGENT.md: don't modify prior
  steps unless the step says to). Per the step's *"reuse … where practical"* wording, I re-expressed the
  cadence test compactly in the loan package (a private `RecurringFlow` mirroring the forecast engine's).
  **Follow-up worth considering:** if a third consumer needs it, promote the cadence detector to a shared
  `common`/`transaction` utility and have all three use it.
- **Irregular-income fallback.** When no clean recurring inflow is detected (e.g. `client_003_freelancer`,
  whose `INFLOW_BUSINESS` is bursty/undated), `income` falls back to **mean monthly total credit** so the
  DTI ratio stays meaningful instead of dividing by zero. `essentials` is `0` when no recurring debit.
- **Degenerate guards.** `surplus <= 0` → every verdict comparison fails → "Not affordable" + projected
  clamps to `9`; `income <= 0` → DTI defaults to `1.0`. These can't arise for the seeded personas but keep
  the endpoint total.

## Note (pre-existing, not caused by this step)

The same `MockDataSeederTest` FK fragility flagged in the Step 3.1 handoff resurfaced: the shared local
Postgres held `stress_scores` rows (created lazily by Phase-2/3.3 score reads), and the seeder test's
persona delete violates `stress_scores_client_id_fkey` (ON DELETE NO ACTION). **Proven pre-existing:** an
isolated `MockDataSeederTest` run — no loan/stress tests in it — failed identically, so the blocking rows
predated this session. Applied the documented remedy (`truncate stress_scores, forecasts;`) and the full
suite went green. This remains a latent test-isolation bug the project may want to fix properly (cascade
the FK via a new Liquibase changeset, or purge `stress_scores`/`forecasts` in the seeder test teardown) —
still out of scope for a Phase-3 step.
