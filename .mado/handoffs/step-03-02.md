## What Was Done

### Backend

- **`forecast.ForecastResponse`** (record) — the enveloped wire DTO:
  `{ points: [{date, balance}], deficitDate (nullable), minProjectedBalance, trend }`. Maps from the engine's
  domain `ForecastResult` via `from(...)`; the JPA/entity layer never leaks (Global Rules). Nested
  `Point(date, balance)` record and a `Trend { UP, FLAT, DOWN }` enum.
- **`forecast.ForecastService`** (`@Service`, `@Transactional(readOnly=true)`) — resolves the client through
  `ClientService.requireClient` (reusing the single 404 contract), calls the injected **`ForecastEngine`
  interface** (never the concrete `HeuristicForecast`), and maps to the DTO. Added as a thin service so the
  controller stays truly thin and the layering matches `StressScoreService` (`controller → service → engine`).
- **`forecast.ForecastController`** — `GET /api/v1/clients/{id}/forecast?horizonDays=30`. `@Validated` +
  `@Min(1) @Max(365)` on `horizonDays` → out-of-range becomes the shared `400 VALIDATION_ERROR` envelope via
  the Step 0.4 handler. The same param serves 30 (home) and 90/180/365 (3/6/12-month) horizons. Wraps the
  result in `ApiResponse.ok(...)`.
- **`ForecastControllerTest`** (MockMvc, live Postgres, skips cleanly when DB down) — envelope shape +
  `horizon+1` point count; 90/180/365 horizons; **< 2.5 s NFR** for `client_002_tech_bro` at 365 days; 404
  for an unknown id; 400 for `horizonDays=0`; and the family/freelancer no-base-deficit contrast.

## Decisions Made

- **`trend` derivation (documented for Phase 6 bank-report reuse).** Rule, implemented in
  `ForecastResponse.deriveTrend`:
  1. If `deficitDate != null` → **DOWN** (the line is by definition heading below zero; alert red dominates).
  2. Else compare last vs first projected balance: rise > **1% of |start|** → **UP** (green); fall > 1% →
     **DOWN** (red); within the ±1% deadband → **FLAT** (orange). The relative deadband stops near-flat
     projections from flickering on noise. Colours map to DESIGN §8 tokens.
- **No persistence in this step.** The Step 3.1 handoff pencilled in writing rows to the `forecasts` table
  here, but step-03-02.md's tasks/acceptance-criteria do not require it, and AGENT.md says follow the step
  file (don't add scope). The projection is computed on demand — the engine only walks a 90-day window +
  the horizon, so a single request is ~15 ms (see Verification), far inside the 2.5 s NFR without a
  snapshot cache. **Deferred:** persisting/serving historical `forecasts` rows (jsonb `points`) can be a
  later step if a use-case needs it (e.g. audit trail / bank report history).
- **Thin `ForecastService` added** (not literally listed in the step's file table) to honour the hard Global
  Rule "business logic lives in services, never controllers." The acceptance criterion "controller calls the
  `ForecastEngine` interface (not the concrete class)" is satisfied in spirit and letter: nothing references
  `HeuristicForecast`; the dependency is the `ForecastEngine` interface, injected into the service.
- **Chart y-domain is anchored at 0** (domain = `[min(0, dataMin), max(dataMax)+8%]`) so the dashed
  zero-line is always meaningful — deliberately trading some vertical "zoom" for the deficit-awareness
  narrative the design cares about (§7.1).
- **No new l10n strings.** The chart reuses existing ARB keys (`forecast30Title`, `forecastComingSoon`,
  `loadError`, `retry`) and renders only locale-formatted *numbers* via `Fmt`, so no `flutter gen-l10n`
  regeneration is required to compile.

## ⚠️ Continuation of the Step 3.1 deviation (user-approved)

The acceptance criterion "…a deficit marker at `deficitDate` for the **freelancer** persona; no deficit
marker for the **family** persona" is **not satisfiable on the frozen mock data** and is the same issue the
user already ruled on in Step 3.1 ("Faithful engine, realign test"). Verified live this session:

- **family** (`client_001_family`) 30-day → `trend=UP`, `deficitDate=null` (recurring salary lifts the
  balance 107,827 → 121,974). **No marker** — matches the criterion.
- **freelancer** (`client_003_freelancer`) 30-day → `deficitDate=null`, `min=236,214` (a ~239k accumulated
  buffer that only declines slowly, no recurring inflow → `trend=DOWN` but never crosses zero in-horizon).
  **No base-forecast marker.**

The chart's deficit-marker path is implemented and correct — it renders **iff** `deficitDate` is non-null.
The freelancer's headline deficit is a **Phase-4 what-if scenario** layered on the base projection (already
flagged in the Step 3.1 handoff), not a property of the base forecast. Not re-asking the user, as this is the
identical, already-decided question.
