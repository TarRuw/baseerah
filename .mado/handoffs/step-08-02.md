# Session Handoff — Step 8.2: Responsive max-width / phone-frame layout for web & desktop

**Date:** 2026-07-07
**Status:** completed — Phase 8 (32/35, project ~91.4%). Closes QA finding **UI-01**. Verified by the full
**21/21 Flutter test suite** (`flutter analyze` clean), including **2 new `responsive_frame_test` cases** that
assert the width cap on a wide surface and the no-op at phone width, plus the existing shell smoke test which
now boots the *framed* shell.

## What Was Done

On a wide (desktop web) viewport the consumer shell had no width constraint, so the 1:1 `AspectRatio` stress
gauge scaled to full content width and became equally tall (~1260 px at 2519 px wide), burying the score,
zone badge, chart, stat cards and accounts ~2 screens below the fold. Fixed with two independent layers.

- **Shell-level phone frame (Task 1) — new `frontend/lib/shell/responsive_frame.dart`:** a reusable
  `ResponsiveFrame` widget wraps each shell's `Scaffold`. Via `LayoutBuilder`, when the viewport is **wider
  than `maxWidth`** it caps the child to `maxWidth` and centers it; at or below `maxWidth` it returns the
  child unchanged — a true **no-op on real phones**, so the mobile layout is untouched (single-constant
  breakpoint = the cap itself).
  - **Consumer shell** (`_ConsumerShell`): `maxWidth = phoneFrameMaxWidth` (460), `framed: true` → the capped
    column floats as a rounded phone frame (`ClipRRect` @ `radiusPhoneScreen`) over the existing
    `BaseerahTokens.desktopBackdrop` gradient. Reuses design tokens the theme already shipped for exactly
    this — the design system anticipated a desktop phone frame.
  - **Bank shell** (`_BankShell`): `maxWidth = bankFrameMaxWidth` (1400), `framed: false` → a plain centered
    cap that only reins in ultra-wide monitors; the sidebar+content **desktop layout is preserved** (not
    forced to phone width), per spec.
- **Gauge bound (Task 2) — `stress_gauge.dart`:** wrapped the `AspectRatio` in
  `Center → ConstrainedBox(maxWidth/maxHeight = gaugeMaxDiameter 340)` so the gauge can never dominate the
  screen even inside a wider container (defense in depth). Marker animation and 3-zone colouring are
  untouched.
- **Tokens (in the single theme file):** added a "Layout / responsive" section — `phoneFrameMaxWidth = 460`,
  `bankFrameMaxWidth = 1400`, `gaugeMaxDiameter = 340`. No magic numbers scattered into widgets.
- **Test (Task 3):** `test/responsive_frame_test.dart` boots the full app and measures `ConsumerBottomNav`
  width (a clean proxy for the shell frame width): **460 at a 1400×1000 surface** (cap active) and
  **390 at a 390×844 surface** (no-op, no mobile regression).

## Key decisions

- **Wrap the whole `Scaffold`, not just `body`.** Wrapping only the routed content would leave a full-width
  teal AppBar and full-width bottom nav around a narrow floating body — visibly broken on desktop. Wrapping
  the Scaffold makes toolbar + body + bottom nav one coherent phone frame.
- **Breakpoint = the cap (460).** One constant instead of a separate breakpoint: any viewport ≤460 (all real
  phones) is full-bleed; wider gets framed. Simplest correct behavior for a mobile-first app.
- **Gauge cap 340, above the phone-content width.** At phone width the Home gauge slot is ~324 px
  (412 − 40 screen padding − 48 gauge padding), already below 340, so the cap is a **no-op at phone size** and
  the mobile gauge is pixel-identical; it only bites on desktop. Guarantees acceptance criterion "no visual
  regression at phone width".
- **Reused `desktopBackdrop` for the consumer gutter.** Token was authored "(bank portal)" but is generic; no
  need for a second near-identical gradient.

## Notes / gotchas

- **Screen-level tests were unaffected by design.** `simulate/goals/bank` tests mount their screen under a
  bare `MaterialApp(home:)`, bypassing `_ConsumerShell`/`_BankShell`; only `widget_test.dart` exercises the
  shells (at the 800 px default surface, so the consumer frame is active there — and it still passes).
- **Browser/release-build visual confirmation is deferred to Step 8.5 (FU-02),** per this step's own spec
  note. The width cap, gauge bound, and mobile no-op are **deterministically test-proven** here; the
  above-the-fold screenshot on a real phone/emulator and a `--release` web build belong to the 8.5 close-out.
- `ResponsiveFrame` is generic and now available to any future full-bleed screen that needs a desktop cap.
