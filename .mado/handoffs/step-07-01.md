# Session Handoff — Step 7.1: SAMA tokenization + NDMO flags + risk-policy enforcement

**Date:** 2026-07-06
**Status:** completed — **live-verified** against the running Postgres (`baseerah-postgres` up on `localhost:5432`,
JDK 21): full suite **87 tests pass** (was 79; +8 new), migration `007` shows `EXECUTED` in `databasechangelog`,
and `accounts.tokenized_account_id` is now `is_nullable = NO`.

## What Was Done

Honoured the DESIGN §9 compliance NFRs on the bank side (FR-08), all *simulated* per BRD out-of-scope — no live
SIMAH/SAMA sync, no production crypto — but provably leak-free and policy-driven.

- **Migration `007-tokenize-accounts.sql`** — backfills any null `tokenized_account_id` and makes the column
  `NOT NULL`. The backfill token is byte-for-byte identical to `MockDataSeeder.tokenize()`
  (`'TKN-' || upper(substring(encode(digest(coalesce(external_id, id::text),'sha256'),'hex') from 1 for 24))`),
  so a seeder-tokenized row and a SQL-backfilled row produce the same token. pgcrypto (`digest`/`encode`) is
  already enabled by 001.
- **`BankComplianceMapper`** (new) — the single choke-point turning a linked consumer's accounts into the
  compliance surface a bank DTO may carry. The only method that maps an `Account` to a bank field
  (`accountTokensFor`) reads **exclusively** `getTokenizedAccountId()` — there is no code path to the raw
  `external_id`/UUID. Also owns the NDMO residency marker + export gate.
- **Bank DTOs extended:** `UnderwritingReportDto` gains `tokenizedAccounts` + `dataResidency` + `exportAllowed`;
  `PortfolioDto` gains top-level `dataResidency` + `exportAllowed` and per-row `tokenizedAccounts`. `BankService`
  reads the singleton policy and feeds these through the mapper (never a raw id).
- **Risk-policy enforcement in `UnderwritingService`:** injected `RiskPolicyRepository`; `generateReport` now
  overlays `enforcePolicy(report, staminaFloor, autoDeclineThreshold)` on top of the pure §5.5 core — forcing
  verdict `BAD` / tier `C` when `stamina < staminaFloor` **or** `forecastDti >= autoDeclineThreshold`. `assemble`
  stays pure (design thresholds only), so the verdict-band tests are untouched.
- **`CompliancePolicyTest`** (new, DB-gated) — scans serialized report + portfolio bodies against the DB's real
  raw account ids (no leak), asserts the tokenization/NDMO toggles change observable behaviour, and that a
  raised stamina floor / lowered DTI ceiling auto-declines. Plus two pure `enforcePolicy` unit tests in
  `UnderwritingServiceTest`.

## Key decisions

- **`auto_decline_threshold` is a forecast-DTI ceiling (%), not a default-probability threshold.** The step text
  says "default probability / risk metric", but the persisted default (71) matches the §5.5 `BAD_DTI_FLOOR`, the
  modelled PD is capped at 60% (so 71 could never trigger on PD), and `BankService.policyEligible` already treats
  it as a DTI ceiling. I stayed consistent with the established codebase semantics: **auto-decline when
  `stamina < staminaFloor` OR `forecastDti >= autoDeclineThreshold`** — the exact negation of `policyEligible`.
- **`tokenization == false` semantics (for the Settings screen copy, DESIGN §7.7):** the toggle changes the
  **token surface, never the leak guarantee**. Off → `tokenizedAccounts` is an **empty list**; Baseerah does
  *not* fall back to raw account ids. So Settings copy should read as "off hides the tokenized reference from the
  bank view", not "off reveals raw account numbers".
- **NDMO residency semantics:** `ndmoResidency == true` (default) → payload stamped `dataResidency = "KSA"` and
  `exportAllowed = false` (export path gated closed); `false` → `"UNRESTRICTED"` / `exportAllowed = true`.
  Flipping via `PUT /api/v1/bank/risk-policy` changes the stamp on the very next request (test-verified).
- **Encryption-layer note (DESIGN §9, simulated):** documented in the `BankComplianceMapper` class Javadoc — a
  real deployment would wrap the bank report/portfolio DTOs with at-rest envelope encryption (KMS-managed data
  key) and in-transit mTLS + payload signing. Not implemented for the MVP (no key management); the tokenization
  + residency controls are the demonstrable, leak-provable stand-ins, and the mapper marks where that boundary
  would sit.
- **Enforcement lives in the shell, not the pure core.** `enforcePolicy` is a package-visible static overlay
  applied in `generateReport` after `assemble`, so the §5.5 band unit tests keep driving the pure core with no
  policy dependency, while the bank's tunable guardrail is unit- and integration-tested separately.

## Notes / gotchas for the next session

- **Constructor signatures changed** (necessary consequence of the mandated `UnderwritingService`/`BankService`
  edits): `UnderwritingService` now takes a 5th arg (`RiskPolicyRepository`); `BankService` a 5th
  (`BankComplianceMapper`). `UnderwritingServiceTest` was updated to match (it constructs the service directly).
- **Frontend not touched** — the step's file list is backend-only. The new DTO fields (`tokenizedAccounts`,
  `dataResidency`, `exportAllowed`) are additive; existing Flutter models ignore unknown JSON keys, so nothing
  breaks. A later polish step could surface the residency marker / export gate on the Bank Settings or Report
  screen if desired.
- **VIP has stamina 100**, so it can't demonstrate the stamina-floor path (a perfect-stamina applicant correctly
  passes any valid floor ≤ 100). `CompliancePolicyTest` self-selects a linked applicant with a passing verdict
  and sub-100 stamina for that assertion; the DTI-ceiling path is demonstrated on the VIP.
- Seeded `loan_applications` are transient in the test DB (rolled back at context shutdown); `clients`/`accounts`
  persist. This is pre-existing behaviour and irrelevant to the compliance controls.
