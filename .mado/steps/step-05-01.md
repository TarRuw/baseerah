# Phase 5 — Step 5.1: ChallengeService + RewardsService + Flyway + seed challenges

## Objective

Implement gamified micro-saving (FR-09/FR-10, DESIGN §5.6): a `ChallengeService` that generates
challenges tailored to a client's real spending anomalies, and a `RewardsService` that manages the
client's Akhtar Points balance and risk tier, backed by the `challenges`, `challenge_progress`, and
`rewards_ledger` tables — so the endpoints (Step 5.2) and Goals screen (Step 5.3) have real data.

## Dependencies

- **Requires:** Step 1.3 — JPA entities/repositories for `clients`, `accounts`, `transactions` must
  exist and the 5 `data-mocks/` personas must be seeded into Postgres (challenges are derived from real
  seeded transactions).
- **Assumes:** Spring Boot + Postgres + Flyway skeleton from Phase 0 is running; the `com.baseerah` base
  package and Flyway migration location (`backend/src/main/resources/db/migration`) are in place; Global
  Rules apply (UUID PKs, `numeric(14,2)` money, `timestamptz`, business logic in services, `ddl-auto`
  never creates schema).

## Tasks

Complete in order:

1. **Flyway migration `V5__gamification.sql`**
    - Create the three tables per DESIGN §4.2:
        - `challenges` — `id (uuid pk)`, `client_id (uuid fk → clients)`, `code (text)`, `title (text)`,
          `subtitle (text)`, `icon (text)`, `target_value (numeric(14,2))`, `reward_points (int)`,
          `category_trigger (text)`.
        - `challenge_progress` — `id (uuid pk)`, `challenge_id (uuid fk → challenges)`,
          `current_value (numeric(14,2))`, `pct (int, 0–100)`, `claimed (boolean default false)`,
          `claimed_at (timestamptz null)`.
        - `rewards_ledger` — `id (uuid pk)`, `client_id (uuid fk → clients)`,
          `points_delta (int)`, `reason (text)`, `created_at (timestamptz default now())`.
    - Add a unique constraint on `challenges (client_id, code)` so challenge generation is idempotent
      per persona, and one on `challenge_progress (challenge_id)` (one progress row per challenge).
    - Place at `backend/src/main/resources/db/migration/V5__gamification.sql`. Schema comes from Flyway
      only (no `ddl-auto` creation, per Global Rules).

2. **JPA entities + repositories** — package `com.baseerah.gamification`
    - `Challenge`, `ChallengeProgress`, `RewardsLedgerEntry` entities mapping the three tables; entities
      never leave the service layer (mapped to DTOs in Step 5.2).
    - Repositories:
        - `ChallengeRepository`: `List<Challenge> findByClientId(UUID clientId)`,
          `Optional<Challenge> findByClientIdAndCode(UUID clientId, String code)`.
        - `ChallengeProgressRepository`: `Optional<ChallengeProgress> findByChallengeId(UUID challengeId)`.
        - `RewardsLedgerRepository`: `List<RewardsLedgerEntry> findByClientId(UUID clientId)` (used to sum
          the points balance).

3. **`ChallengeService` — DESIGN §5.6**
    - Implement in `com.baseerah.gamification`. Generate challenges tailored to the client's spending
      anomalies detected from seeded transaction categories over a trailing window:
        - Detect high-frequency / high-share discretionary categories (e.g. coffee, food-delivery,
          dining) and generate a **spending-cap** challenge with a `target_value` derived from the
          client's own spend (e.g. cap below their observed average), not a hardcoded constant.
        - Generate a **micro-saving** target challenge sized from the client's surplus/discretionary
          burn (per §5.6, tailored to the persona — e.g. the student persona).
        - Each generated challenge carries `code`, `title`, `subtitle`, `icon`, `target_value`,
          `reward_points`, and the `category_trigger` it was derived from.
    - Generation is idempotent per client via the `(client_id, code)` unique key (upsert; re-running does
      not duplicate challenges).
    - Compute/refresh each challenge's `challenge_progress` (`current_value`, `pct` in `[0,100]`) from
      current telemetry; a challenge is completable when `pct >= 100` and not yet `claimed`.
    - Category detection is data-driven (enum-with-fallback per DESIGN §4.1) — do not hardcode a fixed
      category list. Document the anomaly-detection heuristic and target-sizing formula in the handoff.

4. **`RewardsService` — DESIGN §5.6**
    - Implement in `com.baseerah.gamification`. Expose the client's **Akhtar Points** balance (sum of
      `rewards_ledger.points_delta` for the client) and derived **risk tier** (map the points balance to
      a tier — decide and document the tier thresholds).
    - `claimChallenge(clientId, challengeId)`: guarded so it only awards a **completed** (`pct >= 100`)
      and **unclaimed** challenge — reject double-claim and incomplete claims. On success: mark
      `challenge_progress.claimed = true` + `claimed_at = now()`, insert a `rewards_ledger` row with
      `points_delta = reward_points` and a `reason`, and return the updated balance + tier.

5. **Seed initial challenges/progress for the seeded clients**
    - On boot (idempotent, alongside/after the mock data seeder), run `ChallengeService` generation for
      the seeded personas so at least the student persona (`client_004_student`) has active challenges
      with progress — including at least one that is already completed (`pct = 100`, unclaimed) so the
      claim flow is demoable end-to-end. Re-running boot must not duplicate rows.

6. **Unit test — claim awards points + writes ledger**
    - Test that claiming a **completed** challenge increases the client's points balance by
      `reward_points`, writes exactly one `rewards_ledger` row, flips the progress to `claimed`, and
      updates the risk tier accordingly.
    - Test that claiming an **incomplete** challenge and **re-claiming** an already-claimed challenge are
      both rejected and write no ledger row.
    - Test that a persona's challenges are generated from real category anomalies (a challenge exists
      whose `category_trigger` matches a category actually present in that persona's seeded transactions).

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] `V5__gamification.sql` runs cleanly via Flyway; `challenges`, `challenge_progress`, and
      `rewards_ledger` exist with the §4.2 columns and the stated unique constraints.
- [ ] `ChallengeService` generates challenges from real category anomalies in the client's seeded
      transactions (each challenge's `category_trigger` corresponds to observed spending), with
      `target_value` sized from the client's own telemetry — no prototype magic constants.
- [ ] Challenge progress (`current_value`, `pct`) is tracked per challenge; `pct` is clamped to `[0,100]`.
- [ ] `RewardsService` returns the points balance (sum of the ledger) and a derived risk tier.
- [ ] Claiming a **completed, unclaimed** challenge increases points by `reward_points`, writes one
      `rewards_ledger` row, sets `claimed = true` + `claimed_at`, and updates the risk tier.
- [ ] Claiming an incomplete challenge, or re-claiming a claimed one, is rejected and writes no ledger row.
- [ ] Boot seeds challenges/progress for the seeded personas (student persona has active challenges incl.
      one completed-unclaimed); re-running boot is idempotent (no duplicate rows).
- [ ] Unit tests pass; all existing tests still pass; no `ddl-auto` schema creation is introduced.

## Notes

- Anomaly detection and target-sizing are deliberately under-specified in DESIGN §5.6: choose the
  discretionary categories to cap and the micro-saving target formula from the client's own transactions,
  and document the heuristic + risk-tier thresholds in the handoff so Step 5.2/5.3 and reviewers can
  reason about the numbers. No prototype magic numbers copied as constants (Global Rules).
- Keep challenge progress/anomaly computation deterministic over its input window so tests are stable;
  the seeder and claim path are the only pieces that touch the clock and mutate the ledger.
- Risk tier here is the consumer-facing gamification tier surfaced on the Goals screen; keep it distinct
  from the bank-side `risk_tier` on loan applications (§4.2) unless a shared mapping is deliberately
  chosen and documented.
