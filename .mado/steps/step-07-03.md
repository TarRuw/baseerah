# Phase 7 — Step 7.3: Performance hardening + integration tests (5 personas)

## Objective

Meet the < 2.5s analytics NFR (DESIGN §9) and prove end-to-end correctness: profile the
forecast/score/report paths, add caching or query tuning where needed so every analytics endpoint
stays under 2.5s even for the largest dataset (`client_002_tech`), and add integration tests that
exercise the full API across all 5 seeded personas (DESIGN §4, §11).

## Dependencies

- **Requires:** Steps 2.2, 3.2, 4.2, 5.2, 6.2 — the score, forecast, loan-simulate, rescue,
  challenges/rewards, and bank report/portfolio endpoints (DESIGN §6) must all be implemented and
  serving the 5 seeded personas.
- **Assumes:** all 5 mock clients are seeded (Step 1.3), the API envelope + error handling from
  Phase 0 are in place, and the test toolchain (JUnit + Spring Boot test / MockMvc or `@SpringBootTest`
  with a real/Testcontainers Postgres) is available.

## Tasks

Complete in order:

1. **Profile the hot analytics paths**
    - Measure end-to-end latency for the score (`/stress-score`), forecast (`/forecast`), loan-simulate,
      rescue, and bank report/portfolio endpoints, for each persona, with focus on `client_002_tech`
      (largest transaction volume, DESIGN §11).
    - Identify N+1 queries, full-window recomputation, and repeated per-request scans; record baseline
      numbers in the handoff note.

2. **Tune queries + add caching where needed**
    - Fix any N+1s (fetch joins / batch fetch / projections) and add DB indexes for the query patterns
      used by analytics (e.g. `transactions(account_id, booking_date)`, `stress_scores(client_id,
      as_of_date)`) via a Flyway migration `V<next>__perf_indexes.sql` if not already present.
    - Add caching for recomputable analytics where correctness allows (e.g. cache the daily stress-score
      snapshot / forecast points keyed by client + horizon), with a clear invalidation story documented.
    - Keep all business logic in services; do not push tuning into controllers. Everything stays
      computed from seeded transactions (Global Rule) — caching must not hardcode results.

3. **Assert the < 2.5s budget in an automated check**
    - Add a latency assertion (in the integration suite or a dedicated perf test) that each analytics
      endpoint responds within 2.5s for every persona, including `client_002_tech`. Use a warm run so
      the check reflects steady-state, and document the measurement method.

4. **Full-surface integration tests across all 5 personas**
    - Create `backend/src/test/java/com/baseerah/integration/PersonaJourneyIT.java` (or a small suite).
    - For each of the 5 seeded personas, exercise the full journey end-to-end through the real API:
      score, forecast, loan-simulate, rescue, challenges/rewards, and bank report/portfolio.
    - Assert persona-specific expectations from DESIGN §11: `client_001_family` → OPTIMAL score;
      `client_003_freelancer` → non-null deficit + rescue options; `client_004_student` →
      micro-saving challenges present; `client_005_vip` → bank applicant/portfolio populated;
      `client_002_tech` → WARNING-zone score and within the latency budget.
    - Assert every response uses the `{ "status":"OK", "data": ... }` envelope (Global Rule).

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] Every analytics endpoint (score, forecast, loan-simulate, rescue, bank report, portfolio)
      responds in < 2.5s for all 5 personas, including `client_002_tech`, on a warm run.
- [ ] An automated latency check asserts the 2.5s budget and passes.
- [ ] `PersonaJourneyIT` exercises score + forecast + loan + rescue + challenges + bank report across
      all 5 personas and passes green.
- [ ] Persona-specific expectations (DESIGN §11) are asserted (family OPTIMAL, freelancer deficit,
      student challenges, vip bank portfolio, tech WARNING + perf).
- [ ] Any new Flyway `V<next>__perf_indexes.sql` applies cleanly on a fresh DB; caching (if added) has
      a documented invalidation story and results remain computed from seeded data.
- [ ] All pre-existing tests still pass.
