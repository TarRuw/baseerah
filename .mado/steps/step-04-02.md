# Phase 4 — Step 4.2: Rescue endpoints

## Objective

Expose the Smart Rescue flow over REST (DESIGN §6, FR-06/07): a GET that returns the predicted
shortfall + bridge options (or a clean no-deficit state) and a POST that confirms an option and
returns the before/after score recovery, persisting a `rescue_events` row.

## Dependencies

- **Requires:** Step 4.1 — `RescueService` (`assess` / `confirm`), `RescueOption`, `RescueEvent` +
  repository, and `004-rescue-events.sql`.
- **Requires:** Step 0.4 — shared API envelope `{ "status":"OK", "data": ... }` / error
  `{ "status":"ERROR", "error":{...} }` and the global exception handler (ORCHESTRATION §Global Rules).
- **Assumes:** consumer prefix `/api/v1/...` and DTO/mapper conventions from Phase 1; controllers are
  thin and never expose entities.

## Tasks

Complete in order:

1. **Response DTOs**
    - Create response DTOs in `com.baseerah.rescue` (e.g. `RescueResponse`, `RescueOptionDto`,
      `RescueConfirmResponse`).
    - `RescueResponse`: `{ shortfall, deficitInDays, options: [ {type, label, amount, term, detail} ] }`.
      For the no-deficit case return an explicit state (e.g. `deficitInDays: null` / empty options with
      a `hasDeficit:false` flag) rather than an error.
    - `RescueConfirmResponse`: `{ scoreBefore, scoreAfter, message }`.
    - Map from `RescueAssessment` / `RescueOutcome`; entities never leave the service layer.

2. **`GET /api/v1/clients/{id}/rescue`**
    - Create `RescueController` in `com.baseerah.rescue`.
    - Call `RescueService.assess(id)`; return the enveloped `RescueResponse`.
    - Deficit persona → shortfall + `deficitInDays` + two options; healthy persona → clean no-deficit
      response. Unknown client → enveloped `ERROR` via the global handler.

3. **`POST /api/v1/clients/{id}/rescue/confirm`**
    - Request body `{ "option": "MURABAHA" | "LIQUIDATE" }`; validate the value.
    - Call `RescueService.confirm(id, option)`; return the enveloped `RescueConfirmResponse`
      (`scoreBefore`, `scoreAfter`, `message`) and rely on the service to persist the `rescue_events` row.
    - Invalid/unknown option → enveloped `ERROR`.

4. **Controller/endpoint test**
    - Add a slice or integration test (e.g. `RescueControllerTest`):
      - GET on the freelancer persona → `status:OK`, positive shortfall, two options.
      - GET on a healthy persona → `status:OK`, no-deficit state.
      - POST `{option:"LIQUIDATE"}` → `scoreAfter > scoreBefore` and a new `rescue_events` row.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] `GET /api/v1/clients/{id}/rescue` on a deficit persona returns `{status:"OK", data:{shortfall,
      deficitInDays, options:[…2…]}}` with one `MURABAHA` (amount+term) and one `LIQUIDATE` option.
- [ ] `GET /api/v1/clients/{id}/rescue` on a healthy persona returns an enveloped no-deficit state
      (not an error).
- [ ] `POST /api/v1/clients/{id}/rescue/confirm` with `{option:"MURABAHA"|"LIQUIDATE"}` returns
      `{scoreBefore, scoreAfter, message}` with `scoreAfter > scoreBefore` and persists one
      `rescue_events` row.
- [ ] Invalid client id or option returns an enveloped `ERROR` response via the global handler.
- [ ] Responses use the shared envelope; no JPA entity is serialized directly.
- [ ] Endpoint tests pass; all existing tests still pass; analytics response < 2.5 s (NFR §9).
