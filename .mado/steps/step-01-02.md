# Phase 1 — Step 1.2: JPA entities & repositories

## Objective

Map the V1 core schema to Spring Data JPA entities and repositories so the rest of the backend works with typed domain objects instead of raw SQL. Validated against the migration so schema and code cannot drift.

## Dependencies

- **Requires:** Step 1.1 — `001-core-schema.sql` must define `clients`, `accounts`, `transactions` with the columns from DESIGN.md §4.2.
- **Assumes:** Spring Data JPA is on the classpath (Step 0.2); the base package is `com.baseerah`; a test/dev profile can point at a Postgres instance with V1 applied.

## Tasks

Complete in order:

1. **Configure `ddl-auto=validate`**
    - Set `spring.jpa.hibernate.ddl-auto=validate` for the relevant profiles so Hibernate verifies entities against the Liquibase-managed schema and never mutates it (per ORCHESTRATION global rules).

2. **`Client` entity** — package `com.baseerah.client` (or `com.baseerah.domain.client`; pick one feature-package convention and keep it)
    - Maps `clients`. Fields: `UUID id`, `String externalId`, `String profileLabel`, `String persona`, `Instant createdAt`.
    - `@Id` UUID strategy consistent with the DB `gen_random_uuid()` default (let DB generate, or generate app-side — decide and document).
    - `@OneToMany` to `Account` (mappedBy `client`), lazy.

3. **`Account` entity** — same feature area, package `com.baseerah.account`
    - Maps `accounts`. Fields: `UUID id`, `Client client` (`@ManyToOne`, FK `client_id`), `String externalId`, `String bankName`, `String displayColor`, `String currency`, `BigDecimal latestBalance`, `String tokenizedAccountId`.
    - `@OneToMany` to `Transaction` (mappedBy `account`), lazy.

4. **`Transaction` entity** — package `com.baseerah.transaction`
    - Maps `transactions`. Fields: `UUID id`, `Account account` (`@ManyToOne`, FK `account_id`), `String externalId`, `Direction direction`, `BigDecimal amount`, `String currency`, `String rawDescription`, `String descriptionCleansed`, `String category`, `BigDecimal categoryConfidence`, `Instant bookingDate`, `BigDecimal closingBalance`.
    - `direction` maps to a `Direction` enum (`DEBIT`, `CREDIT`) via `@Enumerated(EnumType.STRING)`.
    - `category` maps to a `String` (enum-with-fallback per DESIGN.md §4): keep an application-level `Category` enum of the known values (`SALARY`, `GROCERIES`, `TRANSPORTATION`, …) with a resolver that falls back to a raw/`OTHER` value for unknown categories, but persist the raw text so no data is lost. Document the fallback approach in the handoff note.

5. **`Direction` enum**
    - `com.baseerah...Direction { DEBIT, CREDIT }`.

6. **Repositories** — Spring Data `JpaRepository<Entity, UUID>` per entity
    - `ClientRepository`: `Optional<Client> findByExternalId(String externalId)`, `boolean existsByExternalId(String externalId)`.
    - `AccountRepository`: `List<Account> findByClientId(UUID clientId)`, `Optional<Account> findByExternalId(String externalId)`.
    - `TransactionRepository`: finder for transactions by client and booking-date range — e.g. `findByAccount_Client_IdAndBookingDateBetween(UUID clientId, Instant from, Instant to)`; plus `findByAccount_Client_Id(UUID clientId, Pageable pageable)` to support paged history in Step 1.4.

7. **Context test**
    - Add a `@DataJpaTest` or `@SpringBootTest` that boots the context against V1 with `ddl-auto=validate` and asserts the three repositories are wired (autowire them / assert non-null, or a trivial `count()` call).

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] Application boots with `ddl-auto=validate` against the V1 schema with no Hibernate validation errors (entities match every column/type in DESIGN.md §4.2).
- [ ] `Client`, `Account`, `Transaction` entities and the `Direction` enum exist under `com.baseerah` feature sub-packages.
- [ ] Relations resolve: `Account.client` and `Transaction.account` are `@ManyToOne`; back-references are lazy.
- [ ] `category` persists the raw string while an application-level enum-with-fallback resolver handles unknown values without throwing.
- [ ] Repositories expose the finder methods listed (by `external_id`; transactions by client + date range; paged transactions by client).
- [ ] A context/JPA test passes, confirming the three repositories load and validate against the schema.

## Notes

- Do not let JPA manage schema — Liquibase owns it. `validate` only; never `update`/`create` in any committed profile.
- Exact feature-package layout (`com.baseerah.client` vs `com.baseerah.domain.client`) is an open decision; choose one and apply it consistently across all four Phase 1 steps. Document the choice in the handoff note so Step 1.3/1.4 follow it.
- Keep entities free of business logic; they never leave the service layer (mapped to DTOs in Step 1.4).
