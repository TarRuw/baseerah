# Phase 4 — Step 4.1: Deficit alert + RescueService + Liquibase rescue_events

## Objective

Implement **Smart Rescue Mode** (FR-06/07, DESIGN §5.4): reuse the Phase 3 `ForecastEngine` to
detect an upcoming cash-flow deficit, raise an alert **exactly 15 days before** the expected failure,
and prepare two Sharia-aware bridge options with a computed score recovery — the headline
`client_003_freelancer` demo.

## Dependencies

- **Requires:** Step 3.1 — `ForecastEngine.project(clientId, horizonDays)` + `HeuristicForecast` must
  exist and return `{points:[{date,balance}], deficitDate, minProjectedBalance}` (DESIGN §5.2).
- **Requires:** Step 2.1 — `StressScoreCalculator` + `stress_scores` must exist so a before/after
  score can be computed (DESIGN §5.1).
- **Assumes:** Liquibase is wired (changesets `001`–`003` applied), `clients` table seeded with the 5
  personas (Phase 1), and the `com.baseerah` package layout with Liquibase migrations under
  `backend/src/main/resources/db/changelog/changes/`.

## Tasks

Complete in order:

1. **Liquibase migration `004-rescue-events.sql`**
    - Create `backend/src/main/resources/db/changelog/changes/004-rescue-events.sql` per DESIGN §4.2.
    - Columns: `id (uuid pk)`, `client_id (uuid fk → clients.id)`, `predicted_shortfall numeric(14,2)`,
      `deficit_in_days int`, `option_chosen text null` (constrained to `MURABAHA | LIQUIDATE | null`),
      `score_before int`, `score_after int`, `created_at timestamptz not null default now()`.
    - UUID PKs, money `numeric(14,2)`, timestamps `timestamptz` (ORCHESTRATION Global Rules).
    - Index on `client_id` for lookups.

2. **`RescueEvent` entity + repository**
    - Create `RescueEvent` JPA entity in `com.baseerah.rescue` mapping the `rescue_events` table.
    - Create `RescueEventRepository extends JpaRepository<RescueEvent, UUID>` with a finder for the
      latest event per client. Entity stays in the service layer — never returned from a controller.

3. **`RescueService` — deficit detection + 15-day alert (FR-06)**
    - Create `RescueService` in `com.baseerah.rescue`.
    - Inject the `ForecastEngine` **interface** (never the concrete `HeuristicForecast`; ORCHESTRATION
      rule) and `StressScoreCalculator`.
    - Method e.g. `RescueAssessment assess(UUID clientId)`: project a horizon long enough to reach the
      deficit; if `deficitDate` is null → return a no-deficit assessment.
    - When a deficit exists, compute `deficitInDays` (today → `deficitDate`) and raise the alert flag
      when the deficit is within the **15-day** lead window (DESIGN §5.4 / §1 "12–15 days before").
    - `predictedShortfall` = magnitude of the projected shortfall at/around the deficit (e.g. the
      negative `minProjectedBalance`, as a positive SAR amount).

4. **`RescueService` — two bridge options (FR-07)**
    - Produce exactly two options (DESIGN §5.4):
      - **MURABAHA** — pre-approved Sharia-compliant micro-finance: `amount` (covers the shortfall) +
        `term` (months). Derive amount/term from the shortfall and client telemetry, not hardcoded
        prototype numbers (ORCHESTRATION "everything computed from seeded transactions").
      - **LIQUIDATE** — liquidate safe investment-fund assets to cover the shortfall: `amount` + detail.
    - Each option carries `type`, `label`, `amount`, `term` (null for liquidate), `detail`.

5. **`RescueService` — confirm + recovered score (FR-07)**
    - Method e.g. `RescueOutcome confirm(UUID clientId, RescueOption option)`:
      - `scoreBefore` = current stress score; compute `scoreAfter` as the recovered score once the
        bridge removes the deficit (liquidate recovers slightly more than murabaha, per §5.4 — the
        exact recovery curve is an engineer decision, but `scoreAfter > scoreBefore`).
      - Persist a `rescue_events` row (`option_chosen`, `score_before`, `score_after`,
        `predicted_shortfall`, `deficit_in_days`).
      - Return `{scoreBefore, scoreAfter, message}`.

6. **Unit test — freelancer persona**
    - Add a test (e.g. `RescueServiceTest`) driving `client_003_freelancer`.
    - Assert: assessment returns a positive shortfall, a 15-day-lead alert, and exactly two options;
      `confirm` yields `scoreAfter > scoreBefore` and writes one `rescue_events` row.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] `004-rescue-events.sql` applies cleanly on a fresh DB (`liquibase update`/boot) with UUID PK,
      `numeric(14,2)` money, `timestamptz`, and the `MURABAHA|LIQUIDATE|null` constraint.
- [ ] `RescueService.assess()` on a deficit persona returns a positive `predictedShortfall`, a
      `deficitInDays` value, an alert flag true within the 15-day lead window, and exactly two options
      (one `MURABAHA` with amount+term, one `LIQUIDATE`).
- [ ] `RescueService.assess()` on a healthy persona returns a no-deficit assessment (no options / alert
      false).
- [ ] `RescueService.confirm()` returns `scoreAfter > scoreBefore` and inserts exactly one
      `rescue_events` row with the chosen option and both scores.
- [ ] `RescueService` depends on the `ForecastEngine` interface, not `HeuristicForecast`.
- [ ] The freelancer unit test passes; all existing tests still pass.

## Notes

- Recovery values in DESIGN §5.4 (62→78 liquidate / 62→74 murabaha) are **prototype stand-ins**;
  compute from the client's telemetry and the removed deficit — do not hardcode them (ORCHESTRATION).
- The alert lead time anchors on "15 days before" (§5.4); §1 frames it as a 12–15 day window — treat 15
  as the trigger threshold.
- No REST controller in this step — endpoints are Step 4.2. Keep entities behind the service layer.
