# Phase 3 — Step 3.2: Forecast endpoints + wire Home chart

## Objective

Expose the forecast engine over REST (FR-04) and complete the Home/Radar 30-day forecast chart so it
renders live projected balances, the deficit marker, and trend colour driven entirely by the API,
within the < 2.5 s NFR (DESIGN §6, §7.1, §8, §9).

## Dependencies

- **Requires:** Step 3.1 — `ForecastEngine`/`HeuristicForecast` and `V3__forecasts.sql` must exist.
- **Requires:** Step 2.3 — the Flutter Home/Radar screen with the 30-day forecast **placeholder** must
  be in place.
- **Assumes:** Phase 0 API envelope (`{ "status":"OK", "data": ... }`), error handling, the Flutter
  theme/i18n/RTL file, and the dio client are available.

## Tasks

Complete in order:

1. **Forecast controller endpoint**
    - Create `backend/src/main/java/com/baseerah/forecast/ForecastController.java`.
    - `GET /api/v1/clients/{id}/forecast?horizonDays=30` (DESIGN §6). Support the 30-day home horizon
      and 3/6/12-month horizons (90/180/365 days) through the same query param.
    - Controller stays thin: resolve client, call the injected `ForecastEngine` interface (never the
      concrete `HeuristicForecast`), map the result to a DTO. Business logic stays in the service layer.

2. **Enveloped forecast DTO + mapper**
    - Create `backend/src/main/java/com/baseerah/forecast/ForecastResponse.java` (record) shaped as
      `{ points: [{date, balance}], deficitDate, minProjectedBalance, trend }`.
    - `trend` is a derived enum/string — `UP | FLAT | DOWN` — from the slope of the projected balance
      (first vs last / min crossing); used by the chart to pick line colour.
    - Return wrapped in the standard envelope; map from `ForecastResult` in the service/mapper, never
      leaking JPA entities.

3. **Wire the Flutter Home 30-day forecast chart**
    - Replace the Step 2.3 placeholder in the Flutter Home/Radar screen under `frontend/` with a live
      chart bound to `GET /api/v1/clients/{id}/forecast?horizonDays=30`, via a dio-backed provider/repo.
    - Render an SVG/line chart per DESIGN §7.1/§8: gradient fill under the line, a **dashed zero-line**,
      and a **deficit marker** at `deficitDate` when non-null (alert red `#E0574F`).
    - Line colour follows `trend`: success green `#1D9E63` (UP), warning orange `#E5A63A` (FLAT/soft),
      alert red `#E0574F` (DOWN / deficit present). Pull colours from the single theme token file (DESIGN §8).
    - Format axis/tooltip amounts with the `money()` helper (SAR / ر.س), RTL-correct for Arabic.

4. **Performance check**
    - Ensure the endpoint returns under 2.5 s for a seeded client, including the high-volume tech persona
      (`client_002_tech`) per DESIGN §9/§11.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] `GET /api/v1/clients/{id}/forecast?horizonDays=30` returns an enveloped
      `{points:[{date,balance}], deficitDate, minProjectedBalance, trend}` for a seeded client.
- [ ] 3/6/12-month horizons (90/180/365) return valid point series through the same endpoint.
- [ ] Response completes in **< 2.5 s** for a seeded client (including `client_002_tech`).
- [ ] The Home chart renders live API data with gradient fill, dashed zero-line, and a deficit marker at
      `deficitDate` for the freelancer persona; no deficit marker for the family persona.
- [ ] Chart line colour matches `trend`, colours come from the theme file, and layout is RTL-correct in Arabic.
- [ ] Controller calls the `ForecastEngine` interface (not the concrete class); existing tests still pass.
