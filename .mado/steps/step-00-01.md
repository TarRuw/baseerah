# Phase 0 — Step 0.1: Repo layout, monorepo tooling & README

## Objective

Establish the Baseerah monorepo skeleton and top-level documentation so both stacks
(Spring Boot backend, Flutter frontend) live side by side with consistent tooling, and record the
Phase 0 state-management decision. This is the foundation every later step builds on.

## Dependencies

- **Requires:** None — this is the first step of the project.
- **Assumes:** A git repository already exists at the project root; `docs/DESIGN.md`,
  `.mado/ORCHESTRATION.md`, `data-mocks/`, and `design/` are already present and must not be modified
  (see ORCHESTRATION.md Global Rules).

## Tasks

Complete in order:

1. **Create the two stack directories**
    - Create `backend/` (Spring Boot app, populated in Step 0.2).
    - Create `frontend/` (Flutter app, populated in Step 0.3).
    - Add a `.gitkeep` in each so the empty directories are committed if no other files land yet.

2. **Write the root `README.md`**
    - Project summary: one paragraph derived from DESIGN.md §1 (Baseerah — "Early Warning Financial
      Radar," proactive AI FinTech for the Saudi market, consumer app + bank portal over one backend).
    - Repository structure section: a tree listing `backend/`, `frontend/`, `docs/`, `data-mocks/`,
      `design/`, `.mado/` with a one-line purpose for each.
    - Tech stack summary: reference DESIGN.md §2 (Spring Boot 3 / Java 21 / Gradle Kotlin DSL, PostgreSQL
      16 + JPA + Liquibase, Flutter 3 / Dart 3 with Riverpod + dio). Do not duplicate the full table — link
      to `docs/DESIGN.md`.
    - Prerequisites: JDK 21, Docker + docker-compose, Flutter 3.x SDK, Dart 3.
    - How to run the backend: `docker-compose up -d` (Postgres) then `./gradlew bootRun` from `backend/`
      (forward-reference Step 0.2 — exact commands finalized there).
    - How to run the app: `flutter pub get` then `flutter run` from `frontend/` (forward-reference
      Step 0.3).
    - State-management decision note: record **Riverpod** as the chosen Flutter state solution per
      DESIGN.md §2 (Bloc was the alternative; Riverpod is locked for the whole project).

3. **Write the root `.gitignore`**
    - Java/Gradle: `.gradle/`, `build/`, `!gradle/wrapper/gradle-wrapper.jar`, `*.class`, `out/`.
    - Flutter/Dart: `.dart_tool/`, `.packages`, `build/`, `.flutter-plugins`, `.flutter-plugins-dependencies`,
      `frontend/ios/Pods/`, `frontend/**/GeneratedPluginRegistrant.*`.
    - IDE/editor: `.idea/`, `*.iml`, `.vscode/`, `*.swp`, `.DS_Store`.
    - Secrets/env: `.env`, `*.env`, `**/local.properties`.
    - Do NOT ignore `data-mocks/`, `design/`, or `docs/`.

4. **Write the root `.editorconfig`**
    - `root = true`; UTF-8; LF line endings; trailing-whitespace trim; final newline.
    - Java (`*.java`, `*.gradle.kts`): 4-space indent.
    - Dart (`*.dart`): 2-space indent.
    - Markdown/YAML/JSON: 2-space indent; do not trim trailing whitespace for `*.md`.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] `backend/` and `frontend/` directories exist at the repo root and are tracked by git.
- [ ] Root `README.md` exists and documents the project summary, repository structure, prerequisites,
      and separate run instructions for the backend and the Flutter app.
- [ ] `README.md` explicitly records Riverpod as the chosen Flutter state-management approach.
- [ ] Root `.gitignore` covers Java/Gradle, Flutter/Dart, IDE, and `.env` artifacts, and does not ignore
      `data-mocks/`, `design/`, or `docs/`.
- [ ] Root `.editorconfig` exists with the per-language indent rules described above.
- [ ] No files under `data-mocks/`, `design/`, the BRD, or `docs/DESIGN.md` were modified.
