# Phase 8 — Step 8.2: Responsive max-width / phone-frame layout for web & desktop

## Objective

Fix QA finding **UI-01**: the consumer shell has no max-width/responsive constraint, so on a wide (desktop
web) viewport the semicircular stress-score gauge scales to full content width (~1260 px tall at 2519 px),
pushing the score value, zone badge, forecast chart, stat cards, and accounts list ~2 screens below the fold.
Constrain the app to a phone-sized frame on wide screens so it is usable at any width. Reference:
[`docs/QA_E2E_FINDINGS.md`](../../docs/QA_E2E_FINDINGS.md) UI-01.

## Dependencies

- **Requires:** Phase 0.3 (Flutter skeleton: theme, routing, shell) and all consumer screens (Phases 2–6).
- **Assumes:** `frontend/lib/shell/` hosts the app shell (top toolbar + consumer bottom nav / bank sidebar);
  `frontend/lib/theme/baseerah_theme.dart` holds design tokens. This is a **mobile-first** app — the mobile
  layout is correct; the defect is only visible on wide web/desktop.

## Tasks

Complete in order:

1. **Add a responsive frame at the shell level.**
    - In the app shell, wrap the routed content in a centered container that caps width to a phone size on wide
      viewports (e.g. `maxWidth ≈ 430–480` for the consumer shell) using a `LayoutBuilder`/`ConstraintedBox`
      + `Center`. Below the breakpoint (real phones) the layout is unchanged/full-bleed.
    - Consider a subtle "device frame" background/gutter on wide screens (optional, matches DEMO.md's
      "mobile phone frame" language), but the functional requirement is the width cap.
    - The **bank shell** is a desktop layout by design (sidebar + wide content); give it its own, larger cap
      (or none) — do not force the phone width on the bank portal.

2. **Bound the stress gauge independently (defense in depth).**
    - In the Home/Radar gauge widget, cap the gauge's max diameter (e.g. it never exceeds the frame width or a
      fixed max), so even if the frame changes the gauge cannot dominate the screen. Keep the existing marker
      animation and zone colours.

3. **Verify no regression to the mobile layout.**
    - Confirm the phone-width rendering (narrow viewport) is visually identical to before — the frame is a
      no-op at/below the breakpoint.

## Acceptance Criteria

- [ ] On a ≥1280 px-wide browser, the consumer shell renders inside a centered phone-width frame; the Home
      score value and zone badge are visible **above the fold** (no need to scroll past the whole gauge).
- [ ] The stress gauge has a bounded maximum size and no longer scales with full desktop width.
- [ ] The bank portal still uses its intended wide/desktop layout (not forced to phone width).
- [ ] At phone width the consumer layout is unchanged (no visual regression); existing widget tests pass.
- [ ] The gauge marker animation and three-zone colouring still work.
