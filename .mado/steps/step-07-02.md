# Phase 7 — Step 7.2: RemoteGenAi adapter + streaming (TTFT < 1.0s)

## Objective

Provide a real GenAI provider behind the existing `GenAiClient` interface (DESIGN §5, §9): a
`RemoteGenAi` implementation that calls the provider API, is selected by `GENAI_PROVIDER=remote`,
streams so time-to-first-token is under 1.0s, and falls back cleanly to `MockGenAi` when no API key
is present — so the demo always works offline (DESIGN §9 reliability).

## Dependencies

- **Requires:** Step 3.4 — the `GenAiClient` interface, the default `MockGenAi` implementation, and the
  chat endpoint `POST /api/v1/clients/{id}/chat` (DESIGN §6, FR-03) must exist.
- **Assumes:** provider selection is already env-driven (`GENAI_PROVIDER`, default `mock`) as
  established in Phase 3; Spring config/profiles and a WebClient/HTTP client are available.

## Tasks

Complete in order:

1. **Implement `RemoteGenAi` behind the interface**
    - Create `backend/src/main/java/com/baseerah/genai/RemoteGenAi.java` implementing `GenAiClient`.
    - Call the the provider's Messages API using the model named in DESIGN (§2/§5) and the API key from an
      env var (e.g. `GENAI_API_KEY`); base URL/model/key all configurable, never hardcoded secrets.
    - Do **not** change the `GenAiClient` interface signature — controllers keep calling the interface
      (Global Rule: swappable engines stay behind their interface).
    - Map the Baseerah chat request (client context + message) to an the provider request and map the
      response back to the existing chat DTO/envelope.

2. **Provider selection + graceful fallback**
    - Wire selection so `GENAI_PROVIDER=remote` activates `RemoteGenAi`; any other value (or unset)
      keeps `MockGenAi` (DESIGN §2).
    - If `GENAI_PROVIDER=remote` but no API key is present (or the key is blank), log a clear warning
      and fall back to `MockGenAi` instead of failing startup or requests — the demo must run without
      keys (DESIGN §9 reliability).
    - Selection happens once at wiring/config time (e.g. a `@Configuration` factory / `@Conditional`),
      not inside the controller.

3. **Streaming for TTFT < 1.0s**
    - Use the provider streaming so the first token/chunk is emitted to the client as soon as it arrives
      (server-sent stream / chunked response) rather than buffering the full reply (DESIGN §9:
      time-to-first-token < 1.0s).
    - Ensure the chat endpoint can surface the stream to the Flutter client; keep the non-streaming
      contract working for callers that don't consume the stream.

4. **Tests**
    - Create `backend/src/test/java/com/baseerah/genai/GenAiProviderSelectionTest.java`.
    - Assert that with `GENAI_PROVIDER=remote` and **no** key, the wired `GenAiClient` resolves to
      `MockGenAi` and chat returns a valid reply (no exception).
    - Assert that with `GENAI_PROVIDER` unset/`mock`, `MockGenAi` is used.
    - Assert the `GenAiClient` interface is unchanged (same method signature both providers implement).
    - Stub/mock the the provider HTTP call to verify `RemoteGenAi` issues a streaming request and yields a
      first chunk without buffering the whole response (no live network call in the test).

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] With a valid key and `GENAI_PROVIDER=remote`, `POST /api/v1/clients/{id}/chat` returns a real,
      streamed reply with time-to-first-token < 1.0s.
- [ ] With `GENAI_PROVIDER=remote` but no key present, the app cleanly falls back to `MockGenAi` and
      chat still succeeds (no startup or request failure).
- [ ] With `GENAI_PROVIDER` unset or `mock`, `MockGenAi` is used.
- [ ] The `GenAiClient` interface signature is unchanged; controllers call only the interface.
- [ ] No API key or secret is hardcoded; model/base-URL/key are configurable via env.
- [ ] `GenAiProviderSelectionTest` passes and all pre-existing tests still pass.

## Notes

- DESIGN §2 decision record: GenAI is abstracted so the deterministic mock drives live demos and the
  remote adapter switches on with a key. Preserve that — the mock stays the default.
- Consult the provider's current API docs for the exact model id, streaming event
  shape, and headers before implementing; do not rely on remembered API details.
- Keep client context sent to the provider minimal and free of raw account ids (ties to Step 7.1's no-leak
  guarantee).
