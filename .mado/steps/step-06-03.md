# Phase 6 — Step 6.3: Flutter Bank shell + Applications pipeline screen

## Objective

Build the Flutter bank portal shell and the Applications pipeline screen (DESIGN §7.5, §8): a desktop
layout with a left sidebar (Applications · Portfolio · Settings) and a split-pane Applications screen
where selecting an applicant and generating a report shows a live predictive report with an
Approve/Decline decision, all wired to the Step 6.2 bank endpoints and RTL-correct for Arabic.

## Dependencies

- **Requires:** Step 6.2 — the bank endpoints must be live: `GET /api/v1/bank/applicants`,
  `POST /api/v1/bank/applicants/{id}/report`, `POST /api/v1/bank/applicants/{id}/decision` (all
  enveloped; report < 2.5 s).
- **Requires:** Step 0.3 — the Flutter skeleton with the single design-token theme file (DESIGN §8),
  i18n/RTL support (Arabic first), Riverpod state management, and the `dio` HTTP client + shared
  envelope decoding must be in place.
- **Assumes:** the shared top toolbar / Consumer↔Bank segmented control (DESIGN §7) exists; the bank
  shell lives under the Flutter `frontend/` project; design tokens are consumed from the one theme
  file, not re-declared.

## Tasks

Complete in order:

1. **Bank shell + left sidebar**
    - Build a desktop layout for the bank portal with a persistent **left sidebar** offering
      Applications · Portfolio · Settings (DESIGN §7 bank-portal shell). Selecting an item switches the
      main content pane. Use the desktop backdrop / palette from DESIGN §8 and the theme file's tokens.
    - Wire the three sidebar destinations; Applications is this step (Portfolio + Settings screens land
      in Step 6.4 — stub or route to placeholders for now).

2. **Applications split-pane layout**
    - Left pane: the **applicant list** from `GET /api/v1/bank/applicants` — each row shows avatar
      (initials), name, loan purpose, amount, and a colour-coded **risk badge** (verdict), and is
      selectable/highlightable (DESIGN §7.5).
    - Right pane: the applicant **detail**, driven by selection state.

3. **Detail states — empty → generating → report**
    - **Empty:** no applicant selected → a neutral empty/placeholder state.
    - **Generating:** on "generate report", show a live spinner using the `bsr-spin` (0.8 s) animation
      with the copy **"Analyzing 24-month telemetry…"** (DESIGN §7.5, §8) while
      `POST /api/v1/bank/applicants/{id}/report` is in flight.
    - **Report:** render the returned report — a **verdict panel**, a **stamina box** colour-coded by
      band (OPTIMAL green / WARNING orange / CRITICAL red per the DESIGN §8 palette), **3 KPI boxes**
      (forecast DTI, income stability, 12-month default probability), a **12-month cash-flow chart**
      from the report's points, and **Approve / Decline** actions.

4. **Decision flow**
    - Approve/Decline calls `POST /api/v1/bank/applicants/{id}/decision` with `approve`/`decline`,
      shows a **decision confirmation**, and reflects the persisted decision on return (e.g. the
      applicant's state/badge updates).

5. **State + API wiring (Riverpod + dio)**
    - Add providers/repository for the bank applicants, report generation, and decision calls using the
      shared `dio` client and envelope decoding. Handle loading/error states (surface API errors, not
      silent failures).

6. **RTL / i18n**
    - Ensure the shell, split-pane, badges, KPI boxes, chart, and buttons lay out correctly in RTL when
      the language is Arabic, and all labels are localised (ar|en) via `Accept-Language`.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] The bank shell renders a left sidebar (Applications · Portfolio · Settings) and selecting an item
      switches the visible page.
- [ ] The Applications screen shows a split pane; the left list is populated from
      `GET /api/v1/bank/applicants` with avatar, name, purpose, amount, and a colour-coded risk badge,
      and rows are selectable.
- [ ] Selecting an applicant shows the empty→generating→report flow: generating shows the `bsr-spin`
      loader with "Analyzing 24-month telemetry…", then a live report (verdict panel, colour-coded
      stamina box, 3 KPI boxes, 12-month cash-flow chart) from the report endpoint.
- [ ] Approve/Decline calls the decision endpoint, shows a confirmation, and the persisted decision is
      reflected in the UI.
- [ ] The whole bank shell + Applications screen is RTL-correct in Arabic and localised in ar/en.
- [ ] All UI colours/animations come from the single theme file (DESIGN §8); no hardcoded token
      duplicates.

## Notes

- Exact widget/file layout is flexible; follow whatever pattern the consumer shell established in
  earlier phases and keep both shells consuming the one theme file (Global Rules / DESIGN §8).
- The stamina box colour bands and verdict panel should map to the same zone colours used elsewhere
  (green `#1D9E63` / orange `#E5A63A` / red `#E0574F`) so the two shells stay visually consistent.
- Portfolio and Settings pages are stubbed here and completed in Step 6.4 — keep the sidebar routing
  ready for them.
- Do not compute report values client-side; render exactly what the report endpoint returns (the < 2.5 s
  work lives in the backend).
