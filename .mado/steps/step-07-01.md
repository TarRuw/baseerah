# Phase 7 — Step 7.1: SAMA tokenization + NDMO flags + risk-policy enforcement

## Objective

Honour the compliance non-functional requirements (DESIGN §9): mimic SAMA Open-Banking account
tokenization so raw account identifiers never reach the bank side, wire the NDMO data-residency and
tokenization flags from `risk_policy` into real behaviour toggles, and enforce the risk-policy
thresholds (stamina floor / auto-decline) inside `UnderwritingService`. These are *simulated*
controls per BRD out-of-scope (no live SIMAH/SAMA sync) but must be provably leak-free.

## Dependencies

- **Requires:** Step 6.2 — `UnderwritingService`, the applicant/report/decision endpoints
  (`/api/v1/bank/applicants`, `.../report`, `.../decision`), the portfolio endpoint, and the
  `risk_policy` singleton + `GET/PUT /api/v1/bank/risk-policy` must exist (DESIGN §5.5, §6, §7 screen 7).
- **Assumes:** Step 1.2 JPA entities/repositories exist; `accounts` carries `tokenized_account_id`
  and `transactions` reference accounts (DESIGN §4.2); the bank-side DTOs/mappers from Phase 6 are in
  place. The `risk_policy` row exposes `stamina_floor`, `auto_decline_threshold`, `ndmo_residency`,
  `tokenization`, `sama_last_sync`.

## Tasks

Complete in order:

1. **Guarantee account tokenization end-to-end**
    - Ensure every `accounts` row has a stable, non-reversible `tokenized_account_id` (e.g. a
      deterministic hash/opaque token derived from the account, populated at seed time). If Phase 1
      left it nullable/blank, add a Liquibase migration `007-tokenize-accounts.sql` that backfills it
      and makes it `NOT NULL`.
    - Audit every **bank-side** DTO and mapper (`com.baseerah.bank`) that references an account: they
      must carry `tokenizedAccountId` only — never the raw `accounts.external_id` / `acc-...` id or the
      internal account UUID. Consumer-side responses are unaffected.
    - Add a single mapping helper so no bank DTO is constructed with a raw account id by accident.

2. **Wire the NDMO + tokenization flags as behaviour toggles**
    - Read the `risk_policy` singleton (`tokenization`, `ndmo_residency`) in the bank service layer,
      not the controller.
    - `tokenization == true` (default): bank report/portfolio responses expose tokenized ids only.
      When `false`, behaviour is explicit and documented — but the raw account id must **still never**
      appear in a bank DTO; the toggle changes the token surface, not the leak guarantee. Document the
      exact semantics chosen in the handoff note.
    - `ndmo_residency == true`: enforce the local-data-residency simulation — reject/annotate any bank
      report path that would export data outside the residency boundary (e.g. gate an "export" flag or
      stamp responses with a residency marker). Toggling the flag via `PUT /api/v1/bank/risk-policy`
      must change the observable behaviour on the next request.

3. **Encryption-layer note for bank reporting**
    - Add a clearly-scoped note (code comment + handoff note) documenting the encryption layer that
      wraps bank predictive-report payloads (DESIGN §9: "encryption layer note on bank reporting").
      This is a *simulated* control for the MVP — document where a real at-rest/in-transit encryption
      boundary would sit around the report DTO without implementing production crypto.

4. **Enforce risk-policy thresholds in `UnderwritingService`**
    - Read `stamina_floor` and `auto_decline_threshold` from `risk_policy` inside
      `UnderwritingService` (DESIGN §5.5).
    - When a generated report's stamina score is below `stamina_floor`, or its computed default
      probability / risk metric meets/exceeds `auto_decline_threshold` (%), the verdict/decision path
      must auto-decline (or force `BAD`/`DECLINE`) rather than allow approval.
    - Thresholds come from the persisted policy, not constants; changing them via
      `PUT /api/v1/bank/risk-policy` changes verdicts on the next `report`/`decision` call.

5. **Compliance tests (leak-proof + toggle-driven)**
    - Add `backend/src/test/java/com/baseerah/bank/CompliancePolicyTest.java`.
    - Assert that serialized bank report + portfolio responses contain the `tokenized_account_id`
      value and do **not** contain any raw account id (`acc-*` external id or account UUID) — scan the
      JSON body for the raw values pulled from the DB.
    - Assert that flipping `tokenization` / `ndmo_residency` via the policy changes the observed
      response behaviour.
    - Assert that a stamina score below `stamina_floor` and a default metric at/above
      `auto_decline_threshold` each force an auto-decline verdict.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] Every bank report / portfolio response exposes only `tokenized_account_id`; no raw account
      external id (`acc-*`) or account UUID appears in any bank-side DTO.
- [ ] `CompliancePolicyTest` verifies no raw account id leaks into serialized bank DTOs (scans the
      response body against DB raw values) and passes.
- [ ] Toggling `tokenization` and `ndmo_residency` via `PUT /api/v1/bank/risk-policy` changes the
      observable behaviour on the next request.
- [ ] `UnderwritingService` reads `stamina_floor` + `auto_decline_threshold` from `risk_policy`; a
      report with stamina below the floor or default metric at/above the threshold auto-declines.
- [ ] The encryption-layer note for bank reporting is documented (comment + handoff note).
- [ ] Any new Liquibase migration applies cleanly on a fresh DB and all pre-existing tests still pass.

## Notes

- DESIGN §9 + §10: these controls are *simulated* — no live SIMAH/SAMA sync, no production crypto.
  The point is a demonstrable no-leak guarantee and policy-driven behaviour, not real key management.

  the service layer). The leak test is the backstop for that rule on the bank side.
- Document the exact semantics of `tokenization == false` in the handoff note so the Settings screen
