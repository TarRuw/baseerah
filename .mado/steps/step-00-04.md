# Phase 0 — Step 0.4: Shared API envelope, error handling & health endpoint

## Objective

Establish a consistent response envelope and centralized error handling for the whole API, plus a first
enveloped health endpoint and CORS for the Flutter dev origin — so every later endpoint returns the same
`{status,data}` / `{status,error}` shape the frontend expects (DESIGN.md §6, ORCHESTRATION Global Rules).

## Dependencies

- **Requires:** Step 0.2 — a bootable Spring Boot app (`com.baseerah.BaseerahApplication`) wired to Postgres.
- **Assumes:** Step 0.3's Flutter app runs on a known dev origin (e.g. `http://localhost` web / device),
  which CORS must permit.

## Tasks

Complete in order:

1. **Define the generic success envelope**
    - `com.baseerah.common.ApiResponse<T>` — carries `status` (default `"OK"`) and `data` (`T`).
    - Provide a static factory `ApiResponse.ok(data)`. Serializes as `{"status":"OK","data":...}`
      (mirrors the mock feed, DESIGN.md §6).

2. **Define the error shape**
    - `com.baseerah.common.ApiError` — `code` (string, e.g. `VALIDATION_ERROR`, `NOT_FOUND`, `INTERNAL_ERROR`)
      and `message` (human-readable, i18n-friendly).
    - `com.baseerah.common.ApiErrorResponse` — `status` fixed to `"ERROR"` and an `error` (`ApiError`).
      Serializes as `{"status":"ERROR","error":{"code":...,"message":...}}`.

3. **Add a global exception handler**
    - `com.baseerah.common.GlobalExceptionHandler` annotated `@RestControllerAdvice`.
    - Handle `MethodArgumentNotValidException` / `ConstraintViolationException` → HTTP 400,
      code `VALIDATION_ERROR`, message aggregating field errors.
    - Handle a domain `NotFoundException` (create `com.baseerah.common.NotFoundException`) → HTTP 404,
      code `NOT_FOUND`.
    - Handle generic `Exception` → HTTP 500, code `INTERNAL_ERROR`, generic message (do not leak stack traces).
    - Every handler returns the `ApiErrorResponse` body.

4. **Add the enveloped health endpoint**
    - `com.baseerah.health.HealthController` — `GET /api/v1/health` returning
      `ApiResponse.ok({"service":"baseerah","status":"UP"})` (or similar), HTTP 200.
    - This is the app-level health check distinct from Actuator's `/actuator/health`.

5. **Configure CORS for the Flutter dev origin**
    - `com.baseerah.common.WebCorsConfig` (a `WebMvcConfigurer` or `CorsConfigurationSource` bean).
    - Allow the Flutter dev origins (e.g. `http://localhost:*` for web/dev; keep it configurable), methods
      `GET,POST,PUT,DELETE,OPTIONS`, headers incl. `Accept-Language` and `Content-Type`.
    - Apply to `/api/**`.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] `GET /api/v1/health` returns HTTP 200 with body `{"status":"OK","data":{...}}`.
- [ ] A deliberately invalid request (validation failure) returns the error envelope
      `{"status":"ERROR","error":{"code":"VALIDATION_ERROR","message":...}}` with HTTP 400.
- [ ] A request that triggers `NotFoundException` returns the error envelope with HTTP 404 and code `NOT_FOUND`.
- [ ] An unhandled exception returns `{"status":"ERROR","error":{"code":"INTERNAL_ERROR",...}}` with HTTP 500
      and no stack trace in the body.
- [ ] A cross-origin request from the Flutter dev origin receives the expected `Access-Control-Allow-Origin`
      (and related) CORS headers, including allowance of the `Accept-Language` header.
- [ ] `./gradlew build` passes (including any tests added for the handler/health endpoint).

## Notes

- This envelope is the contract for the entire API surface in DESIGN.md §6 — all later controllers must
  return `ApiResponse<T>` and rely on `GlobalExceptionHandler` rather than building ad-hoc error bodies.
- Error `message` should be i18n-aware later (Accept-Language, DESIGN.md §6); for Phase 0 a clear English
  default is acceptable, but keep the structure ready for localization.
- Keep `/api/v1/health` (app envelope) separate from Actuator `/actuator/health` (Step 0.2 infra check);
  both should stay green.
