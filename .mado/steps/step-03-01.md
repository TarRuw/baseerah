# Phase 3 — Step 3.1: ForecastEngine interface + HeuristicForecast + Liquibase forecasts

## Objective

Deliver predictive cash-flow forecasting (FR-04) that projects a client's balance forward from
seeded telemetry and flags the first liquidity deficit, all behind a swappable `ForecastEngine`
interface so a Python Prophet/XGBoost sidecar can replace the heuristic later without touching callers.
Implements DESIGN.md §5.2.

## Dependencies

- **Requires:** Step 1.3 — the 5 mock clients must be seeded (`clients`, `accounts`, `transactions`
  populated with ~6 months of SAMA-style rows; `direction`, `amount`, `category`,
  `description_cleansed`, `booking_date`, `closing_balance` available per DESIGN §4.2).
- **Assumes:** Step 1.2 JPA entities/repositories (`Transaction`, `Account`, `Client`) and the Liquibase
  toolchain from Phase 0 are in place; latest `closing_balance` is retrievable per client/account.

## Tasks

Complete in order:

1. **Define the `ForecastEngine` interface**
    - Create `backend/src/main/java/com/baseerah/forecast/ForecastEngine.java`.
    - Single method `ForecastResult project(UUID clientId, int horizonDays)`.
    - Introduce a package-local result type `ForecastResult` (record) carrying:
      `List<ForecastPoint> points` (each `{LocalDate date, BigDecimal projectedBalance}`),
      `LocalDate deficitDate` (nullable — null when balance never crosses zero), and
      `BigDecimal minProjectedBalance`.
    - The interface must expose only domain types (records/enums), never JPA entities and never a
      reference to the concrete `HeuristicForecast` class (Global Rule: engines stay behind interfaces).

2. **Implement `HeuristicForecast` per DESIGN §5.2**
    - Create `backend/src/main/java/com/baseerah/forecast/HeuristicForecast.java`, a Spring
      `@Service`/`@Component` implementing `ForecastEngine`.
    - Load the client's trailing-window transactions via the repository layer (default window 90 days;
      make the window a private constant, not a magic literal scattered around).
    - **Recurring detection:** group transactions on `category` + `description_cleansed`; classify a
      group as recurring when it repeats on an approximately monthly cadence (tolerance in days).
      Separate recurring **inflows** (CREDIT, e.g. `SALARY`) from recurring **outflows** (DEBIT: rent,
      subscriptions, instalments). Derive each recurring event's typical amount and day-of-month.
    - **Discretionary burn:** `dailyBurn = mean of non-recurring DEBIT amounts per day` over the window.
    - **Day-by-day projection:** start from the latest `closing_balance`; for each day in
      `[today, today+horizonDays]`, apply any recurring inflow/outflow scheduled on that day, then
      subtract `dailyBurn`; record `{date, projectedBalance}`.
    - **Outputs:** the full point list, the first date `projectedBalance < 0` as `deficitDate`
      (null if none), and the running `minProjectedBalance`.
    - Everything is computed from seeded transactions — no prototype constants (Global Rule).

3. **Liquibase migration `003-forecasts.sql`**
    - Create `backend/src/main/resources/db/changelog/changes/003-forecasts.sql` matching DESIGN §4.2
      `forecasts`: `id uuid pk`, `client_id uuid fk → clients(id)`, `horizon_days int`,
      `generated_at timestamptz`, `deficit_date date null`, `min_projected_balance numeric(14,2)`,
      `points jsonb` (array of `{date,balance}`).
    - UUID PK, `numeric(14,2)` money, `timestamptz` timestamps (Global Rules). Add FK + an index on
      `client_id`.

4. **Unit tests (persona-driven)**
    - Create `backend/src/test/java/com/baseerah/forecast/HeuristicForecastTest.java`.
    - Assert the **freelancer** persona (`client_003_freelancer`) forecast returns a non-null
      `deficitDate` within the horizon (volatile, irregular income → projected deficit per DESIGN §11).
    - Assert the **family** persona (`client_001_family`) forecast returns `deficitDate == null`
      (stable salaried → no deficit).
    - Assert `points` is non-empty, ordered by date, and spans the requested horizon; assert
      `minProjectedBalance` equals the minimum over `points`.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] `ForecastEngine.project(clientId, horizonDays)` returns projected points + `deficitDate` +
      `minProjectedBalance`; the interface references no JPA entity and no concrete implementation class.
- [ ] `HeuristicForecast` detects recurring inflows/outflows by `category`+`description_cleansed` cadence,
      computes daily discretionary burn, and projects day-by-day from the latest `closing_balance`.
- [ ] The freelancer persona forecast produces a correct non-null `deficit_date` within the horizon;
      the family persona produces `deficitDate == null`.
- [ ] `003-forecasts.sql` applies cleanly on a fresh DB (UUID PK, `numeric(14,2)`, `timestamptz`, jsonb
      `points`, FK to `clients`).
- [ ] `HeuristicForecastTest` passes and all pre-existing tests still pass.
