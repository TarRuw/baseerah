# Phase 6 — Step 6.1: UnderwritingService + loan_applications + seed applicants

## Objective

Deliver the B2B credit-verification core (FR-08): a Flyway migration for the bank-side
`loan_applications` and singleton `risk_policy` tables (DESIGN §4.2), and an `UnderwritingService`
that produces a predictive report per DESIGN §5.5 (stamina score, forecast DTI, income stability,
12-month default probability, verdict), reusing the Phase 3 `ForecastEngine`. This gives the bank
endpoints (Step 6.2) and the Applications screen (Step 6.3) real reports to serve.

## Dependencies

- **Requires:** Step 3.1 — `ForecastEngine` interface + `HeuristicForecast` must exist and be
  callable behind the interface (DESIGN §5.2); the report reuses it for forecast DTI / cash-flow
  endurance rather than re-implementing forecasting.
- **Requires:** Step 1.3 — the 5 `data-mocks/` personas (incl. `client_005_vip`) must be seeded into
  Postgres with `clients`/`accounts`/`transactions` entities and repositories available, so applicants
  can be linked to real telemetry via `client_ref`.
- **Assumes:** Spring Boot + Postgres + Flyway skeleton (Phase 0) is running; base package
  `com.baseerah` with the bank feature package `com.baseerah.bank`; Flyway migration location
  `backend/src/main/resources/db/migration`; Global Rules apply (UUID PKs, `numeric(14,2)` money,
  `timestamptz`, `ddl-auto=validate`, business logic in services, entities never leave the service
  layer, engines used only behind their interface).

## Tasks

Complete in order:

1. **Flyway migration `V6__bank.sql`**
    - Create `loan_applications` per DESIGN §4.2: `id (uuid pk)`, `applicant_name (text)`,
      `initials (text)`, `purpose (text)`, `amount (numeric(14,2))`, `client_ref (uuid null,
      fk → clients)`, `stamina_score (int)`, `forecast_dti (numeric(14,2))`,
      `income_stability (numeric(14,2))`, `default_prob_12mo (numeric(14,2))`,
      `verdict (text, OK|WARN|BAD, nullable until a report is generated)`, `risk_tier (text)`,
      `decision (text null, APPROVE|DECLINE)`, plus `created_at (timestamptz default now())`.
    - Create `risk_policy` as a **singleton** per DESIGN §4.2 / §7.7: `id (uuid pk)`,
      `stamina_floor (int)`, `auto_decline_threshold (int, percent)`, `ndmo_residency (bool)`,
      `tokenization (bool)`, `sama_last_sync (timestamptz)`. Insert exactly one default row in the
      migration so the settings screen (Step 6.4) always reads a live policy; enforce singleton-ness
      (e.g. a `CHECK`/single-row guard or a fixed sentinel key documented in the handoff).
    - Place at `backend/src/main/resources/db/migration/V6__bank.sql`. Schema comes from Flyway only —
      no `ddl-auto` creation (Global Rules).

2. **JPA entities + repositories**
    - `LoanApplication` entity + `LoanApplicationRepository` and `RiskPolicy` entity +
      `RiskPolicyRepository` in `com.baseerah.bank`. Entities never leave the service layer (mapped to
      DTOs in Step 6.2). `RiskPolicyRepository` exposes a "load the single policy" finder.
    - `verdict` and `decision` map to string-valued enums with `@Enumerated(EnumType.STRING)`
      (`Verdict { OK, WARN, BAD }`, `Decision { APPROVE, DECLINE }`).

3. **`UnderwritingService` — DESIGN §5.5**
    - Implement in `com.baseerah.bank`. For a given applicant, compute a predictive report from the
      linked client's seeded telemetry (via `client_ref`):
        - **Stamina score** — long-term cash-flow endurance (0–100), derived from projected balance
          resilience; compute using the injected `ForecastEngine` over a long horizon (e.g. 12 months
          per §5.2), never a concrete implementation directly.
        - **Forecast DTI** — obligations + requested-loan servicing over forecast income, expressed as
          a percentage.
        - **Income stability** — regularity/consistency of income inflows (0–1 or %).
        - **12-month default probability** — derived from stamina + forecast DTI + income stability.
    - Assign the **verdict** per §5.5 thresholds: `OK` when `stamina >= 70 AND DTI <= 34%`;
      `BAD` when `stamina <= 48 OR DTI >= 71%`; otherwise `WARN` (mixed). Also set a `risk_tier`
      label consistent with the verdict.
    - Persist the computed fields back onto the `loan_applications` row and return a full report DTO.
    - Exact scoring/normalisation curves within the §5.5 framework are the implementing engineer's
      decision — everything is computed from seeded transactions, no prototype magic numbers copied as
      constants (Global Rules). Document chosen curves in the handoff note.

4. **Seed the applicant queue**
    - Seed a queue of loan applicants into `loan_applications` (name, initials, purpose, amount).
      At least some applicants must link to seeded clients via `client_ref` — **including
      `client_005_vip`** (the bank-portal demo persona per DESIGN §11) — while others may be
      unlinked/synthetic to fill the pipeline. Seed applicants that exercise all three verdict bands.
    - Follow the existing seeding pattern (e.g. the `MockDataSeeder` approach from Phase 1); seed
      idempotently so reboots do not duplicate rows.

5. **Unit tests for the three verdict bands**
    - Test `UnderwritingService` produces a full report (stamina, forecast DTI, income stability,
      12-month default probability, verdict, risk tier) and asserts the correct verdict for each band:
      an `OK` case (`stamina >= 70 & DTI <= 34%`), a `WARN` case (mixed), and a `BAD` case
      (`stamina <= 48 OR DTI >= 71%`).
    - Assert the service invokes `ForecastEngine` (verify reuse, e.g. via a spy/mock or asserting the
      forecast-derived fields are populated) rather than duplicating forecast logic.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] `V6__bank.sql` runs cleanly via Flyway; `loan_applications` exists with the DESIGN §4.2 columns
      and `risk_policy` exists as a singleton with exactly one default row after migration.
- [ ] `UnderwritingService` generates a full report — stamina score, forecast DTI, income stability,
      12-month default probability, verdict, and risk tier — for a linked applicant.
- [ ] The verdict follows §5.5 exactly: `OK` (stamina ≥ 70 & DTI ≤ 34%), `BAD`
      (stamina ≤ 48 OR DTI ≥ 71%), otherwise `WARN`.
- [ ] The report is computed via the injected `ForecastEngine` interface (forecast reuse), not a
      re-implementation and not a concrete class referenced directly.
- [ ] An applicant queue is seeded, linked applicants include `client_005_vip`, and the seed covers all
      three verdict bands; re-seeding does not duplicate rows.
- [ ] Unit tests pass and assert the correct verdict for each of the OK / WARN / BAD bands.
- [ ] All existing tests still pass; no `ddl-auto` schema creation is introduced.

## Notes

- DESIGN §5.5 fixes only the verdict thresholds; the stamina/DTI/stability/default-probability
  formulas are deliberately under-specified — pick deterministic curves over seeded telemetry and
  record them in the handoff so Step 6.2/6.3 and reviewers can reason about the numbers.
- Keep `UnderwritingService` deterministic over its input window so the band tests are stable; only the
  seeder and persistence touch the clock/DB.
- `risk_policy` singleton feeds Step 6.2's risk-policy endpoints and Step 6.4's Settings screen — its
  default row must be sensible (stamina floor, auto-decline threshold, NDMO + tokenization flags, an
  initial `sama_last_sync`).
- Category set is data-driven (enum-with-fallback per DESIGN §4.1) — do not hardcode a fixed category
  list when detecting income/obligations for the report.
