# Phase 8 — Step 8.1: Localize server-provided content strings (Accept-Language)

## Objective

Close the systemic i18n gap (QA finding **I18N-01**, plus **COPY-01**): the backend returns hardcoded
**English** content strings for both `Accept-Language: ar` and `en`, so English text leaks into the
Arabic-first (RTL) UI. Validation error messages already localize, but the *content* fields do not. Make
every user-facing content string the backend produces resolve to the request locale, and fix the awkward
challenge-copy grammar. Reference: [`docs/QA_E2E_FINDINGS.md`](../../docs/QA_E2E_FINDINGS.md) I18N-01 / COPY-01.

## Dependencies

- **Requires:** Phases 3–7 complete — the endpoints that emit these strings exist:
  `loan-simulate` (Phase 3.3, `LoanCalculator` verdict), `chat` (Phase 3.4, `ChatService`/`MockGenAi` reply),
  `rescue` (Phase 4.2, `RescueService` option `label`/`detail` + confirmation), `challenges` (Phase 5,
  `ChallengeService` `title`/`subtitle`), bank `report` (Phase 6.2/7.1, `UnderwritingService` verdict).
- **Assumes:** Spring's locale plumbing is available (Accept-Language → `LocaleContextHolder`). No custom
  content `MessageSource` exists yet; `ChallengeService`/`GamificationController` already touch the request
  locale but still emit English.

## Tasks

Complete in order:

1. **Establish a content MessageSource.**
    - Add `messages_en.properties` and `messages_ar.properties` under `backend/src/main/resources/` and a
      `ResourceBundleMessageSource` (`@Bean messageSource`) with UTF-8 encoding. Confirm the request locale
      resolves from `Accept-Language` (Spring Boot auto-configures `AcceptHeaderLocaleResolver`); if a fixed
      locale resolver is registered anywhere, switch it to accept-header based.
    - Add a small helper (e.g. `common/Messages.java`) that resolves a key for the current request locale via
      `LocaleContextHolder.getLocale()`, with parameter substitution.

2. **Localize the loan affordability verdict (I18N-01).**
    - In `loan/LoanCalculator.java` (and/or the loan service/mapper), replace the hardcoded English verdict
      (`"Comfortably affordable"`, and its sibling bands) with message keys
      (e.g. `loan.verdict.comfortable`, `.manageable`, `.stretch`, `.unaffordable`). Keep `verdictColor` as-is.

3. **Localize the GenAI chat reply (I18N-01).**
    - `MockGenAi` must produce an **Arabic** grounded reply when the request locale is `ar` and English when
      `en` (same numbers/grounding: score, income, outflow, surplus). Template the reply from message keys with
      injected figures rather than a single English literal. Ensure `RemoteGenAi` passes the target language
      into its prompt so a real reply also follows the locale.

4. **Localize rescue option labels/details + the confirmation line (I18N-01).**
    - In `rescue/RescueService.java`, resolve the bridge-option `label` and `detail`
      (`"Murabaha micro-finance"`, `"Liquidate fund assets"`, their descriptions) and the confirmation string
      (`"Confirmed MURABAHA bridge of SAR … recovers X → Y"`) from message keys. Keep the option `type` enum
      stable (do not translate the enum).

5. **Localize challenge title/subtitle + fix grammar (I18N-01, COPY-01).**
    - In `gamification/ChallengeService.java`, resolve `title`/`subtitle` from message keys per locale. Fix the
      category interpolation so English reads naturally ("restaurant dining", not "restaurants dining
      spending"); provide the correct Arabic equivalent.

6. **Localize the bank underwriting verdict (I18N-01).**
    - In `bank/UnderwritingService.java`, resolve the verdict headline/subtitle (`"Sustains the debt"`, etc.)
      from message keys. Keep any decision enum/threshold logic unchanged.

7. **Tests.**
    - Add/extend backend tests asserting that each endpoint returns Arabic content for `Accept-Language: ar`
      and English for `en` (assert on the resolved strings differing and matching the bundles). Cover
      loan verdict, chat reply, rescue label/detail + confirmation, challenge title/subtitle, bank verdict.

## Acceptance Criteria

- [ ] With `Accept-Language: ar`, `loan-simulate.verdict`, `chat.reply`, `rescue.options[].label/detail` +
      confirmation, `challenges[].title/subtitle`, and bank report verdict are all **Arabic**; with `en` they
      are **English** (verified by tests and by curl).
- [ ] No hardcoded English content literal remains in the six services above; strings come from
      `messages_*.properties`.
- [ ] Challenge English copy reads naturally (COPY-01 fixed) and has a correct Arabic counterpart.
- [ ] `verdictColor`/`dtiColor`, option `type` enums, and all numeric values are unchanged.
- [ ] All pre-existing backend tests still pass; new locale tests pass.

## Notes

- Highest-value step in Phase 8: one coherent change resolves ~6 visible symptoms and the most jarring
  Arabic-first defect. Frontend needs **no** change for these fields — it already renders the server strings
  raw; once the backend localizes, the UI follows.
- Global Rule: everything stays computed from seeded data — only the *wording* becomes locale-aware; figures,
  colours, and enums are untouched.
- Keep enum `type` (e.g. `MURABAHA`/`LIQUIDATE`) and decision codes in English/stable; only human-facing copy
  is localized.
