# Phase 5 — Step 5.2: Challenge/rewards endpoints

## Objective

Expose the gamification feature over REST (DESIGN §6, FR-09/FR-10): list a client's challenges with
progress, claim a completed challenge, and read the points balance + risk tier — all enveloped — so the
Flutter Goals screen (Step 5.3) can drive the live claim flow.

## Dependencies

- **Requires:** Step 5.1 — `ChallengeService`, `RewardsService`, the `challenges`/`challenge_progress`/
  `rewards_ledger` tables, and the seeded persona challenges must exist. **Step 0.4** — the shared API
  envelope (`{ "status":"OK", "data": ... }` / `{ "status":"ERROR", "error":{...} }`) and global
  exception handler must be in place.
- **Assumes:** `com.baseerah` base package with the existing controller/DTO/mapper conventions from
  Phase 1; consumer prefix `/api/v1/...`; Global Rules apply (thin controllers, entities never leave the
  service layer — map to DTOs).

## Tasks

Complete in order:

1. **Response DTOs** — package `com.baseerah.gamification` (or its `dto` sub-package, per existing
   convention)
    - `ChallengeDto`: `{ id, icon, title, subtitle, reward, pct, progressText, claimable, claimed }` where
      `reward` is the challenge's `reward_points`, `pct` is `[0,100]`, `progressText` is a human-readable
      progress string (e.g. `"12 / 20 SAR"` derived from `current_value` / `target_value`), `claimable` is
      `pct >= 100 && !claimed`, and `claimed` reflects the progress row.
    - `RewardsDto`: `{ points, riskTier }` from `RewardsService`.
    - Map entities → DTOs in the service/mapper layer; entities never reach the controller.

2. **`GET /api/v1/clients/{id}/challenges`**
    - Return the enveloped list of `ChallengeDto` for the client (from `ChallengeService`), each with its
      current progress. Unknown client id → enveloped error via the global handler (404-style).

3. **`POST /api/v1/clients/{id}/challenges/{cid}/claim`**
    - Delegate to `RewardsService.claimChallenge`. On success return the enveloped updated state:
      the new `points` balance + `riskTier` and the claimed challenge's updated `claimed` flag (e.g.
      `{ points, riskTier, challenge: ChallengeDto }`).
    - Reject a **double-claim** (already `claimed`) and an **incomplete** claim (`pct < 100`) with an
      enveloped error and an appropriate 4xx status — no points awarded, no ledger row written.

4. **`GET /api/v1/clients/{id}/rewards`**
    - Return the enveloped `RewardsDto` (`points` balance + `riskTier`) from `RewardsService`.

5. **Controller wiring**
    - Add a thin `GamificationController` (or extend the existing consumer controller pattern) that only
      delegates to the services; no business logic in the controller.

6. **Web-layer tests (`@WebMvcTest` or `@SpringBootTest`)**
    - `GET .../challenges` returns the enveloped seeded challenges with progress fields populated.
    - `POST .../claim` on a completed challenge returns the increased `points` and `claimed = true`.
    - A second `POST .../claim` on the same challenge is rejected (4xx, enveloped error) and does not
      change the balance.
    - `GET .../rewards` returns `points` + `riskTier`.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] `GET /api/v1/clients/{id}/challenges` returns an enveloped list; each item has
      `{id, icon, title, subtitle, reward, pct, progressText, claimable, claimed}` and reflects seeded
      progress.
- [ ] `POST /api/v1/clients/{id}/challenges/{cid}/claim` on a completed, unclaimed challenge returns the
      updated `points` (increased by the challenge's `reward`) and flips the challenge to `claimed`.
- [ ] Claiming the same challenge a second time is rejected with an enveloped error and a 4xx status; the
      balance is unchanged and no `rewards_ledger` row is added.
- [ ] Claiming an incomplete challenge (`pct < 100`) is rejected the same way.
- [ ] `GET /api/v1/clients/{id}/rewards` returns the enveloped `{ points, riskTier }`.
- [ ] All responses use the shared envelope; unknown ids produce enveloped errors via the global handler.
- [ ] Web-layer tests pass and all existing tests still pass.

## Notes

- Reuse the Phase 0.4 envelope and global exception handler; do not hand-roll a second error shape.
- The claim guard (completed + unclaimed) lives in `RewardsService` (Step 5.1); the controller only
  surfaces the rejection as an enveloped 4xx — keep the business rule out of the controller.
- `progressText` formatting should match the DESIGN §8 money/number helpers where a SAR amount is shown
  (e.g. `fmt(n)` grouping); keep the currency label locale-aware for the Accept-Language header.
