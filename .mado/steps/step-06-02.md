# Phase 6 — Step 6.2: Bank endpoints (applicants, report, decision, portfolio, risk-policy)

## Objective

Expose the bank portal API (FR-08) over the Step 6.1 domain: the underwriting queue, on-demand
predictive report generation, approve/decline decision, portfolio KPIs + monitoring rows, and
read/update of the singleton risk policy. All responses use the standard envelope and report
generation stays under the 2.5 s NFR (DESIGN §6, §9).

## Dependencies

- **Requires:** Step 6.1 — `loan_applications` + singleton `risk_policy` tables, `LoanApplication` /
  `RiskPolicy` entities + repositories, `UnderwritingService`, and the seeded applicant queue must
  exist.
- **Requires:** Step 0.4 — the shared response envelope (`{ "status":"OK", "data": ... }`) / error
  envelope (`{ "status":"ERROR", "error":{...} }`) and the global exception handler must be in place;
  i18n via `Accept-Language` (ar|en) is honoured.
- **Assumes:** thin controllers in `com.baseerah.bank`; business logic stays in services (Global
  Rules); entities are mapped to DTOs and never returned directly; bank prefix is `/api/v1/bank/...`.

## Tasks

Complete in order:

1. **`GET /api/v1/bank/applicants` — underwriting queue**
    - Return the applicant queue (id, applicant name, initials, purpose, amount, risk badge/verdict if
      already computed) as an enveloped list DTO. Thin controller delegating to a service that reads
      via `LoanApplicationRepository`.

2. **`POST /api/v1/bank/applicants/{id}/report` — generate predictive report**
    - Invoke `UnderwritingService` for the applicant, persist and return the full report DTO
      (stamina score, forecast DTI, income stability, 12-month default probability, verdict, risk tier,
      and the 12-month cash-flow points for the chart) enveloped.
    - Must complete in **< 2.5 s** (DESIGN §9). Return `404` (error envelope) for an unknown applicant.

3. **`POST /api/v1/bank/applicants/{id}/decision` — approve/decline**
    - Accept `{ "decision": "APPROVE" | "DECLINE" }` (case-insensitive per the `Decision` enum),
      persist it onto the `loan_applications` row, and return the updated applicant/decision DTO
      enveloped. Validate the body and reject unknown values via the error envelope.

4. **`GET /api/v1/bank/portfolio` — KPIs + monitoring rows**
    - Return a portfolio DTO with the **4 KPIs** (active facilities, average stamina, NPL rate — with
      the delta vs baseline, at-risk accounts) and a list of **monitoring rows** (borrower, facility,
      health score, trend, status ∈ Healthy|Watch|At-risk) per DESIGN §7.6. Compute from the seeded
      applicants/decisions and their stamina/verdict data. Enveloped.

5. **`GET` + `PUT /api/v1/bank/risk-policy` — read/update policy**
    - `GET` returns the singleton risk policy DTO: stamina floor, auto-decline threshold (%),
      NDMO residency flag, tokenization flag, SAMA last-sync (DESIGN §7.7).
    - `PUT` accepts the same fields, updates the single row, and returns the persisted policy so it
      **round-trips** (a subsequent `GET` reflects the written values). Enveloped; validate ranges
      (e.g. thresholds within sane bounds) via the error envelope.

6. **DTOs + mappers**
    - Add request/response DTOs (queue item, report, decision request, portfolio, risk policy) in
      `com.baseerah.bank`; map from entities in the service layer. Entities never leave services.

7. **Tests**
    - Web-layer / integration tests (e.g. `@WebMvcTest` or `@SpringBootTest`) asserting each endpoint
      returns the documented enveloped payload; that a decision persists (re-fetch reflects it); that
      `PUT` then `GET` on risk-policy round-trips; and a timing assertion (or documented measurement)
      that report generation stays under 2.5 s.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] `GET /api/v1/bank/applicants` returns the enveloped queue with name, initials, purpose, amount,
      and risk badge.
- [ ] `POST /api/v1/bank/applicants/{id}/report` returns the enveloped full report (stamina, forecast
      DTI, income stability, 12-mo default probability, verdict, risk tier, 12-mo cash-flow points) and
      completes in < 2.5 s; unknown id returns the error envelope.
- [ ] `POST /api/v1/bank/applicants/{id}/decision` with `approve`/`decline` persists the decision and
      returns the updated enveloped payload; invalid decisions return the error envelope.
- [ ] `GET /api/v1/bank/portfolio` returns the enveloped 4 KPIs (active facilities, avg stamina, NPL
      rate with baseline delta, at-risk accounts) plus monitoring rows with health/trend/status.
- [ ] `GET` and `PUT /api/v1/bank/risk-policy` read and update the singleton, and a `PUT` followed by a
      `GET` round-trips the written stamina floor, auto-decline threshold, NDMO + tokenization flags,
      and SAMA last-sync.
- [ ] Every response uses the standard envelope; controllers are thin and entities never leave the
      service layer.
- [ ] Tests pass, including the decision-persist, risk-policy round-trip, and report-under-2.5 s checks;
      all existing tests still pass.

## Notes

- Reuse the existing envelope/error-handler infrastructure from Step 0.4 — do not hand-roll a second
  envelope shape.
- NPL rate's "−% vs baseline" delta and monitoring health/trend/status are computed from the seeded
  underwriting data (verdicts/stamina/decisions), not hardcoded — document the baseline source in the
  handoff so Step 6.4's Portfolio screen matches.
- The report endpoint reuses `UnderwritingService` (which itself reuses `ForecastEngine`) — keep the
  controller thin and let the service own the < 2.5 s work.
- `Accept-Language` (ar|en) applies to any human-readable labels (verdict text, status badges) in the
  responses.
