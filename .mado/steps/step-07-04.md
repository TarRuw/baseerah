# Phase 7 — Step 7.4: Demo script, persona walkthrough & run instructions

## Objective

Produce a repeatable hackathon demo: a `docs/DEMO.md` that maps each persona to the feature it
showcases (DESIGN §11) with a scripted click-through of both shells, plus finalized root README run
instructions (docker-compose, gradlew bootRun, flutter run) and the environment variables
(`GENAI_PROVIDER`, keys) — all followable against a fresh checkout.

## Dependencies

- **Requires:** Step 7.3 — the full API surface is performance-hardened and green across all 5 personas,
  so the walkthrough describes real, working behaviour.
- **Assumes:** Steps 7.1 (compliance/settings), 7.2 (GenAI provider + env vars), the Flutter consumer +
  bank shells (all 7 screens, DESIGN §7), and docker-compose (DESIGN §2) are in place.

## Tasks

Complete in order:

1. **Write `docs/DEMO.md` — persona → feature map**
    - Create `docs/DEMO.md`. Open with the persona→feature demo map from DESIGN §11:
      - `client_001_family` → healthy **Home / Radar** gauge (OPTIMAL) + loan-simulator baseline.
      - `client_003_freelancer` → deficit forecast → **Smart Rescue** headline demo.
      - `client_004_student` → micro-saving **Goals** + rewards.
      - `client_005_vip` → **Bank portal** applicant + portfolio cross-sell.
      - `client_002_tech` → **warning-zone** score + large-volume performance check (< 2.5s).
    - For each persona: which client to select, which screen, and the exact "wow" moment to point at.

2. **Scripted click-through of both shells**
    - Consumer shell (mobile frame, bottom nav Home · Simulate · Rescue · Goals): step-by-step clicks
      showing the animated stress-score gauge, 30-day forecast + deficit warning, loan sliders +
      live verdict, Ask-Baseerah chat, Smart Rescue confirm (before→after recovery), and Goals claim.
    - Bank shell (desktop, sidebar Applications · Portfolio · Settings): generate a predictive report
      (verdict + stamina + KPIs + cashflow chart), approve/decline, review portfolio KPIs, and the
      Settings screen showing SAMA sync, NDMO residency + tokenization toggles, stamina-floor and
      auto-decline sliders (ties to Step 7.1).
    - Include the EN/ع language toggle + RTL moment and the Consumer/Bank segmented-control switch.
    - Number the steps so a presenter can follow them verbatim end-to-end.

3. **Finalize root README run instructions**
    - Update `README.md` with a clean, ordered getting-started: (a) `docker-compose up` for Postgres,
      (b) `./gradlew bootRun` for the backend, (c) `flutter run` for the app (note the two shells).
    - Document environment variables: `GENAI_PROVIDER` (`mock` default vs `remote`), the provider API
      key var from Step 7.2, and DB connection settings — including the offline/no-key path (mock).
    - State prerequisites (Java 21, Flutter 3, Docker) and confirm the seed loads the 5 mock clients on
      first boot. Cross-link `docs/DEMO.md` and `docs/DESIGN.md`.

4. **Dry-run against a fresh checkout**
    - Follow README + DEMO.md exactly on a clean clone; fix any missing step, wrong command, or env var
      so the walkthrough is genuinely followable end-to-end.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] `docs/DEMO.md` maps all 5 personas to their showcase feature per DESIGN §11.
- [ ] `docs/DEMO.md` contains a numbered click-through covering both the consumer and bank shells,
      including the language/RTL toggle and the compliance Settings screen.
- [ ] Following `docs/DEMO.md` end-to-end against a fresh checkout works (dry-run completed).
- [ ] `README.md` run instructions (docker-compose up, gradlew bootRun, flutter run) are complete and
      correct, with prerequisites listed.
- [ ] `README.md` documents `GENAI_PROVIDER`, the the provider key var, and the offline/mock fallback path.
- [ ] No implementation regressions; docs only (plus any trivial fixes surfaced by the dry-run).
