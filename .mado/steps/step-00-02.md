# Phase 0 — Step 0.2: Spring Boot + Postgres + Flyway skeleton, docker-compose

## Objective

Stand up a bootable Spring Boot 3 / Java 21 application wired to a Dockerized PostgreSQL 16 instance
with Flyway managing schema, so every later backend phase has a running, migration-driven API to build on.

## Dependencies

- **Requires:** Step 0.1 — the `backend/` directory and root tooling (`.gitignore`, `.editorconfig`) must
  exist.
- **Assumes:** JDK 21 and Docker + docker-compose are installed locally (README prerequisites).

## Tasks

Complete in order:

1. **Initialize the Gradle (Kotlin DSL) project under `backend/`**
    - Use the Gradle wrapper (commit `gradlew`, `gradlew.bat`, `gradle/wrapper/`).
    - `build.gradle.kts` + `settings.gradle.kts` (`rootProject.name = "baseerah"`).
    - Java toolchain = 21; Spring Boot 3.x plugin + Spring dependency-management plugin.
    - Group `com.baseerah`.

2. **Declare dependencies**
    - `spring-boot-starter-web`, `spring-boot-starter-data-jpa`, `spring-boot-starter-validation`,
      `spring-boot-starter-actuator`.
    - `org.flywaydb:flyway-core` (+ `flyway-database-postgresql` if required by the Flyway version).
    - `org.postgresql:postgresql` (runtime driver).
    - `spring-boot-starter-test` (test scope) for later steps.

3. **Create the application main class**
    - `backend/src/main/java/com/baseerah/BaseerahApplication.java`.
    - Standard `@SpringBootApplication` class with a `main` method calling `SpringApplication.run`.

4. **Configure `application.yml` with a `dev` profile**
    - `backend/src/main/resources/application.yml`.
    - Datasource pointing at the docker Postgres: `jdbc:postgresql://localhost:5432/baseerah`, user/password
      matching docker-compose (e.g. `baseerah` / `baseerah`), driver `org.postgresql.Driver`.
    - `spring.jpa.hibernate.ddl-auto: validate` (per ORCHESTRATION Global Rules — never `update`/`create`).
    - `spring.jpa.open-in-view: false`; show-sql off by default.
    - `spring.flyway.enabled: true`, locations `classpath:db/migration`.
    - Actuator: expose the `health` endpoint.
    - Set `spring.profiles.active: dev` as the local default.

5. **Author `docker-compose.yml`**
    - At `backend/docker-compose.yml` (or repo root — keep the README run command consistent).
    - Service `postgres` using image `postgres:16`; env `POSTGRES_DB=baseerah`, `POSTGRES_USER=baseerah`,
      `POSTGRES_PASSWORD=baseerah`; port `5432:5432`; a named volume for data persistence.

6. **Create the empty Flyway baseline directory**
    - `backend/src/main/resources/db/migration/` (add a `.gitkeep`).
    - No migrations yet — Flyway must run cleanly with an empty locations set.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] `docker-compose up -d` starts a `postgres:16` container exposing database `baseerah` on port 5432.
- [ ] `./gradlew bootRun` (from `backend/`) starts the application against the docker Postgres with no errors.
- [ ] `GET http://localhost:8080/actuator/health` returns HTTP 200 with `{"status":"UP"}`.
- [ ] Flyway initializes successfully on boot (creates `flyway_schema_history`) even with zero migrations.
- [ ] `spring.jpa.hibernate.ddl-auto` is `validate`; the app does not auto-create/alter schema.
- [ ] The Gradle wrapper is committed and `./gradlew build` succeeds.

## Notes

- Per ORCHESTRATION Global Rules: all schema changes go through Flyway; production profiles use
  `ddl-auto: validate` only. Keep `dev` on `validate` too — entities come in Phase 1 with matching
  migrations.
- Real schema (clients, accounts, transactions, etc.) arrives in Phase 1; this step intentionally ships
  zero migrations to prove the wiring boots clean.
- Keep the docker-compose credentials in sync with `application.yml`; consider reading them from `.env`
  later, but a hardcoded dev default is fine for now (matches README run instructions).
