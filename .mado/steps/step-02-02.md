# Phase 2 — Step 2.2: Stress-score endpoint

## Objective

Expose the Financial Stress Score over REST so the Flutter Home/Radar screen (Step 2.3) can render a live
gauge: `GET /api/v1/clients/{id}/stress-score` returns an enveloped DTO with the score, zone, zone colour,
the three sub-scores, and the snapshot date — responding under the 2.5 s NFR (DESIGN §9).

## Dependencies

- **Requires:** Step 2.1 — `StressScoreCalculator`, the `stress_scores` snapshot table, and the snapshot
  writer must exist and produce a score for a seeded client.
- **Requires:** Step 1.4 — the shared response envelope and global exception handler (`{status, data}` /
  `{status, error}`) and the `clients` endpoint conventions must be in place.
- **Assumes:** the 5 personas are seeded; consumer routes use the `/api/v1/...` prefix (DESIGN §6).

## Tasks

Complete in order:

1. **`StressScoreResponse` DTO**
    - Fields: `score (int)`, `zone (OPTIMAL|WARNING|CRITICAL)`, `color (hex string)`,
      `spendingVelocity`, `incomeConsistency`, `liabilityRatio`, `asOfDate`.
    - Map zone → colour hex per DESIGN §8 / §5.1: `OPTIMAL #1D9E63`, `WARNING #E5A63A`, `CRITICAL #E0574F`.
    - DTO lives in `com.baseerah.stress`; the JPA entity never leaves the service layer (Global Rules).

2. **Controller endpoint**
    - Add `GET /api/v1/clients/{id}/stress-score` in a thin controller (`com.baseerah.stress`).
    - Resolve the client, fetch the latest snapshot (or compute + snapshot on demand via Step 2.1's
      service), and return the DTO wrapped in the standard success envelope `{ "status":"OK", "data": {...} }`.
    - Keep all logic in the service; the controller only wires request → service → envelope.

3. **Unknown-client handling**
    - For an unknown `{id}`, return the standard 404 error envelope `{ "status":"ERROR", "error":{...} }`
      via the global exception handler from Step 1.4 (do not invent a bespoke error shape).

4. **Performance check**
    - Ensure the endpoint returns in under 2.5 s for a seeded client (DESIGN §9), including the
      large-volume persona `client_002_tech`. Prefer serving the persisted latest snapshot rather than
      recomputing over the full window on every request.

5. **Endpoint test**
    - Test (e.g. `@SpringBootTest` / `MockMvc` or `WebTestClient`) that a seeded client returns the
      enveloped payload with the correct zone and matching colour hex, and that an unknown id returns the
      404 error envelope. Optionally assert the response time budget.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] `GET /api/v1/clients/{id}/stress-score` for a seeded client returns `200` with envelope
      `{ "status":"OK", "data": { score, zone, color, spendingVelocity, incomeConsistency, liabilityRatio, asOfDate } }`.
- [ ] `zone` and `color` agree with DESIGN §8 (`OPTIMAL #1D9E63`, `WARNING #E5A63A`, `CRITICAL #E0574F`).
- [ ] The response completes in under 2.5 s for a seeded client (verified including `client_002_tech`).
- [ ] An unknown `{id}` returns the standard 404 `{ "status":"ERROR", "error":{...} }` envelope.
- [ ] The JPA entity does not appear in the controller/DTO layer.
- [ ] All existing tests still pass.

## Notes

- Colour belongs to the API response (per the endpoint spec), but the same hex values must match the
  Flutter theme tokens in Step 2.3 (DESIGN §8) — keep them identical so the gauge zone and the served
  colour never disagree.
- Serving the persisted snapshot keeps the endpoint comfortably inside the 2.5 s budget; only recompute
  when no snapshot exists for the client.
