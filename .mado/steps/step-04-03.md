# Phase 4 — Step 4.3: Flutter Rescue screen

## Objective

Build the consumer **Rescue** screen (DESIGN §7.3, §8) with its two states — *open* (shortfall banner +
selectable bridge cards + confirm) and *complete* (success + before→after score recovery + run-again) —
wired to the Step 4.2 rescue endpoints, RTL-correct for Arabic.

## Dependencies

- **Requires:** Step 4.2 — `GET /api/v1/clients/{id}/rescue` and
  `POST /api/v1/clients/{id}/rescue/confirm` returning the enveloped shortfall/options and
  before/after recovery.
- **Requires:** Step 0.3 — Flutter skeleton: single theme/token file (DESIGN §8), i18n/RTL,
  Riverpod state management, `dio` HTTP client, and the consumer bottom-nav shell
  (Home · Simulate · **Rescue** · Goals, DESIGN §7).
- **Assumes:** the shared envelope is already unwrapped by the dio layer and a client id is available
  from app state.

## Tasks

Complete in order:

1. **Rescue data layer (models + repository + provider)**
    - Add Dart models mirroring the API: `RescueState` (`shortfall`, `deficitInDays`, `hasDeficit`,
      `options:[RescueOption{type,label,amount,term,detail}]`) and `RescueOutcome`
      (`scoreBefore`, `scoreAfter`, `message`).
    - Add a repository calling the two endpoints via `dio`, and a Riverpod provider/notifier holding
      the screen state (loading / open / complete / no-deficit / error).

2. **Open state — shortfall banner + bridge cards + confirm (DESIGN §7.3)**
    - Dark-red shortfall banner showing **−SAR amount** and **"in N days"** (use the `money()`/`fmt()`
      helpers and `red-warning` gradient `135deg #c0433c→#8f2f2a`, DESIGN §8); optional `bsr-pulse`.
    - Two selectable bridge cards from `options`: **Murabaha** (green icon `#1D9E63`) and **Liquidate**
      (gold icon `#C4A24C`), each showing label/amount/term/detail; selecting one highlights it.
    - **Confirm** button disabled until an option is selected; on tap calls `rescue/confirm` with the
      selected `type`.

3. **Complete state — recovery panel + run-again (DESIGN §7.3)**
    - On confirm success show a success check, a **before→after** score recovery panel
      (`scoreBefore` → `scoreAfter`, e.g. gauge/number transition), and the `message`.
    - A **"run again"** action resets the notifier back to the open (re-fetched) state.

4. **No-deficit + loading/error handling**
    - When the API reports no deficit, show a calm no-deficit/healthy state (not an error banner).
    - Handle loading (spinner) and error (retry) paths.

5. **Theme, RTL & wiring**
    - Use only tokens from the shared theme file (DESIGN §8) — no ad-hoc colors.
    - Verify RTL layout and Arabic numerals/labels render correctly (`ar`), and mount the screen in the
      consumer bottom-nav Rescue tab.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] On a deficit persona the screen shows a live dark-red banner with the real shortfall amount and
      "in N days", plus two bridge cards (Murabaha green / Liquidate gold) from the API.
- [ ] The confirm button is disabled until a bridge card is selected and enabled after selection.
- [ ] Confirming calls `POST …/rescue/confirm`, transitions to the complete state, and shows the
      `scoreBefore → scoreAfter` recovery panel with the returned message.
- [ ] "Run again" resets the screen back to the open state.
- [ ] A no-deficit persona shows a healthy/no-deficit state, not an error.
- [ ] Layout is RTL-correct in Arabic and uses only shared theme tokens (DESIGN §8); the screen is
      reachable from the consumer Rescue nav tab.
