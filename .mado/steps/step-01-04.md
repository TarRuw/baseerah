# Phase 1 — Step 1.4: Core DTOs + client/account/transaction endpoints

## Objective

Expose the seeded data over REST as DTOs (entities never leave the service layer) through the shared API envelope, delivering the first three read endpoints from DESIGN.md §6 so the Flutter shells can list personas, accounts, and transaction history.

## Dependencies

- **Requires:** Step 1.3 — a seeded database (5 clients with accounts and transactions).
- **Requires:** Step 0.4 — the shared `ApiResponse` success/error envelope (`{ "status":"OK", "data": ... }` / `{ "status":"ERROR", "error":{...} }`) and the global exception handler that maps a not-found to a 404 error envelope.
- **Assumes:** the feature-package layout and `Direction`/`Category` handling chosen in Steps 1.2–1.3.

## Tasks

Complete in order:

1. **DTOs** — immutable records under each feature package's `dto`
    - `ClientDto`: `id`, `externalId`, `profileLabel`, `persona` (and `createdAt` if useful). Never expose entity internals.
    - `AccountDto`: `id`, `bankName`, `displayColor`, `currency`, `latestBalance`, `tokenizedAccountId`. **Do not expose the raw account `external_id`** to callers (SAMA tokenization, DESIGN.md §9) — expose `tokenizedAccountId` only.
    - `TransactionDto`: `id`, `direction`, `amount`, `currency`, `rawDescription`, `descriptionCleansed`, `category`, `categoryConfidence`, `bookingDate`, `closingBalance`.

2. **Mappers**
    - Plain mapper classes/methods (`ClientMapper`, `AccountMapper`, `TransactionMapper`) that convert entity → DTO inside the service layer. No entity crosses the controller boundary.

3. **Services**
    - `ClientService`: `list()` → all clients as `ClientDto`; a lookup that resolves a client by id (UUID) or throws a not-found exception when absent (feeds the 404 envelope).
    - `AccountService`: `accountsForClient(clientId)` → `List<AccountDto>` (validates the client exists first).
    - `TransactionService`: `transactionsForClient(clientId, pageable/limit)` → paginated `TransactionDto`, ordered by `bookingDate` descending (validates the client exists first).

4. **Controllers** — thin, under `com.baseerah...` web/controller packages
    - `GET /api/v1/clients` → enveloped `List<ClientDto>`.
    - `GET /api/v1/clients/{id}/accounts` → enveloped `List<AccountDto>`.
    - `GET /api/v1/clients/{id}/transactions` → enveloped paginated `List<TransactionDto>`; support pagination/limit via query params (`?page=&size=` or `?limit=`), with a sane default page size and a max cap.
    - All responses wrapped in the Step 0.4 `ApiResponse` envelope. Controllers contain no business logic.

5. **Not-found handling**
    - Resolving `{id}` (accept the client UUID; optionally also the `external_id`) that does not exist must produce the error envelope with HTTP 404 via the global handler — controllers do not build error bodies by hand.

6. **Web-layer tests**
    - `@WebMvcTest`/`@SpringBootTest` + MockMvc covering: each endpoint returns the enveloped DTO shape for a seeded client; unknown id returns `status:ERROR` + 404; the transactions endpoint honours the limit/page param.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] `GET /api/v1/clients` returns the 5 seeded clients as `ClientDto` inside the `{ "status":"OK", "data":[...] }` envelope.
- [ ] `GET /api/v1/clients/{id}/accounts` returns that client's accounts as `AccountDto`; the raw account `external_id` is not present in the response (only `tokenizedAccountId`).
- [ ] `GET /api/v1/clients/{id}/transactions` returns `TransactionDto`s in the envelope, ordered newest-first, and respects the pagination/limit param.
- [ ] An unknown `{id}` on any of the three endpoints returns HTTP 404 with the `{ "status":"ERROR", "error":{...} }` envelope from the Step 0.4 handler.
- [ ] No JPA entity is serialized to the client — only DTOs (verified by mapper usage / response shape).
- [ ] Web-layer tests pass for the happy path, the 404 path, and pagination.

## Notes

- Reuse the Step 0.4 `ApiResponse` envelope and global exception handler — do not invent a second envelope or hand-roll error bodies in controllers (ORCHESTRATION rule: controllers are thin, business logic in services).
- The stress-score, forecast, loan, rescue, challenge, and bank endpoints from DESIGN.md §6 are later phases — only the three read endpoints in this step's title are in scope.
- Consider whether `{id}` should accept the human-readable `external_id` in addition to the UUID (convenient for demos/Flutter). If supported, keep it explicit and consistent across all three endpoints, and document it in the handoff.
- Keep money as `BigDecimal`/`numeric(14,2)` end-to-end; formatting to `SAR`/`ر.س` strings is the Flutter layer's job (DESIGN.md §8), not the DTO's.
