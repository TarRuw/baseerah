# Phase 1 — Step 1.1: Liquibase core schema (clients, accounts, transactions)

## Objective

Create the core relational schema for the platform — the `clients`, `accounts`, and `transactions` tables — as the first Liquibase migration, so seeded telemetry has a durable, correctly-typed home. This is the foundation every later phase reads from.

## Dependencies

- **Requires:** Step 0.2 — the Spring Boot + PostgreSQL + Liquibase skeleton must exist and boot, with Liquibase configured to scan `backend/src/main/resources/db/changelog/changes` and a working Postgres connection (docker-compose).
- **Assumes:** DESIGN.md §4.2 is the authoritative column list for these three tables; global rules in `.mado/ORCHESTRATION.md` apply (UUID PKs, `numeric(14,2)` money, `timestamptz` timestamps).

## Tasks

Complete in order:

1. **Create the first Liquibase changeset file**
    - Create `backend/src/main/resources/db/changelog/changes/001-core-schema.sql` as a **Liquibase
      formatted-SQL** file: first line `--liquibase formatted sql`, then `--changeset baseerah:001-core-schema`,
      followed by the SQL (per the ORCHESTRATION changelog convention). `includeAll` in the master changelog
      picks it up automatically — no master edit needed.
    - Pure PostgreSQL 16 SQL, no entity/JPA references (that is Step 1.2).
    - Enable UUID generation via `gen_random_uuid()` (built into PG 13+, `pgcrypto`); do not rely on `uuid-ossp`.

2. **`clients` table** — per DESIGN.md §4.2
    - Columns: `id uuid primary key default gen_random_uuid()`, `external_id text not null unique` (e.g. `client_001_family`), `profile_label text`, `persona text`, `created_at timestamptz not null default now()`.

3. **`accounts` table** — per DESIGN.md §4.2
    - Columns: `id uuid primary key default gen_random_uuid()`, `client_id uuid not null references clients(id)`, `external_id text`, `bank_name text`, `display_color text`, `currency text`, `latest_balance numeric(14,2)`, `tokenized_account_id text`.
    - Foreign key `client_id → clients(id)`.

4. **`transactions` table** — per DESIGN.md §4.2
    - Columns: `id uuid primary key default gen_random_uuid()`, `account_id uuid not null references accounts(id)`, `external_id text`, `direction text not null` (values `DEBIT|CREDIT`), `amount numeric(14,2) not null`, `currency text`, `raw_description text`, `description_cleansed text`, `category text`, `category_confidence numeric`, `booking_date timestamptz not null`, `closing_balance numeric(14,2)`.
    - Foreign key `account_id → accounts(id)`.

5. **Indexes**
    - `transactions (account_id, booking_date)` — history queries and forecasting scans.
    - `accounts (client_id)` — linked-accounts lookups.
    - A unique constraint/index on `transactions (external_id)` is optional; if added, keep it nullable-safe. Note the decision in the handoff.

6. **Verify migration applies**
    - Boot the app (or run the Liquibase task) against a fresh Postgres and confirm V1 applies and is recorded in `databasechangelog`.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] `001-core-schema.sql` applies cleanly against a fresh database with no errors.
- [ ] Tables `clients`, `accounts`, `transactions` exist with exactly the columns and types from DESIGN.md §4.2 (UUID PKs, `numeric(14,2)` money, `timestamptz` timestamps).
- [ ] FK `accounts.client_id → clients.id` and FK `transactions.account_id → accounts.id` exist and are enforced.
- [ ] Indexes exist on `transactions (account_id, booking_date)` and `accounts (client_id)`.
- [ ] `clients.external_id` has a unique constraint.
- [ ] Re-running Liquibase (second boot) is a no-op — V1 is not re-applied and no error is raised.

## Notes

- Only the three core tables are introduced here. The remaining tables in DESIGN.md §4.2 (`stress_scores`, `forecasts`, `loan_applications`, `challenges`, `challenge_progress`, `rewards_ledger`, `risk_policy`, `rescue_events`) are added by later phases in their own changesets — do not create them now.
- Migrations are immutable once applied. Never edit `001-core-schema.sql` after it has run in any shared environment; correct it with a new changeset.
- `category` is stored as `text` (enum-with-fallback per DESIGN.md §4 — the category set is data-driven, not a fixed DB enum). The JPA-level enum handling is Step 1.2's concern.
