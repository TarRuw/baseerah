# Phase 3 — Step 3.3: LoanCalculator + loan-simulate endpoint

## Objective

Deliver interactive loan affordability (FR-05): a `LoanCalculator` that mirrors the DESIGN §5.3 formulas
exactly and derives income and essentials from the client's own telemetry (not prototype constants),
exposed via `POST /api/v1/clients/{id}/loan-simulate` for the Simulate screen sliders.

## Dependencies

- **Requires:** Step 2.1 — the StressScoreCalculator / score baseline must exist (loan simulation
  reports a **projected score** derived from the client's current score).
- **Requires:** Step 1.4 — client/account/transaction data access must be in place to derive income and
  essentials from seeded transactions.
- **Assumes:** Phase 0 API envelope and error handling are available.

## Tasks

Complete in order:

1. **`LoanCalculator` service (DESIGN §5.3, exact formulas)**
    - Create `backend/src/main/java/com/baseerah/loan/LoanCalculator.java` as a Spring `@Service`.
    - Implement precisely:
      - `r = rate/100/12`
      - `installment = P * r / (1 - (1+r)^-n)`, with the **`r == 0` edge case → `P / n`**
      - `surplus = income - essentials`
      - `DTI = (essentials + installment) / income`
      - **verdict:** `installment <= 0.50*surplus` → `"Comfortably affordable"` (green);
        `installment <= 0.85*surplus` → `"Strains liquidity"` (orange); else → `"Not affordable"` (red)
      - **score impact:** `strain = installment/surplus`;
        `projectedScore = currentScore - max(0, (strain - 0.35)*90)`, **clamped to [9, 84]**
    - Colours per DESIGN §8: green `#1D9E63`, orange `#E5A63A`, red `#E0574F`.

2. **Derive income & essentials from telemetry (not constants)**
    - Compute `income` = mean recurring inflow (SALARY/CREDIT) from the client's transactions.
    - Compute `essentials` = sum of recurring debits (rent, instalments, subscriptions) — reuse the
      recurring-detection logic from `HeuristicForecast` (Step 3.1) where practical rather than
      duplicating it.
    - The prototype family constants (essentials 14,200 / income 18,500 / surplus 4,300) are **demo
      stand-ins only** and must NOT be hardcoded (Global Rule; DESIGN §5.3).

3. **`POST /api/v1/clients/{id}/loan-simulate` endpoint**
    - Create `backend/src/main/java/com/baseerah/loan/LoanController.java` and request/response DTOs.
    - Request body: `{ principal, rate, term }`.
    - Enveloped response: `{ installment, total, dti, dtiColor, verdict, verdictColor, projectedScore }`
      where `total = installment * term`, and `dtiColor`/`verdictColor` are the DESIGN §8 hex codes.
    - Thin controller → `LoanCalculator`; validate inputs (positive principal/term, non-negative rate).

4. **Unit tests (including `r = 0`)**
    - Create `backend/src/test/java/com/baseerah/loan/LoanCalculatorTest.java`.
    - Assert `installment`/`DTI`/`verdict` for a known input set against hand-computed values.
    - Assert the **`r = 0` edge case** returns `installment == P/n` (no divide-by-zero / NaN).
    - Assert `projectedScore` clamps within `[9, 84]` at both extremes.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] `LoanCalculator` reproduces DESIGN §5.3 exactly: installment, surplus, DTI, verdict thresholds
      (0.50 / 0.85 of surplus), and score-impact clamp `[9, 84]`.
- [ ] `income` and `essentials` are derived from the client's seeded transactions — no prototype constants.
- [ ] `POST /api/v1/clients/{id}/loan-simulate` with `{principal,rate,term}` returns enveloped
      `{installment,total,dti,dtiColor,verdict,verdictColor,projectedScore}` with correct values for a
      known input.
- [ ] The `r = 0` case returns `installment == principal/term` and is covered by a passing test.
- [ ] `LoanCalculatorTest` passes and all pre-existing tests still pass.

  table is a separate concern (Phase 6).
