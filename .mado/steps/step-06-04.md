# Phase 6 — Step 6.4: Flutter Bank Portfolio + Settings screens

## Objective

Complete the bank portal (DESIGN §7.6, §7.7): a Portfolio screen with 4 KPI cards and a monitoring
table, and a Settings screen exposing SAMA sync status, NDMO residency + tokenization toggles, and
stamina-floor / auto-decline-threshold sliders that load from and persist to the singleton risk policy.
Both wired to the Step 6.2 endpoints and RTL-correct.

## Dependencies

- **Requires:** Step 6.2 — `GET /api/v1/bank/portfolio` and `GET` + `PUT /api/v1/bank/risk-policy` must
  be live and enveloped.
- **Requires:** Step 0.3 — the Flutter skeleton with the single design-token theme file (DESIGN §8),
  i18n/RTL (Arabic first), Riverpod, and `dio` + envelope decoding.
- **Assumes:** the bank shell + left sidebar from Step 6.3 exists and routes to Portfolio and Settings
  destinations (stubbed there, filled in here); design tokens consumed from the one theme file.

## Tasks

Complete in order:

1. **Portfolio screen — 4 KPI cards (DESIGN §7.6)**
    - From `GET /api/v1/bank/portfolio`, render 4 KPI cards: **active facilities**, **average stamina**,
      **NPL rate** (showing the −% delta vs baseline), and **at-risk accounts**. Use the theme card /
      palette tokens from DESIGN §8.

2. **Portfolio screen — monitoring table**
    - Render the monitoring rows as a table: **borrower**, **facility**, **health score** with a
      **trend arrow** (up/down/flat), and a **status badge** (Healthy / Watch / At-risk) colour-coded
      per the DESIGN §8 palette (green/orange/red).

3. **Settings screen — status + toggles (DESIGN §7.7)**
    - From `GET /api/v1/bank/risk-policy`, show **SAMA sync status** (last-sync timestamp), an **NDMO
      residency toggle**, and a **tokenization toggle**, each reflecting the loaded policy values.

4. **Settings screen — sliders**
    - A **stamina-floor slider** and an **auto-decline-threshold slider (%)**, initialised from the
      loaded policy.

5. **Persist to risk-policy**
    - Changing any toggle or slider persists the full policy via `PUT /api/v1/bank/risk-policy` and the
      UI reflects the round-tripped result (re-loads / confirms the saved values). Handle loading/error
      states (surface API errors; no silent failures).

6. **State + API wiring (Riverpod + dio)**
    - Add providers/repository methods for portfolio and risk-policy read/update using the shared `dio`
      client + envelope decoding (extend the Step 6.3 bank repository/providers).

7. **RTL / i18n**
    - Portfolio cards + table and the Settings toggles/sliders lay out correctly in RTL for Arabic;
      trend arrows mirror correctly; all labels localised (ar|en).

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] The Portfolio screen shows 4 live KPI cards (active facilities, avg stamina, NPL rate with the −%
      vs-baseline delta, at-risk accounts) from `GET /api/v1/bank/portfolio`.
- [ ] The monitoring table lists borrower, facility, health score with a trend arrow, and a colour-coded
      status badge (Healthy / Watch / At-risk).
- [ ] The Settings screen loads SAMA sync status, NDMO residency toggle, tokenization toggle, stamina-
      floor slider, and auto-decline-threshold slider from `GET /api/v1/bank/risk-policy`.
- [ ] Changing a toggle or slider `PUT`s to `/api/v1/bank/risk-policy` and the persisted values
      round-trip (a reload reflects them).
- [ ] Both screens are RTL-correct in Arabic and localised in ar/en, with trend arrows mirrored
      correctly.
- [ ] All colours/tokens come from the single theme file (DESIGN §8); no hardcoded duplicates.

## Notes

- Status badge and trend colours reuse the same zone palette as the rest of the app (green `#1D9E63`,
  orange `#E5A63A`, red `#E0574F`) so both shells stay consistent (DESIGN §8).
- The NDMO residency and tokenization toggles are the UI for the simulated SAMA/NDMO compliance flags
  (DESIGN §9) — they persist to `risk_policy`; no live SAMA/SIMAH sync (out of scope, DESIGN §10).
- Slider ranges should match the backend's validated bounds from Step 6.2 (stamina floor 0–100,
  auto-decline threshold as a percentage) — keep client validation consistent with the server.
- Extend the Step 6.3 bank repository/providers rather than creating a parallel HTTP layer.
