# Phase 2 — Step 2.1: StressScoreCalculator + Flyway V2 stress_scores + snapshots

## Objective

Compute the 0–100 Financial Stress Score from seeded telemetry (DESIGN §5.1) and persist a daily
snapshot of the score plus its three sub-scores, so the score endpoint (Step 2.2) and Home gauge
(Step 2.3) have a real value to serve.

## Dependencies

- **Requires:** Step 1.3 — JPA entities and repositories for `clients`, `accounts`, `transactions`
  must exist and the 5 `data-mocks/` personas must be seeded into Postgres.
- **Assumes:** Spring Boot + Postgres + Flyway skeleton from Phase 0 is running; the `com.baseerah`
  base package and a Flyway migration location (`backend/src/main/resources/db/migration`) are in place;
  Global Rules apply (UUID PKs, `numeric(14,2)` money, `timestamptz`, business logic in services).

## Tasks

Complete in order:

1. **Flyway migration `V2__stress_scores.sql`**
    - Create the `stress_scores` table per DESIGN §4.2: `id (uuid pk)`, `client_id (uuid fk → clients)`,
      `as_of_date (date)`, `score (int, 0–100)`, `zone (text, CRITICAL|WARNING|OPTIMAL)`,
      `spending_velocity (numeric(14,2))`, `income_consistency (numeric(14,2))`,
      `liability_ratio (numeric(14,2))`, `created_at (timestamptz default now())`.
    - Add a unique constraint on `(client_id, as_of_date)` so a persona has one snapshot per day
      (the snapshot writer upserts on this key).
    - Place at `backend/src/main/resources/db/migration/V2__stress_scores.sql`. Do not use `ddl-auto`;
      schema comes from Flyway only (per Global Rules).

2. **`StressScore` JPA entity + repository**
    - Map the `stress_scores` table to an entity in package `com.baseerah.stress`. Entity must not leave
      the service layer (map to DTO in Step 2.2).
    - Add a Spring Data repository with a finder for the latest snapshot by client
      (e.g. `findFirstByClientIdOrderByAsOfDateDesc`) and one by `(clientId, asOfDate)` for upsert.

3. **`StressScoreCalculator` service — DESIGN §5.1**
    - Implement in `com.baseerah.stress`. Compute the three normalised sub-scores over a trailing window
      (default 90 days) from the client's seeded transactions:
        - **Spending velocity** — burn rate `debitSum / creditSum`; lower is better.
        - **Income consistency** — regularity of CREDIT/SALARY inflows (coefficient of variation of
          monthly income + cadence); steadier is better.
        - **Liability structure** — recurring-obligation share of income plus buffer (closing balance vs
          monthly expense); lower obligation + higher buffer is better.
    - Combine per §5.1: `score = round(100 * (w1*velocityScore + w2*consistencyScore + w3*liabilityScore))`
      with default weights `w1=0.4, w2=0.3, w3=0.3`; higher score = healthier.
    - Assign the zone from §5.1 thresholds: `>=70 OPTIMAL`, `40–69 WARNING`, `<40 CRITICAL`.
    - Expose the three sub-scores on the result object alongside the final score and zone.
    - The exact normalisation curves and any weight tuning are the implementing engineer's design
      decision within the §5.1 framework — document the chosen curves and weights in the handoff note.

4. **Snapshot writer**
    - A service method that computes today's score for a client and upserts one row into `stress_scores`
      for `as_of_date = today` (using the `(client_id, as_of_date)` unique key), storing the score, zone,
      and all three sub-scores.

5. **Unit tests over ≥2 personas**
    - Test `StressScoreCalculator` against at least `client_001_family` and `client_003_freelancer`.
    - Assert each score is in `[0,100]` with the correct zone, and assert the sensible relative ordering:
      the stable salaried family scores healthier (higher) than the irregular-income freelancer.
    - Add a test that the snapshot writer persists exactly one row per client/day and updates (not
      duplicates) on a second run for the same day.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] `V2__stress_scores.sql` runs cleanly via Flyway; the `stress_scores` table exists with the §4.2
      columns and a unique constraint on `(client_id, as_of_date)`.
- [ ] `StressScoreCalculator` returns an integer score in `[0,100]` and the correct zone
      (OPTIMAL/WARNING/CRITICAL) per the §5.1 thresholds.
- [ ] The three sub-scores (spending velocity, income consistency, liability structure) are exposed on
      the result object.
- [ ] Running the snapshot writer persists exactly one `stress_scores` row for the client for the current
      date; a second run for the same day updates rather than duplicates it.
- [ ] Unit tests pass and assert `client_001_family` scores higher than `client_003_freelancer`.
- [ ] All existing tests still pass; no `ddl-auto` schema creation is introduced.

## Notes

- Normalisation is deliberately under-specified in DESIGN §5.1: pick monotonic curves that map each raw
  ratio into `[0,1]` (higher = healthier), and record the exact curves + final weights in the handoff so
  Step 2.2 and reviewers can reason about the numbers. No prototype magic numbers copied as constants —
  everything is computed from seeded transactions (Global Rules).
- Category set is data-driven (enum-with-fallback per DESIGN §4.1) — do not hardcode a fixed category
  list when detecting recurring obligations or income.
- Keep the calculator pure/deterministic over its input window so the unit tests are stable; the snapshot
  writer is the only piece that touches the clock and the database.
