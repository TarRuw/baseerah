# Phase 1 — Step 1.3: MockDataSeeder — load 5 clients on boot

## Objective

Ingest the five `data-mocks/*.json` SAMA Open-Banking payloads into Postgres on application startup, idempotently, so every subsequent feature has realistic seeded telemetry to work from (per DESIGN.md §3, §4).

## Dependencies

- **Requires:** Step 1.2 — `Client`, `Account`, `Transaction` entities and their repositories, validated against the V1 schema.
- **Assumes:** Jackson is on the classpath (Spring Boot default); the five files exist under `data-mocks/` and follow the shape in DESIGN.md §4.1; the persona table is DESIGN.md §4 (file → external client_id → persona).

## Tasks

Complete in order:

1. **Configurable data-mocks path**
    - Add a config property (e.g. `baseerah.seed.data-mocks-path`, default `data-mocks`) resolved relative to the working directory, with the ability to override via env/property.
    - Add a toggle (e.g. `baseerah.seed.enabled`, default `true`) so tests can disable seeding.

2. **Jackson binding records for the input payload**
    - Model the DESIGN.md §4.1 shape as immutable DTO records (input-only, not the JPA entities): envelope `{status, timestamp, data}`; `data` `{type, client_profile, transactions[]}`; transaction `{transaction_id, account_id, credit_debit_indicator, amount{currency,amount}, transaction_information, booking_date_time, insights{description_cleansed, category, category_confidence}, balance{type, amount{currency,amount}}}`.
    - Ignore unknown properties defensively.

3. **Seeder component**
    - A `@Component` implementing `ApplicationRunner` (or `@EventListener(ApplicationReadyEvent.class)`), placed under `com.baseerah...seed`.
    - On startup, enumerate `*.json` files in the configured path. Derive each client's `external_id` from the `account_id` / filename convention (mock `account_id` is `acc-<client_external_id>`, e.g. `acc-client_001_family` → `client_001_family`); the filename also encodes it (`client_001_family_6_months_data.json`).

4. **Idempotency guard**
    - For each file, resolve the client `external_id` and skip the entire file if `clientRepository.existsByExternalId(...)` is true. Log a skip.

5. **Create the Client**
    - `external_id` from the derived id; `profile_label`/`persona` from the payload's `client_profile` (the Arabic persona label, e.g. `الموظف المستقر (أبو عائلة)`). Persist.

6. **Derive Accounts from distinct account_ids**
    - Group the file's transactions by `account_id`; create one `Account` per distinct id.
    - `external_id` = the raw `account_id`; `currency` from the transactions (SAR); `bank_name` and `display_color` assigned **deterministically** (e.g. by index/hash of the account id against a fixed bank + palette list — same input always yields the same bank/color across reboots); `tokenized_account_id` generated (deterministic token, never the raw id — SAMA tokenization per DESIGN.md §9); `latest_balance` = the `closing_balance` of the most recent transaction (max `booking_date_time`) for that account.

7. **Insert Transactions**
    - For each transaction map: `external_id` = `transaction_id`; `direction` from `credit_debit_indicator` (`DEBIT|CREDIT`); `amount` from `amount.amount`; `currency` from `amount.currency`; `raw_description` = `transaction_information`; `description_cleansed` = `insights.description_cleansed`; `category` = `insights.category` (via the Step 1.2 enum-with-fallback); `category_confidence` = `insights.category_confidence`; `booking_date` = parsed `booking_date_time`; `closing_balance` = `balance.amount.amount`. Link to the correct `Account`.
    - Persist in a batch/transaction per client for efficiency.

8. **Logging & summary**
    - After the run, log per-client counts (accounts, transactions) and a total, so the operator can verify against the files.

## Acceptance Criteria

All the following must be true before marking this step complete:

- [ ] Against a fresh (V1-migrated, empty) DB, first boot loads exactly 5 clients with their derived accounts and all their transactions.
- [ ] Transaction counts per client match the number of `transactions[]` entries in each source JSON; account counts match the distinct `account_id`s per file.
- [ ] Each client's `persona`/`profile_label` equals the file's `client_profile`; each account's `tokenized_account_id` is set and is **not** equal to the raw account id.
- [ ] `bank_name` and `display_color` are assigned deterministically — a second seed run (or re-seed on a fresh DB) yields identical assignments.
- [ ] `latest_balance` per account equals the closing balance of that account's most recent transaction.
- [ ] Second boot against the already-seeded DB inserts nothing new (all 5 files skipped via `existsByExternalId`); row counts are unchanged.
- [ ] The `data-mocks` path is configurable via property/env and seeding can be disabled for tests.
- [ ] The `data-mocks/*.json` files are unmodified.

## Notes

- **Do not modify the `data-mocks/` files** (ORCHESTRATION global rule).
- Idempotency is at client granularity (skip whole file if `external_id` exists) — this keeps re-seeding cheap and avoids partial-duplicate rows. If a file was partially loaded due to a crash, the guard will skip it; document this edge case in the handoff (a `--reseed` / manual truncate is the recovery path, not in scope here).
- Determinism for `bank_name`/`display_color` matters for demo stability and for the multi-bank UI badge (DESIGN.md §7 Home); use a fixed ordered list of Saudi bank names + the DESIGN.md §8 palette, indexed by a stable hash of the account external id.
- Parse `booking_date_time` (ISO-8601 `Z`) to `Instant`. The mock lists transactions newest-first; do not assume order — compute "most recent" by max `booking_date`.
